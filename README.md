# Цифровая платформа парковки

Учебный проект курса [Systems Analyst Bootcamp](https://systems.education/systems-analyst-bootcamp).

**Репозиторий на GitHub:** [CeJIDb/sab-win-26-mine-parking](https://github.com/CeJIDb/sab-win-26-mine-parking).

Проект посвящен цифровизации частного паркинга на 600 машиномест в Санкт-Петербурге. В рамках кейса команда выступает в роли системных разработчиков: собирает и уточняет требования, описывает предметную область, моделирует целевые бизнес-процессы, формирует артефакты системного анализа, фиксирует архитектурные решения, определяет требования к будущей разработке системы и подготавливает wireframe-интерфейсы для ключевых пользовательских сценариев.

## О проекте

Цель проекта — спроектировать цифровую платформу парковки, которая поможет:

- сократить очереди на въезде и выезде;
- снизить ручную нагрузку на охрану и управляющего;
- перевести большую часть клиентских операций в онлайн;
- повысить загрузку парковочных мест;
- создать единый источник данных для управления парковкой.

Согласно артефактам проекта, целевой бизнес-результат — довести долю клиентов, использующих онлайн-сервисы, до 80% в течение 6 месяцев после запуска.

## Что входит в решение

В границах проектируемой системы рассматриваются:

- публичный сайт с тарифами, правилами и информацией о парковке;
- клиентское веб-приложение (PWA/личный кабинет) для физических и юридических лиц;
- административный контур для управляющего и владельца парковки;
- рабочий интерфейс охранника на КПП;
- интеграции со СКУД и LPR-камерами, платежной системой, терминалом, ОФД и сервисом уведомлений.

Ключевой фокус проекта — не программная реализация backend, а системный анализ и проектирование целевого решения.

## Роль команды

В этом проекте мы работаем как команда системных разработчиков. Это означает, что основной результат находится в артефактах анализа и проектирования:

- описании бизнес-контекста и границ системы;
- концептуальной модели предметной области;
- use case и функциональных требованиях;
- архитектурных решениях и допущениях;
- прототипах интерфейсов для проверки пользовательских сценариев.

## Как развивался проект

Проект шел поэтапно: от исследования AS-IS бизнеса и проблемной ситуации к проектированию целевого решения, требованиям, архитектуре, модели данных и подготовке интеграционного контура.

- Этап 1 — моделирование бизнеса: AS-IS процессы, ED AS-IS, BPMN, UML Class Diagram, UML StateChart Diagram.
- Этап 2 — концептуальное проектирование IT-решения: Opportunity Canvas, Impact Map, карточка проекта, User Story Map, контекстная диаграмма, концептуальная модель, use case, схема навигации, wireframe, ES TO-BE.
- Этап 3 — требования к ПО: функциональные требования, нефункциональные требования и ограничения.
- Этап 4 — техническое проектирование: DDD, модульный монолит, C4, ERD, SQL-практика, инфобез, DRAKON.
- Этап 5 — интеграции: в планировании.

Подробная история развития проекта и рекомендации по размещению новых артефактов вынесены в [`docs/process/project-journey.md`](docs/process/project-journey.md) и [`docs/process/artifact-placement-guide.md`](docs/process/artifact-placement-guide.md).

## Структура репозитория

### `docs/`

Основной раздел проекта. Здесь находятся канонические артефакты системного анализа, требований и архитектуры:

- `artifacts/` — карточка проекта, бизнес-контекст, концептуальная модель, use case, TO-BE процессы, glossary, инфобез и другие аналитические артефакты;
- `specs/` — функциональные, нефункциональные требования и ограничения;
- `architecture/` — ADR, DDD-материалы, C4-диаграммы, архитектура данных и ERD;
- `process/` — правила участия, трассировка, шаблоны PR, release-checklist и рабочие регламенты;
- `interviews/` — планы интервью, протоколы встреч и зафиксированные решения;
- `demo-days/` — материалы для демо и презентаций;
- `readme.md` — индекс проектной документации, удобная точка входа в `docs/`.

Рекомендуем начинать навигацию именно с [`docs/readme.md`](docs/readme.md).

### `ui/`

Статический wireframe-макет интерфейсов, который помогает визуализировать целевые сценарии и сверять артефакты с пользовательскими потоками:

- `site/` — публичный сайт: тарифы, правила, контакты, карта парковки, регистрация и вход;
- `client/` — личный кабинет клиента;
- `admin/` — административный контур;
- `guard/` — рабочий интерфейс охранника на КПП;
- `templates/` — Nunjucks-шаблоны страниц, layout и partials;
- `scss/` — исходники стилей;
- `index.html` — точка входа в собранный wireframe;
- `README.md` — подробное описание состава и навигации UI.

Важно: `ui/` — это демонстрационный макет, а не готовый продуктовый frontend.

### `scripts/`

Вспомогательные скрипты для поддержки репозитория и сборки материалов:

- проверка Markdown и ссылок;
- проверка имени ветки и поддержка git-workflow;
- сборка wireframe-шаблонов и CSS;
- генерация C4 preview;
- утилиты для документации в `scripts/docs/`.

### `sql/`

Практические SQL-материалы по доменной модели парковки:

- `practice/ddl/` — экспорт и DDL-структуры;
- `practice/dml/` — seed-данные;
- `practice/queries/` — учебные и исследовательские запросы.

### `evals/`

Вспомогательные материалы для evals отдельных правил и агентных сценариев. Раздел не является основным источником требований, но полезен для проверки качества автоматизации.

### `temp_context/`

Временные рабочие контексты для промежуточного анализа. Папка не является каноническим источником требований; устойчивые результаты должны переноситься в `docs/`.

### Служебные каталоги

- `.github/` — workflow и шаблоны GitHub;
- `.husky/` — локальные git hooks;
- `.cursor/` — правила, команды и агентные настройки для работы с репозиторием.

Зависимости и локальные окружения (`node_modules/`, `.venv*`) не считаются частью проектных артефактов и в обзорную структуру не входят.

## Основные предметные области

В проекте подробно прорабатываются следующие сущности и сценарии:

- клиенты: физические и юридические лица;
- транспортные средства, парковочные места, сектора, КПП;
- тарифы, бронирования, договоры, парковочные сессии;
- оплата, чеки, уведомления, обращения;
- автоматический и ручной допуск на въезде и выезде;
- администрирование тарифов, долгов, доступов и отчетности.

Отдельное внимание уделяется интеграционному контуру и правилам допуска через КПП. Одно из ключевых архитектурных решений проекта — онлайн-проверка права доступа в платформе, а не в СКУД.

Для визуализации пользовательских сценариев в репозитории есть статический wireframe.

## Для контрибьюторов

Перед началом работы ознакомьтесь с `CONTRIBUTING.md`.

Короткий рабочий цикл:

1. Создайте ветку по соглашению: `feature/*`, `docs/*`, `chore/*`, `hotfix/*`.
2. Вносите изменения небольшими и трассируемыми шагами.
3. Используйте коммиты в стиле Conventional Commits (`docs(...)`, `feat(...)`, `fix(...)`, `chore(...)`).
4. Перед открытием PR выполните:

```bash
npm ci
npm run check:branch
npm run ci:check
```

## Первый вклад

Если вы впервые участвуете в проекте:

1. Прочитайте `CONTRIBUTING.md`.
2. Выберите задачу и свяжите ее с источником требований (артефакт, интервью/протокол, FR/NFR, ADR).
3. Создайте ветку: `docs/<тема>`, `feature/<тема>`, `chore/<тема>`, `hotfix/<тема>`.
4. Внесите минимальный набор трассируемых изменений.
5. Обновите матрицу трассировки (`docs/process/traceability-matrix-log.md`), если это требуется по [регламенту](docs/process/traceability-matrix.md).
6. Запустите проверки:

   ```bash
   npm ci
   npm run check:branch
   npm run ci:check
   ```

7. Откройте PR по шаблону и выполните критерии DoR/DoD (`docs/process/pr-dor-dod.md`).

## Контрольные точки качества

Изменение готово к слиянию (merge), если:

- проверки CI проходят успешно;
- изменения привязаны к артефактам и требованиям;
- формулировки требований проверяемы и однозначны;
- в описании PR указаны риски и шаги валидации.

## Политика релизов и изменений

- При выпуске версии записи переносятся в версионированный блок.
- Релиз выполняется по чеклисту `docs/process/release-checklist.md`.

## Учебный статус проекта

Это учебный репозиторий, созданный в рамках обучения системному анализу. Его главная ценность — не production-код, а связный комплект артефактов, который показывает путь от исследования предметной области и интервью с заказчиком до требований, архитектурных решений и интерфейсных макетов.

При работе с проектом активно применяются инструменты ИИ, в частности `Cursor IDE`. Он используется как рабочая среда для анализа артефактов, подготовки проектной документации, уточнения формулировок требований, навигации по репозиторию и ускорения рутинных задач команды.
