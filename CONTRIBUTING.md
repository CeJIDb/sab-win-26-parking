# Руководство для участников

В репозитории принят **документо-ориентированный** процесс: чаще всего меняются артефакты требований и архитектуры, а также скрипты сборки wireframe.

## Стиль текстов в документации

В русскоязычных markdown-файлах (спеки, архитектура, артефакты, процессы) **не используйте букву «ё»** — пишите **«е»** (например: «все», «еще», «подъем»). Это снижает рассогласованность между правками из разных раскладок и инструментов.

## Именование веток

Используйте один из шаблонов:

- `feature/<краткая-тема>`
- `docs/<краткая-тема>`
- `chore/<краткая-тема>`
- `hotfix/<краткая-тема>`

Примеры: `docs/nfr-auth`, `feature/use-case-registry`, `chore/ci-quality-gates`.

Проверка в CI: регулярное выражение `^(feature|docs|chore|hotfix)\/[a-z0-9._-]+$`.

## Соглашение о коммитах

Формат **Conventional Commits** с проверкой **commitlint** (`commitlint.config.cjs`; при необходимости см. подсказки в `.cursor/agents/git-workflow-master.md`):

- **Тип** и **scope** (если есть) — **латиницей**; краткое **описание после двоеточия — на русском** (до 100 символов в заголовке).
- `docs(scope): …`, `feat(scope): …`, `fix(scope): …`, `chore(scope): …` и другие разрешённые типы из конфига.

Примеры:

- `docs(specs): уточнить NFR по внешнему качеству`
- `chore(ci): добавить проверку ссылок в markdown`

Локальная проверка:

- сообщения коммитов — Husky + Commitlint (`.husky/commit-msg`);
- заголовок PR — CI (`.github/workflows/pr-title.yml`).

### Атомарные коммиты по зонам репозитория

Чтобы разнести несвязанные правки в отдельные коммиты с корректными типом и scope (без ручного `git add` по частям), используйте:

- `npm run commit:atomic` — показать план и запросить подтверждение;
- `npm run commit:atomic:yes` — выполнить без вопроса (удобно в скриптах);
- `node ./scripts/atomic-commit.mjs --dry-run` — только план, без коммитов.

Скрипт группирует изменённые файлы (относительно `HEAD`) по каталогам: `docs/specs/`, `docs/architecture/`, `ui/`, `.cursor/` и т.д., затем создаёт последовательность коммитов в фиксированном порядке.

Заголовки коммитов — **на русском после двоеточия**, в том же духе, что и правила **git-workflow-master**; сам агент Cursor **не вызывается** — шаблоны заданы в скрипте.

Не выполняет `git push`. При незавершённом merge/rebase завершается с ошибкой.

Подробности и порядок групп — в `scripts/atomic-commit.mjs` (комментарии в начале файла).

**Субагент git-workflow-master из git-хуков автоматически не вызывается** (технически недоступно). При крупном диффе Husky запускает `scripts/git-workflow-agent-reminder.mjs` и выводит напоминание в терминал (см. `.husky/pre-commit`, `pre-push`).

## Определение готовности PR (Definition of Ready)

PR можно открыть только если:

- [ ] цель сформулирована в 1–2 предложениях;
- [ ] указан источник истины (`docs/artifacts/*`, `docs/specs/*`, `docs/architecture/*`, `docs/interviews/*`);
- [ ] явно обозначены границы изменений (что входит и что не входит);
- [ ] перечислены риски и зависимости;
- [ ] описан способ проверки результата.

## Определение завершённости PR (Definition of Done)

Перед открытием PR убедитесь, что выполнено:

- [ ] `npm ci`
- [ ] имя ветки соответствует политике (`npm run check:branch`)
- [ ] локально проходит `npm run ci:check`
- [ ] изменения связаны с релевантными артефактами и спецификациями
- [ ] формулировки требований проверяемы и однозначны (предпочтительно: «система должна …»)
- [ ] в `docs/specs/` сохранена структура документов; устаревшие требования помечены как устаревшие/заменённые, а не удаляются без пометки
- [ ] обновлена матрица трассировки (`docs/process/traceability-matrix-log.md`)
- [ ] в описании PR есть объём, риски и шаги валидации

## Контрольные точки качества

PR готов к слиянию, когда:

- CI «зелёный» (`lint:md`, `build`);
- проходят проверки политики (`check:branch`, commitlint, заголовок PR);
- назначенные ревьюеры утвердили изменения;
- нет нерешённых комментариев ревью;
- объём изменений ограничен и прослеживается до исходных требований/протоколов.

## Чеклист для ревьюера

Ревьюер проверяет:

- однозначность и измеримость формулировок требований;
- отсутствие случайного удаления критичных артефактов;
- соответствие имен веток и коммитов политике;
- отражение влияния на `README`, спеки и архитектурную документацию.

## Заметки о релизе

Релиз выполняется по чеклисту проекта и сопровождается актуализацией релевантных артефактов и трассировки.

## Матрица трассировки

Для содержательных изменений поддерживайте связь:
`Источник → Требование → Изменённые файлы → Проверка → Доказательство`.

Используйте:

- регламент: `docs/process/traceability-matrix.md`;
- журнал: `docs/process/traceability-matrix-log.md`;
- шаблон: `docs/process/templates/traceability-matrix-template.md`.

## Чеклист перед релизом

Перед релизом по тегу выполните пункты из `docs/process/release-checklist.md`.
