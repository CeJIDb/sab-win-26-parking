# План: декомпозированная DFD Level 1 платформы парковки

**Дата**: 2026-04-28
**Задача**: построить декомпозированную DFD Level 1 для цифровой платформы парковки на основе существующих C4 L3 и ERD; разместить артефакт в `docs/artifacts/dfd-l1.md`

## Зачем именно так

В проекте уже есть **DFD Level 0** ([docs/artifacts/context-diagram.md](../docs/artifacts/context-diagram.md)) и **C4 L1/L2/L3** ([docs/architecture/c4/c4-diagrams.md](../docs/architecture/c4/c4-diagrams.md)) с 20 компонентами + 11 внешними системами. Между этими двумя слоями отсутствует промежуточный артефакт, который показывает, **как именно данные движутся между процессами и внешними системами**. DFD L1 закрывает этот зазор.

Артефакт нужен для:

- Этап 5 (Интеграции): помечен в [docs/process/artifact-placement-guide.md](../docs/process/artifact-placement-guide.md) и [docs/architecture/integration/readme.md](../docs/architecture/integration/readme.md) как ожидаемый материал.
- Demo Day 5: упомянут в [docs/demo-days/demo-5/slides-map.md](../docs/demo-days/demo-5/slides-map.md) как часть «UC + DFD + Sequence».
- Контроля целостности: трассировка к L0 (мягкая, с фиксацией расхождений) и к C4 L3 (1:1 покрытие компонентов).

**Решения по формату (приняты в диалоге до начала работы):**

- **Глубина — L1.** Без L2 в этой итерации.
- **Гранулярность процессов — 1:1 с C4 L3.** Каждый из 20 компонентов C4 L3 становится отдельным процессом DFD. Без агрегации в макропроцессы. Названия — в стиле образцов курса («Сервис X»), хотя это компоненты, а не процессы по канону DFD. Принятый компромисс ради совместимости с форматом курса (см. примеры `plans/dfd-bank.jpg` и аналог другой команды).
- **Нумерация процессов 1–20.** Для трассировки к C4 L3.
- **Хранилища — 11 логических агрегатов ERD** (`D1`–`D11`). Список зафиксирован в Фазе 0 (сверка с ERD). Это не «одно физическое Postgres», а логические единицы данных.
- **Внешние сущности — 15 по C4 L3** (5 акторов + 10 внешних систем; ОФД удален — фискализацию берет ЮKassa). Отдельным визуальным стилем через Mermaid `classDef` (отличаются от процессов и хранилищ).
- **Нотация — обычные прямоугольники и стрелки.** Без Гейн-Сарсон. Три класса узлов через `classDef`: процесс, хранилище, внешняя сущность.
- **Подписи потоков — существительные.** Слово «данные» допустимо там, где без него подпись становится неоднозначной (например, «данные для авторизации»). Где можно — атомарная сущность («бронь», «пропуск», «отзыв», «событие въезда»).
- **Только потоки данных.** Control flows и события не показываем как отдельный класс связи.
- **Балансировка с L0 — мягкая.** L0 уже сдан и не правится. Расхождения (если на L1 появилось то, чего нет на L0) фиксируем delta-таблицей, без правки L0.
- **Layout — плоско, через Mermaid + ELK-рендерер.** Все узлы на одной плоскости. Fallback при нечитаемом рендере — `subgraph` как чисто визуальные рамки по bounded contexts (без изменения семантики «один компонент = один процесс»).
- **Рендер — Mermaid в `.md`.** Исходник draw.io пользователь нарисует отдельно по этой выжимке.
- **CI-проверка синтаксиса Mermaid.** Подключаем `@mermaid-js/mermaid-cli` как devDependency, скрипт `scripts/lint-mermaid.mjs` извлекает все блоки `mermaid` из `docs/**/*.md` и прогоняет через `mmdc`. Падение — на синтаксической ошибке. Подключаем в `npm run ci:check`.

Отказались:

- L2 для отдельных процессов — отдельный план следующей итерацией.
- Прямой XML draw.io от агента — координаты выйдут плохими, исходник нечитаемый.
- Одно физическое хранилище `D1 PostgreSQL` — теряется DFD-семантика, диаграмма превращается в звезду.
- Гибрид «макропроцесс + список вложенных C4 L3 в подписи» — методологически псевдо-уровень между L1 и L2.
- Гейн-Сарсон-нотация — Mermaid не поддерживает «прямоугольник без правого края» из коробки, рендер становится хрупким.
- Канонические DFD-процессы как «глагол + объект» — не соответствует формату образцов курса; принят компромисс.
- Раздел «Тайминг» — не оцениваем по часам.

## Цель

- В `docs/artifacts/dfd-l1.md` лежит готовый артефакт по структуре аналогично [context-diagram.md](../docs/artifacts/context-diagram.md): назначение, источник, диаграмма (Mermaid), текстовое описание, ключевые элементы, логика, словарь потоков данных, выводы и связанные документы.
- Mermaid-диаграмма содержит **20 процессов** (с номерами 1–20, по C4 L3), **11 логических хранилищ** `D1`–`D11` (по агрегатам ERD) и **15 внешних сущностей** (по C4 L3, без ОФД). Три класса узлов через `classDef`.
- В таблице «Процесс → C4 L3» каждый из 20 процессов сопоставлен с компонентом 1:1.
- В таблице «Хранилище → ERD» каждый `D1`–`D11` сопоставлен с таблицами ERD.
- Все потоки данных именованы существительными (слово «данные» допустимо как часть подписи там, где без него неоднозначно).
- Раздел «Балансировка с L0» — delta-таблица: для каждого внешнего потока L0 указано, как он представлен на L1; для каждого нового потока L1 — пометка «нет на L0, расхождение зафиксировано».
- В проекте подключен `lint:mermaid` (синтаксис), `npm run ci:check` зеленый.
- Обновлены: [docs/artifacts/readme.md](../docs/artifacts/readme.md), [docs/artifacts/context-diagram.md](../docs/artifacts/context-diagram.md) (ссылка в «Связанные документы»), [docs/process/traceability-matrix-log.md](../docs/process/traceability-matrix-log.md).
- Новые файлы соответствуют [стилю текстов в документации](../CONTRIBUTING.md#стиль-текстов-в-документации).

## Scope

**Входит:**

- Новый файл `docs/artifacts/dfd-l1.md` со всеми разделами по styleguide.
- Mermaid-диаграмма L1 с тремя классами узлов и нумерацией процессов.
- Словарь потоков данных (таблица) с привязкой к таблицам ERD и хранилищам `D1`–`D6`.
- Таблица «Процесс → C4 L3 компонент» (1:1, 20 строк).
- Таблица «Хранилище `D1`–`D6` → ERD-таблицы».
- Таблица «Поток L0 → представлен на L1 как» с пометкой расхождений.
- Подключение `@mermaid-js/mermaid-cli` как devDependency, скрипт `scripts/lint-mermaid.mjs`, запись `lint:mermaid` в `package.json`, подключение к `ci:check`.
- Обновление [docs/artifacts/readme.md](../docs/artifacts/readme.md) — ссылка на новый артефакт.
- Обновление [docs/process/traceability-matrix-log.md](../docs/process/traceability-matrix-log.md) — запись формата «Источник → Требование → Изменения → Проверка → Доказательство».
- Обновление [docs/artifacts/context-diagram.md](../docs/artifacts/context-diagram.md) — ссылка на DFD L1 в «Связанные документы».

**Не входит:**

- DFD Level 2 для отдельных процессов.
- Исходник draw.io.
- Любые правки [docs/specs/](../docs/specs/).
- Рефакторинг ERD или C4 L3 — DFD должна **отражать** текущее состояние, а не править его.
- Правки L0 ([context-diagram.md](../docs/artifacts/context-diagram.md), кроме добавления ссылки на DFD L1 в «Связанные документы»). L0 уже сдан.
- Control flows / события как отдельный класс связи на диаграмме.
- Визуальное тестирование рендеринга Mermaid через ИИ — проверяем только синтаксис через `mmdc`.
