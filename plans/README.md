# plans/ — технические планы реализации

Эта папка — место для **технических планов** реализации изменений в репозитории. Здесь живёт ответ на вопрос **как** мы делаем ту или иную правку инфраструктуры, инструментов или процессов.

Для артефактов системного анализа (**что** и **зачем**) по-прежнему используется [docs/specs/](../docs/specs/) и [docs/artifacts/](../docs/artifacts/). Процессные регламенты — [docs/process/](../docs/process/). Планы в этой папке — про шаги и тайминг конкретной задачи.

## Правила

1. **Один план = одно изменение.** Не смешиваем несколько независимых задач в одном файле. Если план уже существует — работаем с ним, а не создаём новый.
2. **Формат имени файла:** `YYYY-MM-DD-название-задачи.md` (например, `2026-04-24-gym-app-tooling-adoption.md`). Имя в kebab-case, только латиница, без буквы «ё».
3. **Фазы.** Каждый план делится на фазы. У каждой фазы статус `[ ]` (не сделано) или `[x]` (сделано).
4. **Итоговый блок.** В конце плана — итог: реализован целиком или нет, что осталось, какие решения приняты по ходу.
5. **Актуализация.** Любой агент или контрибьютор, работающий с планом, **обязан** его актуализировать после сессии: отметить завершённые фазы, дописать что сделал, обновить итоговый блок.
6. **Связь с PR.** Ссылка на план — обязательная часть PR-описания (см. [.github/PULL_REQUEST_TEMPLATE.md](../.github/PULL_REQUEST_TEMPLATE.md) и [docs/process/pr-dor-dod.md](../docs/process/pr-dor-dod.md)).

## Рекомендуемая структура плана

```markdown
# План: [короткое название]

**Дата**: YYYY-MM-DD
**Задача**: [одно предложение — что делаем]
**Время**: [бюджет в часах]

## Зачем именно так

[контекст решения: почему такой scope, какие альтернативы отброшены]

## Цель за N часов

[конкретный проверяемый результат]

## Scope

[что входит, что не входит]

## Тайминг

| Минуты | Блок | Что делаем |
| ------ | ---- | ---------- |

## Правила коммитов и веток

[имя ветки, ожидаемые атомарные коммиты]

## Определение «готово»

- [ ] [чеклист критериев приёмки]

## Фазы и статус

- [ ] Фаза 0. ...
- [ ] Фаза 1. ...

## Итог

[заполняется по ходу и в конце сессии]
```
