## **Ход встречи**

    ### Контекст встречи

        Встреча была  организована, чтобы обсудить с заказчиком и договориться о концепции реализации процесса to-be предоставления парковочного места. В оставшееся время обсуждался Opportunity Canvas.

    ### Обсуждение Event Storming TO-BE (процесс предоставления ПМ)

        - **Команда** представила заказчику свое видение процесса:

            - **предложение команды:** отсутствие обязательной регистрации для почасовых клиентов ФЛ (без ЛК) в системе (заказчик возразил, см. ниже);

            - мотивирование клиентов зарегистрироваться ЛК:

                - удобные способы оплаты и возможность подключения автосписания;

                - скидка за регистрацию и онлайн-оплату;

                - управление списком ТС;

                - история парковочных сессий;

                - оформление договора для длительного пользования услугами парковки на выгодных тарифах.

            - размещение информации о тарифах, условиях и оферте парковки:

                - информационные щиты;

                - информационное табло;

                - сайт.

            - для ФЛ (без ЛК) и ФЛ (почасовая) — постоплата на выезде по фактическому времени парковочной сессии;

            - для ФЛ (договор) — оплата договора онлайн;

            - «бесконтактный» въезд–выезд:

                - автоматическое распознавание номеров (ГРЗ);

                - автоматическая идентификация клиентов по распознанному ГРЗ:

                    - ФЛ (без ЛК) — разовый (гостевой) почасовой;

                    - ФЛ (почасовая, с ЛК);

                    - ФЛ (договор);

                    - ЮЛ.

                - автоматическое принятие решения о доступе:

                    - проверка права доступа по типу клиента/договору/тарифу;

                    - команда на открытие шлагбаума при разрешении доступа;

                    - фиксация факта проезда (въезд/выезд) в системе.

                - создание парковочной сессии (жизненный цикл):

                    - на въезде — создание парковочной сессии (ГРЗ, тип клиента, назначение сектора/ПМ, время начала);

                    - на выезде — завершение парковочной сессии (время окончания, расчет суммы, статус оплаты).

                - автоматический учет занятости и предоставления места:

                    - на въезде — уменьшение доступности (свободных мест) в назначенном секторе / фиксация занятого ПМ (для договорных/ЮЛ);

                    - на выезде — увеличение доступности (свободных мест) / освобождение ПМ.

                - информационный дисплей на въезде:

                    - ГРЗ;

                    - право доступа;

                    - тариф;

                    - инструкции водителю;

                    - назначенное ПМ:

                        - ФЛ (почасовая) — сектор (любое свободное ПМ внутри сектора);

                        - ФЛ (договор) и ЮЛ — ПМ (или диапазон ПМ) по договору.

                - информационный дисплей на выезде:

                    - ГРЗ;

                    - право доступа;

                    - тариф;

                    - сумма к оплате;

                    - статус платежа;

                    - инструкции водителю.

                - правило открытия шлагбаума на выезде:

                    - открытие после подтверждения оплаты (или автосписания для клиентов с ЛК);

                    - отображение результата на дисплее (оплачено/ожидает оплаты).

            - несколько способов оплаты:

                - онлайн:

                    - клиенты с ЛК — через ЛК с возможностью подключения автосписания;

                    - все клиенты — через ссылку по QR-коду и ввод номера ГРЗ.

                - офлайн:

                    - терминал у шлагбаума на выезде (опция, если ставим);

                    - терминал у охранника на КПП (уже есть) — резервный способ.

            - обработка исключений:

                - ГРЗ не распознан / распознан некорректно:

                    - повторное распознавание;

                    - ручной ввод ГРЗ охранником.

                - ГРЗ распознан, но право доступа не подтверждено:

                    - перевод в сценарий разовой почасовой парковки (если разрешено правилами);

                    - либо отказ во въезде с инструкцией на дисплее и перенаправлением к охраннику.

                - ГРЗ в черном списке:

                    - отказ во въезде;

                    - или перевод на ручную обработку охранником.

                - Ошибка оплаты / оплата не подтверждена:

                    - шлагбаум на выезде не открывается;

                    - предложение альтернативного способа оплаты (QR/ЛК/терминал);

                    - эскалация к охраннику как резервный способ.

                - Недоступны внешние сервисы/оборудование (платежный сервис/связь/камера/СКУД):

                    - ручная обработка охранником (идентификация/открытие/фиксация);

                    - последующее внесение/синхронизация данных в системе.

                - Спорные ситуации по сессии:

                    - перевод сессии в статус «спорная» и фиксация основания (тип инцидента, комментарий, фото/видео при наличии);

                    - обработка через охранника/управляющего по регламенту:

                        - урегулирование на месте (разъяснение расчета, повторная попытка оплаты, выбор альтернативного способа);

                        - при отказе оплаты — фиксация задолженности/инцидента по ГРЗ и оформление материалов для дальнейшего взыскания (если предусмотрено правилами/офертой).

                    - при конфликтной ситуации с угрозой персоналу/имуществу:

                        - эскалация охраннику (при необходимости — вызов ГБР/полиции);

                        - фиксация инцидента (записи камер, данные сессии, действия персонала).

                    - при претензионной/судебной работе:

                        - сохранение доказательной базы: данные парковочной сессии, статус оплаты, журнал событий СКУД, записи камер.

        - **Заказчик**

            - **Согласовал:**

                - постоплата;

                - оплата через ЛК (как предпочтительный основной онлайн-сценарий);

                - договор закрепляет ПМ (или диапазон ПМ) за клиентом на весь срок действия;

                - необходимость понятного интерфейса на въезде (ГРЗ, тариф, право доступа, инструкции и т.д.);

                - необходимость явного согласия клиента с офертой/правилами.

            - **Согласовал с оговорками:**

                - распознавание ГРЗ — поддержал как предпочтительный механизм (с учетом влияния на пробки), при этом обсуждал стоимость/альтернативы;

                - терминал у шлагбаума на выезде: возможен, если недорого; не обязателен для первого релиза, т.к. терминал у охранника на КПП уже есть.

            - **Не согласовал:**

                - запускать почасовых ФЛ без регистрации/контактных данных, настаивает на минимальной идентификации (телефон+имя). Как причина было озвучено, что имеется риск порчи имущества (соседних машин) на парковке и необходимость иметь контактные данные для разбирательств в случае инцидентов;

                - множить онлайн-каналы оплаты в MVP (ЛК + QR) — из-за усложнения сценариев и клиентского пути.

            - **Открытый вопрос:** 

                - Сколько онлайн-каналов оплаты оставляем в MVP: только ЛК или ЛК + QR?

            - **Предложил:**

                - компромисс: терминал/кнопка подтверждения согласия с тарифом и правилами на въезде для ФЛ (без ЛК) как “явный акт согласия”.

        

    ### Обсуждение Opportunity Canvas (правая часть)

        - **Идеи решений**

            - Команда показала заполненную правую часть Canvas; отметила, что идеи решений в основном взяты из ранее согласованной Impact Map.

            - В состав решений включены:

                - **Личный кабинет клиента**;

                - **Административная панель управления**;

                - **Публичный портал/сайт парковки**;

                - **Интеграция с модулем онлайн-оплаты**;

                - **Система обращений/обратной связи** (вероятно не MVP);

                - **Биллинг, интеграция с 1С** (не для MVP);

                - Интеграции/оборудование офлайн-части: **СКУД и интеграция со СКУД**, **LPR-камеры**, **терминал для офлайн-оплаты (резервный сценарий)**, **информационные табло на въезде/выезде**.

        - **Ценность для пользователей**

            - Команда представила “ценность для пользователей” (переформулированные воздействия из Impact Map).

            - Заказчик уточнил необходимость канала поддержки: в ЛК должна быть возможность **создать обращение**; команда подтвердила (с оговоркой, что может быть не MVP).

        - **Пользовательские метрики**

            - Вместе с заказчиком были откорректированы значения целевых показателей метрик;

            - Заказчик запросил возможность **добавить/уточнить пользовательские метрики асинхронно** (через комментарии/чат).

        - **Стратегия внедрения**

            - Команда предложила, и заказчик согласовал:

                - **перевод клиентов в онлайн** и мотивация (в т.ч. через рекламные активности);

                - **обучение персонала парковки** работе со СКУД, камерами распознавания и обработке спорных ситуаций;

                - **обучение управляющего/заказчика** работе с админ-панелью (просмотр показателей, изменение параметров, настройка прав доступа и оплат).

        - **Бизнес-преимущества и метрики**

            - **Рейтинг на геосервисах**: целевое значение — **не ниже 4,5**.

            - **Доля клиентов, пользующихся парковкой “через онлайн”** — метрика, выведенная из цели Impact Map.

            - **Пропускная способность парковки** — заказчик предложил подумать над способом расчета целевого значения с учетом поправочного коэффициента на новых клиентов и спорные/проблемные случаи (в т.ч. ошибки распознавания).

                - **Текущая пропускная способность:** 40–70 машин/час.

            - **Простой** — ****снизить до <10%.

        - **Бюджет**

            - Команда озвучила оценку разработки (сайт/ЛК/админка) при сроке **4–6 месяцев**: **5–10 млн руб.** (без оборудования).

            - Заказчик обозначил ориентир бюджета **~10 млн руб. на всё** (включая оборудование), с готовностью рассмотреть превышение ориентира при обосновании до **15 млн (предельный бюджет).**

## Договоренности

    ### **От команды**

        - Обновить Event Storming TO-BE с учетом позиции заказчика по обязательной минимальной идентификации для почасовых клиентов (телефон + имя).

        - Проработать и предложить вариант «явного акта согласия» с офертой/правилами для ФЛ (без ЛК) на въезде (например, терминал/кнопка подтверждения), описать пользовательский сценарий.

        - Подготовить предложение по составу MVP в части оплаты: минимизировать количество онлайн-каналов; вынести варианты (только ЛК vs ЛК + QR) на согласование.

        - Уточнить состав и стоимость оборудования (LPR, табло/дисплеи, терминалы) и показать влияние на бюджетный коридор.

        - Обновить правую часть Opportunity Canvas (метрики/ценности/внедрение) по итогам обсуждения и отправить заказчику.

    ### От заказчика

        - Дать обратную связь по выбору онлайн-канала(ов) оплаты в MVP: только ЛК или ЛК + QR.

        - Подтвердить требования к минимальной идентификации почасовых клиентов (состав данных и точка сбора в сценарии).

        - Дать обратную связь по варианту «явного согласия» на въезде (принимаем ли терминал/кнопку подтверждения и в каком виде).

        - При необходимости — дополнить/уточнить пользовательские метрики в чате асинхронно.

    ### Общие договоренности

        - Зафиксировать состав MVP по оплате и сценарию гостевых клиентов (без ЛК) после обмена обратной связью в чате.

        - Продолжить согласование артефактов асинхронно (через чат/комментарии), чтобы не выносить все уточнения на следующую встречу.

        - Обсудить Definition of Done для User Story Map в части MVP асинхронно в чате.
