## **Ход встречи**

#### **Блок 1: Концепция процесса TO-BE (Event Storming)**

1. **Согласована** итоговая версия диаграммы Event Storming на уровне Big Picture. Она становится основой для дальнейшей детальной проработки (моделирование процессов, выделение агрегатов).

2. **Критическое изменение:** Обязательная регистрация в личном кабинете для **всех** клиентов (как для долгосрочной аренды, так и для краткосрочной). Это необходимо для идентификации пользователя, создания парковочной сессии и обеспечения постоплаты.

3. **Идентификация:** Система распознает клиента по госномеру (ГРЗ), который привязан к его личному кабинету. Это единый идентификатор для проверки прав доступа (активный договор/бронь, отсутствие в черном списке).

4. **Оплата и выезд:**

    - Принята концепция постоплаты: клиент оплачивает сессию **до** выезда (через приложение) или **на** выезде.

    - Введено понятие **«грейс-период»** — временной интервал для выезда после оплаты. Конкретная длительность (напр., 15 минут) требует уточнения как бизнес-правило.

    - Механизм работы с должниками и черным списком будет проработан детально на этапе моделирования подпроцессов.

#### **Блок 2: Навигация и экранные интерфейсы**

1. **Общая структура:** Согласована схема навигации с разделением на:

    - Публичный сайт-визитку (тарифы, контакты, свободные места).

    - Личный кабинет клиента (физлица и юрлица).

    - Интерфейс сотрудника КПП (охранника).

    - Интерфейс администратора (владелец/управляющий).

1. **Авторизация:** Для входа в систему клиентами будут использоваться российские сервисы SSO. По результатам обсуждения, в приоритете: **VK ID, Яндекс ID.**

2. **Клиентский интерфейс:**

    - **На главной странице** личного кабинета необходимо обеспечить быстрые действия: прямая кнопка «Оплатить» для текущей сессии и кнопка/виджет для быстрого создания бронирования.

    - В интерфейсах оплаты необходимо минимизировать количество шагов. Функционал автосписания возможен, хранение данных карт — на стороне платежного сервиса.

    - В интерфейсах необходимо четко разграничивать понятия «Сессия» (факт нахождения на парковке) и «Бронирование» (предварительный заказ места), добавив подсказки для пользователей.

    - ~~Подтверждена~~ Необходимость раздельного хранения полей ФИО будет уточнена на этапе разработки БД.

1. **Интерфейс сотрудника КПП:**

    - **Согласована концепция разделения:** На этапе MVP СКУД (система контроля доступа) и разрабатываемое ПО работают параллельно, вероятно, на двух мониторах. Наше ПО предоставляет информацию по клиентам, черным спискам и журнал событий.

    - Для ручного ввода ГРЗ (при сбое распознавания) интерфейс должен быть максимально простым и предусматривать фиксацию факта открытия шлагбаума с возможностью последующего внесения номера и причины - все поля должны быть заполнены таким же образом, как и при автоматическом внесении.

## Договорённости

Открытые вопросы договорились прояснить асинхронно в чате:

1. **Команда разработки:**

    - Детально проработать подпроцессы (оплаты, работа с долгами, ручной пропуск) и агрегаты на основе согласованной Event Storming диаграммы.

    - Предложить на согласование заказчику какие именно способы авторизации будут: VK ID и Яндекс ID.

    - Согласована реализация отдельного вход для клиентов и входа для сотрудников (с точки зрения безопасности). 

    - Согласована автоматическая блокировка и ручная разблокировка клиентов-должников (сценарий: «оплатил с задержкой», «попал в черный список за нарушение»).

    - **Экранные формы:** доработать с учетом замечаний:

        - быстрые действия на главной странице сразу должны быть доступны (оплата бронирования);

        - порядок полей при создании договора (унификация карточки клиента и ГРЗ, ПМ для охранника);

        - если будет время проработать отображение интерфейсов на мобильных устройствах, т.к. для клиентов это в приоритете (в то время как персонал будет использовать платформу с ПК);

        - непрочитанные уведомления выделять отдельно от прочитанных.

    - **User Story Map:** подготовить и направить Заказчику  для согласования.

    - **Платежный терминал на выезде:** Необходимо оценить целесообразность установки отдельного терминала. Нужно проанализировать частоту сценариев оплаты на выезде (опросы, экспертиза заказчика) и стоимость внедрения/обслуживания такого решения. Пока в фокусе — оплата через охранника как альтернатива.

    - **Интеграция со СКУД:** Требуется изучить типовые интерфейсы и API СКУД, чтобы оценить глубину будущей интеграции. Понять, какие данные мы можем получать в автоматическом режиме, а какие придется вносить вручную, и насколько это критично для пользовательского опыта охранника.

    1. **Синхронизация данных:** Необходимо проработать механизм синхронизации тарифов между внутренней системой и сайтом-визиткой (для MVP допускается ручное обновление, на будущее нужна автоматическая синхронизация).

2. **Со стороны заказчика:**

    - Ожидать от команды доработанные макеты и диаграммы.

    - **По детализации бизнес-правил уточнить/согласовать:**

        - Согласование способа авторизации через VK ID и Яндекс ID.

        - Какая должна быть длительность грейс-периода на выезд?

        - Оплата через охранника как альтернатива платежному терминалу при выезде.

        - Требования к выдаваемым чекам, с учетом статуса ИП, требуется именно фискальный чек, а не просто квитанция.
