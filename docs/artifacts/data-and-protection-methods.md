# Данные и методы защиты

Артефакт фиксирует **категории персональных и иных критических данных** системы управления парковкой, подпадающих под требования 152-ФЗ и смежные требования по безопасности, и **рекомендуемые технические меры** их защиты. Соответствие сущностям — см. «Паспортные данные», «Льготный документ», «Согласие», «Клиент», «Сотрудник», «Организация», «ТС», «Обращение», «Уведомление», «Платеж», «Чек», «Настройки оплаты» в `conceptual-model-with-attributes.md`.

**Граница артефакта:** здесь описаны только **данные и технические меры**. Правовые основания обработки, классификация ИСПДн (УЗ-1/2/3), документы оператора (политика, модель угроз, регламенты, права субъектов) и применение сертифицированных СКЗИ выносятся в отдельные артефакты (см. раздел «Что не входит в артефакт»).

## Таблица: данные и методы защиты

| Данные | Методы защиты |
|--------|----------------|
| **Паспортные данные** (тип документа, серия и номер, дата выдачи, кем выдан, код подразделения) | Шифрование при хранении (AES-256 или сертифицированные СКЗИ по результатам классификации ИСПДн), RBAC с минимально необходимым кругом лиц, маскирование в интерфейсах и логах; передача по сети — TLS/VPN, при необходимости СКЗИ |
| **ФИО, телефон, e-mail** клиента и сотрудника | Хранение в БД на территории РФ, разграничение доступа, логирование обращений к данным с определёнными сроками хранения логов и порядком проверки при инцидентах, шифрование архивных копий |
| **Реквизиты организации** (ИНН, ОГРН/ОГРНИП, КПП, юридический и фактический адрес, контакты, **банковские реквизиты** — р/с, банк, БИК) — ПДн/квази-ПДн для руководителей и ИП | Хранение в РФ, разграничение доступа, логирование обращений, маскирование в UI при массовом просмотре (например, частичное скрытие ИНН); банковские реквизиты — доступ только по ролям (финансы/договоры), не отображать в списках и массовых выборках |
| **Учётные данные** (логин, хэш пароля — клиент и сотрудник) | Только необратимый хэш (bcrypt/Argon2) с солью; разграничение доступа к таблицам учётных записей; политики сложности и смены паролей; защита от перебора (rate limit, блокировка при подозрительной активности); невозможность просмотра/выгрузки хэшей из UI; контроль админопераций с учётками |
| **Секрет TOTP сотрудника** (второй фактор при входе в админку/КПП) | Хранение в зашифрованном виде, привязка к учётной записи сотрудника; доступ только для операций проверки кода при аутентификации; запрет выгрузки и отображения в UI; см. артефакт «Аутентификация и авторизация» |
| **Данные от SSO-провайдера** (идентификатор субъекта sub, минимальный объём scope для клиента ФЛ) | Обработка в соответствии с 152-ФЗ (цели, сроки); пароль провайдера не хранится; связь sub с учётной записью клиента — только для идентификации при входе; минимальный объём данных от провайдера; см. артефакт «Аутентификация и авторизация» |
| **Реквизиты льготного документа** (тип документа, номер/реквизиты, даты выдачи и окончания) — ПДн по концептуальной модели | Те же меры, что и для образов документов: хранение в РФ, разграничение доступа, маскирование в интерфейсах при массовом просмотре; при необходимости — шифрование полей при хранении |
| **Образы документов** (скан льготного документа, файлы договоров) | Шифрование файлов в хранилище, хранение в РФ, доступ по ролям с минимально необходимым кругом лиц, в БД — только ссылка на объект; передача по сети — TLS |
| **Записи согласий** (тип, даты предоставления/отзыва, связь с клиентом) | Хранение в РФ, неизменяемый журнал (при отзыве — фиксация даты отзыва без удаления записи); связь типа согласия с целями и операциями обработки и сроками хранения ПДн после отзыва; разграничение доступа для аудита по 152-ФЗ |
| **ГРЗ и данные ТС** (госномер, марка, модель, цвет) — в связке с клиентом позволяют идентификацию; ГРЗ — особо чувствителен | Хранение в РФ, логирование доступа, ограничение выборок (запрет открытого поиска по всем номерам без служебной необходимости), маскирование части ГРЗ в списках и отчётах; марка/модель/цвет — тот же уровень доступа и ограничение массовых выгрузок, что и для ГРЗ |
| **Свободный текст** (обращения: тема, описание; сформированные уведомления; **причина статуса клиента** — причинаСтатуса, напр. задолженность, решение СБ) — потенциально содержат ПДн | Защита хранилища и логов, ограниченный доступ, поиск и выгрузки только по служебной необходимости; при необходимости — шифрование полей с текстом; причина статуса — доступ только у ролей, имеющих право менять статус клиента, маскирование в массовых списках |
| **Финансовые транзакции** (Платеж: сумма, способ оплаты, провайдерИд; Чек: сумма, фискальные реквизиты) | Логирование всех обращений к данным, ограничение массовых выгрузок, шифрование архивных копий, доступ только по ролям (финансы/биллинг); хранение в РФ |
| **Идентификатор плательщика во внешнем ПС** (Настройки оплаты.externalPayerId) | Хранение в РФ, по возможности — шифрование; разделение ролей (нет доступа у обычной поддержки); запрет отображения в интерфейсах; рассматривать как чувствительный технический идентификатор |
| **Журналы аудита и служебные идентификаторы** (последнееИзменениеСотрудникомИд, способ допуска, комментарий допуска, причины статусов и т.п.) | Защита логов, фиксированные сроки хранения, доступ только ограниченному кругу (ИБ, администраторы); использование для расследования инцидентов без избыточного раскрытия ПДн |

## Риски по сценариям доступа (Реестр Use Case)

С учётом реестра UC при проектировании мер учесть:

- **Массовый просмотр списков в админке** (UC-2.20 список профилей клиентов, UC-3.10 список ТС, UC-4.6 список обращений, UC-5.4 список уведомлений, UC-7.10 список бронирований, UC-8.13 список договоров, UC-12.11 список ПС, UC-15.4 список профилей сотрудников): таблица данных и мер уже предусматривает маскирование в UI для реквизитов организаций и ГРЗ; **дополнительно** — для списков профилей клиентов (ФИО, телефон, e-mail) применять маскирование или постраничный вывод без массовой выгрузки; логирование доступа к таким спискам.
- **Журнал въезда–выезда** (UC-12.12): массовые данные по ГРЗ и времени; ограничение выгрузок, доступ по ролям, фиксированные сроки хранения журнала в соответствии с политикой оператора.
- **Ручной допуск на КПП** (UC-12.13 внесение ТС в ПС вручную, UC-12.14 идентификация клиента вручную): аудит всех ручных допусков (кто, когда, комментарий); минимальный круг лиц с правом просмотра журнала ручных допусков; связь с мерами по строке «Журналы аудита и служебные идентификаторы».
- **Выгрузки (экспорт)**: регламентировать выгрузки ПДн и критичных данных (клиенты, ТС, обращения, платежи); логирование факта и объёма выгрузки, назначение ответственного; запрет выгрузки хэшей паролей и externalPayerId уже зафиксирован в таблице.

## Что уже соответствует 152-ФЗ (в рамках артефакта)

- Учёт категорий ПДн: паспортные данные, контактные данные клиентов и сотрудников, учётные записи, согласия, документы по льготам, реквизиты организаций, ГРЗ, тексты обращений и уведомлений, финансовые данные и идентификаторы плательщика выделены и сопоставлены мерам.
- Локализация: хранение ПДн и критичных данных в БД и файловом хранилище на территории РФ.
- Шифрование при хранении и передаче (TLS) для чувствительных категорий.
- Разграничение доступа (RBAC), минимально необходимый круг лиц для особо чувствительных данных.
- Хранение паролей только в виде необратимого хэша с солью.
- Фиксация согласий с возможностью отзыва и хранением истории для аудита.
- Секрет TOTP сотрудника и данные от SSO-провайдера учтены в таблице мер и согласованы с артефактом «Аутентификация и авторизация».

## Что не входит в артефакт, но требуется для полноты соответствия 152-ФЗ

Артефакт описывает **только данные и технические меры**. Для формального соответствия закону и подзаконным актам дополнительно необходимы:

| Требование | Где отразить |
|------------|--------------|
| Правовые основания и цели обработки для каждой категории ПДн (согласие, договор, закон) | Отдельный артефакт или раздел «Цели и основания обработки ПДн» |
| Классификация ИСПДн, тип угроз, уровень защищённости (УЗ-1/2/3), меры по ФСТЭК/ФСБ | Артефакт «Классификация ИСПДн и модель угроз» |
| Документы оператора: политика обработки ПДн, положение, перечень ПДн, регламент доступа, порядок реагирования на инциденты, журнал обращений субъектов, инструкции персоналу | Артефакт «Документы оператора ПДн» |
| Сертифицированные СКЗИ (где применимо по результатам классификации) | Уточнение в модели угроз и в технических требованиях к шифрованию |
| Процедуры реализации прав субъектов ПДн (доступ, исправление, блокирование, уничтожение) и сроки реакции (например, до 30 дней) | Регламент или раздел в артефакте «Документы оператора ПДн» |
| HTTPS, защита админ-доступов, резервное копирование с шифрованием и хранением в РФ, тестирование восстановления | Требования к инфраструктуре (спецификация NFR или отдельный артефакт) |

При появлении внутренних документов по ПДн (политика, модель угроз, приказы) их состав и ссылки рекомендуется зафиксировать в артефакте «Документы оператора ПДн» и связать с данной таблицей данных и мер.
