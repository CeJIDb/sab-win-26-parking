## Введение

Данный документ развивает базовую концептуальную модель системы управления парковкой и дополняет её **атрибутами ключевых сущностей**.

Цель: зафиксировать, какие данные система должна хранить и обрабатывать по каждой сущности на уровне концептуальной модели (без технических деталей реализации БД).

> ⚠️ Атрибуты приведены на основании артефактов проекта: концептуальной модели, контекстной диаграммы, протоколов интервью и связанных артефактов. Формулировки намеренно бизнес‑ориентированы.

Формат описания сущностей:

- краткое назначение;
- таблица атрибутов: название, описание, тип/домен значений, обязательность, источник.

## Справочники и перечисления (enum)

Для унификации доменов значений используются следующие перечисления (enum) и справочники.  
В этом документе:
- **перечисления (enum)** используются для статусов и технических доменов;
- **бизнес‑классификаторы** (`Тип клиента`, `Тип бронирования`, `Льготная категория`, `Юридическая форма`) трактуются как **справочники**, а не как жёстко вшитые enum (их значения могут расширяться по бизнес‑решению).

- **`Справочник: Тип клиента`**  
  - Значения: `ФЛ` (физическое лицо), `ЮЛ` (юридическое лицо).  
  - Используется в: `Клиент.Тип`.

- **`Enum: Статус эксплуатации`** (унифицированный для парковки/сектора/ПМ)  
  - Значения:  
    - `в эксплуатации` — объект доступен и используется;  
    - `закрыт на обслуживание` — временно недоступен из‑за работ;  
    - `выведен из эксплуатации` — больше не используется;  
    - `планируется к вводу` — зарезервирован, но ещё не введён в эксплуатацию.  
  - Используется в: `Парковка.Статус эксплуатации`, `Сектор.Статус эксплуатации`, `ПМ.Статус эксплуатации`.

- **`Справочник: Тип бронирования`**  
  - Значения: `разовое`, `долгосрочное`, `по договору`, `гостевое`.  
  - Используется в: `Бронирование.Тип бронирования`.

- **`Enum: Статус бронирования`**  
  - Значения: `черновик`, `ожидает оплаты`, `подтверждено`, `активно`, `завершено`, `отменено`, `просрочено`.  
  - Используется в: `Бронирование.Статус`.

- **`Enum: Статус парковочной сессии`**  
  - Значения: `активна`, `завершена`, `просрочена`, `в споре`.  
  - Используется в: `Парковочная сессия.Статус`.

- **`Enum: Статус платежа`**  
  - Значения: `инициализирован`, `в обработке`, `успешен`, `неуспешен`, `возврат инициирован`, `возврат выполнен`.  
  - Используется в: `Платеж.Статус`.

- **`Enum: Статус фискализации`**  
  - Значения: `отправлен в ОФД`, `фискализирован`, `ошибка`, `повторная отправка`.  
  - Используется в: `Чек.Статус фискализации`.

- **`Enum: Статус договора`**  
  - Значения: `черновик`, `на согласовании`, `активен`, `приостановлен`, `расторгнут`, `архив`.  
  - Используется в: `Договор.Статус`.

- **`Enum: Статус контрагента`** (унифицирован для клиента и организации)  
  - Значения: `активен`, `архив`, `заблокирован`.  
  - Используется в: `Клиент.Статус`, `Организация.Статус`.

- **`Enum: Статус обращения`**  
  - Значения: `новое`, `в работе`, `ожидает клиента`, `решено`, `отклонено`, `закрыто`.  
  - Используется в: `Обращение.Статус`.

- **`Enum: Статус доставки уведомления`**  
  - Значения: `в очереди`, `отправлено`, `доставлено`, `ошибка`.  
  - Используется в: `Уведомление.Статус доставки`.

- **`Enum: Тип обращения`**  
  - Значения: `вопрос`, `претензия`, `жалоба`, `предложение`, `отзыв`.  
  - Используется в: `Обращение.Тип`.

- **`Enum: Канал взаимодействия`** (унифицирован для обращений и уведомлений)  
  - Значения: `личный кабинет`, `мобильное приложение`, `сайт`, `звонок`, `e‑mail`, `чат`, `SMS`, `push`, `терминал`, `оператор КПП`.  
  - Используется в:  
    - `Обращение.Канал` (тип канала, по которому поступило обращение — подмножество значений),  
    - `Уведомление.Канал` (канал, по которому система отправила уведомление — в первую очередь `SMS`, `e‑mail`, `push`).

- **`Enum: Тип КПП`**  
  - Значения: `въезд`, `выезд`, `двусторонний`.  
  - Используется в: `КПП.Тип`.

- **`Enum: Тип тарифа`**  
  - Значения: `почасовой`, `суточный`, `абонемент`, `штрафной`.  
  - Используется в: `Тариф.Тип`.

- **`Справочник: Льготная категория`**  
  - Значения: `ОВЗ`, `пенсионер`, `ветеран`, `многодетная семья`, `сотрудник оператора`, `иная льгота`.  
  - Используется в: `Клиент.Льготная категория`, `Льготный документ.Льготная категория`, `Тариф.Льготная категория`.

- **`Справочник: Юридическая форма`**  
  - Значения: `ООО`, `АО`, `ПАО`, `ИП`, `НКО`, `иная`.  
  - Используется при описании типов юридических лиц (ООО, ИП и т.п.) в контрагентских данных.

Отдельные домены, реализованные как сущности‑справочники (`Тип зоны`, `Тип ТС`), используются по ссылке, а не как enum‑значения.

## Сущности и атрибуты

### `Парковка`

**Назначение.** Парковочный объект (площадка/комплекс), в рамках которого организованы парковочные места и действует тарифная политика.

| Атрибут              | Описание                                           | Тип / домен                         | Обязателен | Источник                         |
| ---------------------| ---------------------------------------------------| ------------------------------------| ---------- | --------------------------------|
| Идентификатор        | Уникальный идентификатор парковки                 | UUID / числовой                     | Да         | Концептуальная модель            |
| Наименование         | Название парковки                                 | Строка                              | Да         | Интервью, карточка проекта       |
| Адрес                | Физический адрес                                  | Строка                              | Да         | Контекстная диаграмма, интервью  |
| Тип                  | Тип парковки (открытая, крытая и т.п.)            | Перечисление                        | Нет        | Интервью                         |
| Описание             | Текстовое описание / примечания                   | Текст                               | Нет        | Интервью                         |
| Временной режим      | Часы работы                                       | Строка / интервал времени           | Нет        | Контекстная диаграмма            |
| Статус эксплуатации  | Эксплуатационный статус парковки: активна, закрыта на реконструкцию и т.п. | Перечисление                        | Да         | Операционная модель              |

**Связи:**

- `Парковка` —(1:N)→ `Сектор` — одна парковка содержит несколько секторов.
- `Парковка` —(1:N)→ `КПП` — у парковки может быть несколько КПП (точек въезда/выезда).

---

### `Сектор`

**Назначение.** Логически/физически выделенная часть парковки (зона, этаж и т.п.).

| Атрибут              | Описание                                           | Тип / домен                         | Обязателен | Источник                         |
| ---------------------| ---------------------------------------------------| ------------------------------------| ---------- | --------------------------------|
| Идентификатор        | Уникальный идентификатор сектора                  | UUID / числовой                     | Да         | Концептуальная модель            |
| Парковка             | Ссылка на парковку                                | Ссылка на `Парковка`                | Да         | Концептуальная модель            |
| Наименование         | Человекочитаемое обозначение (А, B, Этаж 1 и т.п.)| Строка                              | Да         | Схема парковки AS‑IS             |
| Тип зоны             | Краткосрочная, долгосрочная, ОВЗ, мото+резерв и т.п.| Ссылка на `Тип зоны`             | Да         | Контекстная диаграмма, AS‑IS     |
| Статус эксплуатации  | Эксплуатационный статус сектора: активен, закрыт на обслуживание и т.п. | Перечисление | Да         | Операционная модель              |

**Связи:**

- `Сектор` ←(N:1)— `Парковка` — каждый сектор принадлежит одной парковке.
- `Сектор` —(1:N)→ `ПМ` — сектор включает множество парковочных мест.
- `Сектор` ←(N:1)— `Тип зоны` — каждый сектор имеет один тип зоны.

---

### `Тип зоны`

**Назначение.** Справочник типов зон парковки, определяющий бизнес‑режим использования территории и применимые правила тарификации.

| Атрибут              | Описание                                           | Тип / домен                         | Обязателен | Источник                         |
| ---------------------| ---------------------------------------------------| ------------------------------------| ---------- | --------------------------------|
| Идентификатор        | Уникальный идентификатор типа зоны                | UUID / числовой                     | Да         | Концептуальная модель            |
| Код                  | Внутренний код типа зоны (например, `SHORT_TERM`, `LONG_TERM`, `OVZ`, `MOTO_RESERVE`) | Строка | Да | Внутренний справочник            |
| Наименование         | Человекочитаемое название (краткосрочная, долгосрочная, ОВЗ, мото+резерв и т.п.) | Строка | Да | Контекстная диаграмма, схема AS‑IS |
| Описание             | Описание бизнес‑режима зоны и общих правил использования | Текст                         | Нет        | Протоколы, бизнес‑правила       |

**Связи:**

- `Тип зоны` —(1:N)→ `Сектор` — один тип зоны может быть назначен многим секторам.
- `Тип зоны` —(1:N)→ `Разрешённый тип ТС в типе зоны` — задаёт, какие типы ТС разрешены в этой зоне.
- `Тип зоны` —(1:N)→ `Применимость тарифа к типу зоны` — один тип зоны может иметь несколько записей применимости тарифов.

---

### `Тип ТС`

**Назначение.** Справочник типов/категорий транспортных средств, используемый для зонирования и тарификации.

| Атрибут              | Описание                                           | Тип / домен                         | Обязателен | Источник                         |
| ---------------------| ---------------------------------------------------| ------------------------------------| ---------- | --------------------------------|
| Идентификатор        | Уникальный идентификатор типа ТС                  | UUID / числовой                     | Да         | Концептуальная модель            |
| Код                  | Внутренний код типа ТС (например, `CAR`, `MOTO`, `MUNICIPAL`) | Строка                      | Да         | Внутренний справочник            |
| Наименование         | Название типа ТС (легковой, мотоцикл, муниципальная техника и т.п.) | Строка                | Да         | Контекстная диаграмма           |
| Описание             | Дополнительное описание особенностей категории    | Текст                               | Нет        | Протоколы                       |

**Связи:**

- `Тип ТС` —(1:N)→ `Разрешённый тип ТС в типе зоны` — используется при определении допустимых типов ТС в зоне.
- `Тип ТС` —(1:0..N)→ `ТС` — задаёт категорию конкретных транспортных средств.

---

### `Разрешённый тип ТС в типе зоны`

**Назначение.** Связующая сущность, задающая, какие типы ТС разрешены для каждого типа зоны (нормализация связи «Тип зоны → множество Типов ТС»).

| Атрибут              | Описание                                           | Тип / домен                         | Обязателен | Источник                         |
| ---------------------| ---------------------------------------------------| ------------------------------------| ---------- | --------------------------------|
| Тип зоны             | Тип зоны, для которого задаётся разрешённый тип ТС | Ссылка на `Тип зоны`               | Да         | Концептуальная модель            |
| Тип ТС               | Тип транспортного средства, разрешённый в этой зоне | Ссылка на `Тип ТС`                | Да         | Концептуальная диаграмма         |

> Для каждого `Тип зоны` может быть несколько связанных записей `Разрешённый тип ТС в типе зоны` (разные `Тип ТС`). Кардинальность: `Тип зоны` (1) — (N) `Разрешённый тип ТС в типе зоны` — (N) (1) `Тип ТС`.

**Связи:**

- `Разрешённый тип ТС в типе зоны` ←(N:1)— `Тип зоны`.
- `Разрешённый тип ТС в типе зоны` ←(N:1)— `Тип ТС`.

---

### `Применимость тарифа к типу зоны`

**Назначение.** Нормализующая сущность для связи «`Тип зоны` — `Тариф`» многие‑ко‑многим: задаёт, какие тарифы применимы к каким типам зон.

| Атрибут    | Описание                                           | Тип / домен          | Обязателен | Источник              |
| ---------- | -------------------------------------------------- | -------------------- | ---------- | ----------------------|
| Тип зоны   | Тип зоны, к которому применяется тариф            | Ссылка на `Тип зоны` | Да         | Концептуальная модель |
| Тариф      | Тариф, применимый к указанному типу зоны          | Ссылка на `Тариф`    | Да         | Концептуальная модель |

**Связи:**

- `Применимость тарифа к типу зоны` ←(N:1)— `Тип зоны` — один тип зоны может иметь несколько применимых тарифов.
- `Применимость тарифа к типу зоны` ←(N:1)— `Тариф` — один тариф может применяться к нескольким типам зон.

---

### `ПМ` (Парковочное место)

**Назначение.** Конкретное физическое место для размещения транспортного средства.

| Атрибут              | Описание                                           | Тип / домен                         | Обязателен | Источник                         |
| ---------------------| ---------------------------------------------------| ------------------------------------| ---------- | --------------------------------|
| Идентификатор        | Уникальный идентификатор ПМ                       | UUID / числовой                     | Да         | Концептуальная модель            |
| Сектор               | Ссылка на сектор                                  | Ссылка на `Сектор`                  | Да         | Концептуальная модель            |
| Номер места          | Номер/обозначение на схеме                        | Строка / целое                      | Да         | Схема парковки AS‑IS             |
| Бронирование | Ссылка на действующее долгосрочное бронирование, закрепляющее место за клиентом (**операционный атрибут/проекция**: поддерживается логикой приложения на основе записей `Бронирование`) | Ссылка на `Бронирование` | Нет        | Интервью ЮЛ, концепция долгосрочных бронирований |
| Статус эксплуатации  | Эксплуатационный статус места: доступно, заблокировано (техобслуживание), выведено из эксплуатации и т.п. | Перечисление | Да         | Операционная модель              |

> Тип зоны и разрешённые типы ТС для ПМ наследуются от связанного `Сектор` (зонирование и ограничения по типам ТС задаются на уровне сектора; отдельные отклонения моделируются как отдельные секторы).  
> Операционные состояния «зарезервировано» и «занято» выводятся из активных `Бронирование` и `Парковочная сессия` и не хранятся как отдельные атрибуты ПМ; `Статус эксплуатации` описывает именно пригодность места к использованию (доступно/заблокировано/выведено из эксплуатации и т.п.).

**Связи:**

- `ПМ` ←(N:1)— `Сектор` — каждое место принадлежит одному сектору.
- `ПМ` —(1:N)→ `Бронирование` — на одно место может быть несколько бронирований в разные периоды.
- `ПМ` ←(N:1)— `Тариф` (опционально) — место может наследовать/иметь свой тариф.

---

### `Бронирование`

**Назначение.** Запись о заранее зарезервированном использовании парковочного пространства на период: **краткосрочное бронирование резервирует слот в секторе**, а **долгосрочное/договорное — конкретное парковочное место (ПМ) для конкретного ТС**.  
`Бронирование` является **мастер‑сущностью для финансов и прав доступа**: все суммы, платежи, чеки и договорные ограничения (квоты, тарифы, льготы) привязаны именно к бронированию, а не к парковочной сессии.

| Атрибут                | Описание                                         | Тип / домен                         | Обязателен | Источник                         |
| -----------------------| ----------------------------------------------- | ------------------------------------| ---------- | --------------------------------|
| Идентификатор          | Уникальный идентификатор бронирования           | UUID / числовой                     | Да         | Концептуальная модель            |
| ТС                     | Транспортное средство, для которого создано бронирование | Ссылка на `ТС`                  | Да         | Протоколы, договорённости по привязке к ТС |
| Сектор                 | Целевая зона/сектор, в котором резервируется слот (для краткосрочных бронирований; для долгосрочных определяется через `ПМ`) | Ссылка на `Сектор` | Да  | Концептуальная модель, интервью  |
| ПМ                     | Ссылка на конкретное парковочное место (для долгосрочных/договорных бронирований; для краткосрочных может быть заполнено при фактическом назначении места) | Ссылка на `ПМ` | Нет | Концептуальная модель, интервью |
| Договор                | Договор, в рамках которого создано долгосрочное/договорное бронирование (для краткосрочных не заполняется) | Ссылка на `Договор` | Нет | Концептуальная модель, интервью |
| Дата начала            | Дата/время начала брони                         | Дата‑время                          | Да         | Контекстная диаграмма            |
| Длительность           | Планируемая длительность бронирования           | Интервал                            | Да         | Контекстная диаграмма, интервью  |
| Тип       | Разовое, долгосрочное, по договору и т.п.       | Перечисление (`Enum: Тип бронирования`) | Да    | Интервью, протоколы              |
| Статус                 | Создано, подтверждено, активно, завершено, отменено и т.п. | Перечисление (`Enum: Статус бронирования`) | Да | Бизнес‑правила                   |
| Тариф                  | Ссылка на примененный тариф                     | Ссылка на `Тариф`                   | Да         | Концептуальная модель            |
| Сумма к оплате         | Итоговая сумма за бронирование (**результат тарификации** по применённому `Тариф`, дате начала, длительности и льготам; хранится как зафиксированный результат расчёта) | Денежный | Нет        | Контекстная диаграмма, правила тарификации |

Связь с конкретной парковкой и типом зоны для краткосрочных бронирований задаётся через атрибут `Сектор` → `Парковка`/`Тип зоны`; для долгосрочных/договорных — через `ПМ` → `Сектор` → `Парковка`/`Тип зоны`. Отдельных атрибутов "Зона"/"Тип зоны" в бронировании не хранится, они определяются по связанным объектам. Для любого бронирования действует бизнес‑правило: **`Сектор` обязателен всегда; `ПМ` обязателен только для долгосрочных/договорных бронирований (для краткосрочных может оставаться пустым)**. Клиент определяется через `ТС` → `Клиент`.  
**Инвариант:** все платежи, суммы и фискальные документы по парковке проводятся в контексте конкретного `Бронирование`; `Парковочная сессия` используется только как факт времени/КПП и, при необходимости, аналитический след суммы по факту, но не как финансовый мастер.

**Связи:**

- `Бронирование` ←(0..N:1)— `ТС` — одно ТС может иметь ноль или много связанных бронирований; каждое бронирование относится к одному ТС.
- `Бронирование` ←(0..N:1)— `Сектор` — бронирование может ссылаться на сектор (для краткосрочных бронирований); у сектора может быть много бронирований или ни одного.
- `Бронирование` ←(0..N:1)— `ПМ` — бронирование может закреплять конкретное парковочное место (для долгосрочных/договорных бронирований) или не иметь ПМ (для краткосрочных).
- `Бронирование` ←(0..N:1)— `Договор` — долгосрочные/договорные бронирования оформляются в рамках договора; краткосрочные могут не иметь связи с договором.
- `Бронирование` —(1:0..N)→ `Парковочная сессия` — одно бронирование может не породить ни одной сессии или несколько сессий; каждая сессия создаётся в контексте конкретного бронирования (в т.ч. созданного на лету при въезде без предварительной брони; см. ADR‑002, Option D).
- `Бронирование` ←(0..N:1)— `Тариф` — стоимость бронирования рассчитывается по тарифу; один тариф может использоваться во множестве бронирований.
- `Бронирование` —(1:1..N)→ `Платеж` — бронирование может быть оплачено одним или несколькими платежами.
- `Бронирование` —(0..1:0..N)→ `Уведомление` — с бронированием может быть связано ноль или несколько уведомлений.
- `Бронирование` —(0..1:0..N)→ `Обращение` — по бронированию может не быть ни одного обращения или несколько обращений.

### `Парковочная сессия`

**Назначение.** Фактический период нахождения ТС на парковке (от въезда до выезда), зафиксированный для конкретного бронирования: используется для учёта времени пребывания и анализа работы КПП; финансовый результат фиксируется на уровне `Бронирование` и связанных `Платеж`. `Парковочная сессия` не имеет собственных прямых ссылок на `ТС`, `Тариф`, `Договор`, `Клиент`, а использует связанные данные только через `Бронирование`.  
При необходимости `Парковочная сессия` может хранить **аналитическую сумму по факту**, фиксирующую «как посчитали по факту» по результатам тарификации по фактической длительности сессии; при этом мастер‑суммой для платёжных обязательств и чеков остаётся сумма на уровне `Бронирование`.

| Атрибут                | Описание                                         | Тип / домен                         | Обязателен | Источник                         |
| -----------------------| ----------------------------------------------- | ------------------------------------| ---------- | --------------------------------|
| Идентификатор          | Уникальный идентификатор сессии                 | UUID / числовой                     | Да         | Концептуальная модель            |
| Бронирование           | Бронирование, в рамках которого идёт сессия     | Ссылка на `Бронирование`            | Да         | Концептуальная модель, ADR‑002   |
| Время въезда           | Зафиксированное время въезда                    | Дата‑время                          | Да         | Контекстная диаграмма, СКУД      |
| Время выезда           | Зафиксированное время выезда                    | Дата‑время                          | Нет        | Контекстная диаграмма, СКУД      |
| КПП въезда             | КПП, через который выполнен въезд               | Ссылка на `КПП`                     | Нет        | Концептуальная модель            |
| КПП выезда             | КПП, через который выполнен выезд               | Ссылка на `КПП`                     | Нет        | Концептуальная модель            |
| Способ допуска         | Как было принято решение о допуске ТС: автоматически по правилам доступа или вручную сотрудником | Перечисление (`Enum: Способ допуска сессии`) | Да | Бизнес‑правила, UI КПП          |
| Сотрудник              | Сотрудник, принявший решение о ручном допуске/коррекции (для автоматических сценариев не заполняется) | Ссылка на `Сотрудник` | Нет | Концептуальная модель, UI КПП   |
| Комментарий допуска    | Комментарий сотрудника о причине ручного допуска/коррекции | Текст                        | Нет        | Протоколы, UI КПП               |
| Статус                 | Активна, завершена, просрочена, в споре и т.п.  | Перечисление (`Enum: Статус парковочной сессии`) | Да | Бизнес‑правила                   |

Бизнес‑правило реализации (ADR‑002, Option D): **каждая `Парковочная сессия` должна иметь ссылку на `Бронирование` (в т.ч. созданное на лету при въезде без предварительной брони)**. Информация о ТС, секторе/ПМ, тарифе и договоре для сессии всегда берётся только через связанное `Бронирование`; прямых связей с этими сущностями у `Парковочная сессия` нет.

**Связи:**

- `Парковочная сессия` ←(N:1)— `Бронирование` — каждая сессия относится к одному бронированию; одно бронирование может иметь ноль или несколько сессий.
- `Парковочная сессия` ←(0..1:1)— `КПП` (КПП въезда) и `КПП` (КПП выезда) — сессия хранит КПП въезда и выезда.
- `Парковочная сессия` ←(0..N:0..1)— `Сотрудник` — при ручных сценариях допуска сессия хранит, какой сотрудник принял решение и оставил комментарий.

## Связи (обобщенный список)

Ниже приведены основные связи между сущностями концептуальной модели с указанием кардинальностей. Они действуют как для базовой концептуальной модели, так и для варианта с атрибутами в данном документе.

- **`Парковка` —(1:N)→ `Сектор`** — структурирование территории на зоны/этажи.
- **`Сектор` —(1:N)→ `ПМ`** — размещение конкретных парковочных мест в секторе.
- **`Парковка` —(1:N)→ `КПП`** — у парковки несколько точек въезда/выезда.
- **`КПП` —(1:0..N)→ `Парковочная сессия`** — КПП может выступать точкой въезда и/или выезда для множества сессий или не использоваться.
- **`ПМ` —(1:0..N)→ `Бронирование`** — одно место может многократно бронироваться в разные периоды (для долгосрочных/договорных бронирований); краткосрочные бронирования могут ссылаться только на сектор.
- **`Клиент` —(1:0..N)→ `ТС`** — у клиента может быть ноль или несколько транспортных средств.
- **`Клиент` —(1:0..N)→ `Договор`** — у клиента (ФЛ/ЮЛ) может быть ноль или несколько договоров; долгосрочные бронирования оформляются в рамках договора клиента.
- **`Договор` —(0..1:1..N)→ `Бронирование`** — один договор может иметь ноль или много связанных бронирований; каждое долгосрочное бронирование (>24 часов) может быть привязано не более чем к одному действующему договору.
- **`Бронирование` —(0..1:0..N)→ `Парковочная сессия`** — из одного бронирования может не возникнуть ни одной или возникнуть несколько сессий; в реализуемом варианте каждая сессия должна иметь привязку к бронированию (в т.ч. созданному при въезде).
- **`ТС` —(1:0..N)→ `Бронирование`** — бронирования создаются для конкретного ТС.
- **`Тариф` —(1:0..N)→ `Бронирование`** — тариф применяется при расчёте стоимости бронирований.
- **`Бронирование` —(0..1:0..N)→ `Платеж`** — бронирование может оплачиваться одним или несколькими платежами либо не иметь платежей.
- **`Платеж` —(1:0..1)→ `Чек`** — успешный платеж порождает не более одного фискального чека; при неуспешном платеже чек не формируется.
- **`Обращение` ←(N:1)— `Клиент` / `Сотрудник`** — обращение всегда принадлежит клиенту и обрабатывается сотрудником.
- **`Обращение` ←(N:1)— `Бронирование` / `Платеж` / `Чек` (опционально)** — обращение может быть привязано к конкретному объекту.
- **`Уведомление` ←(N:1)— `Клиент`** — уведомления адресуются клиенту и отправляются по его контактным данным.
- **`Уведомление` ←(N:1)— `Бронирование` / `Платеж` / `Чек` / `Обращение` (опционально)** — уведомление связано с конкретным событием или документом.

Дополнительные связи, специфичные для этого документа:

- **`Тип зоны` —(1:N)→ `Сектор`** — каждый сектор имеет один тип зоны.
- **`Тип зоны` —(1:N)→ `Разрешённый тип ТС в типе зоны` —(N:1)→ `Тип ТС`** — задаёт, какие типы ТС разрешены для каждого типа зоны.
- **`Организация` —(0..1:1)→ `Клиент`** — организация может быть привязана к одному клиенту типа ЮЛ или не использоваться.
 - **`Тип зоны` —(1:N)→ `Применимость тарифа к типу зоны` —(N:1)→ `Тариф`** — задаёт, какие тарифы применимы к каким типам зон.
 - **`Тип ТС` —(1:N)→ `ТС`** — каждый тип/категория ТС может быть назначен множеству транспортных средств.

- Уточнить атрибуты по результатам следующих протоколов интервью и согласований с заказчиком.
- Для атрибутов с типом "Структура" и "Полиморфная ссылка" разработать отдельные логические схемы.
- На следующем шаге перенести модель в логический уровень (ER‑диаграмма) с нормализацией и уточнением ключей.
