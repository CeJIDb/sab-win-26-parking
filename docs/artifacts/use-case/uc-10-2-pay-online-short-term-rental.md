| --- | --- |
| **ID** |  |
| **Автор** | Командное |
| **Название** | Сценарий интеграции. Оплатить онлайн (краткосрочная аренда) |
| **Действующее лицо** | Клиент ФЛ, Система, Платежный провайдер, ОФД, Техническая поддержка |

## Предусловия

- Существует активная парковочная сессия, привязанная к номеру ТС клиента.
- Клиент авторизован в системе и имеет доступ к ЛК.
- Тариф для сессии определён и доступен для расчёта.
- Платёжный шлюз доступен и интегрирован с системой.
- Система интегрирована с ОФД.
- Сессия не имеет статуса «Оплачено».

## Триггер

Клиент в ЛК инициирует команду «Оплатить сессию», нажимая кнопку «Оплата»→[Ok].

## Постусловие

- Сессия имеет статус «Оплачено».
- Платеж имеет статус «Успешен» и содержит идентификатор транзакции от платежной системы.
- Чек имеет статус «Зарегистрирован» и содержит номер, присвоенный ОФД.
- В системе существует запись о завершении оплаты с датой и временем.
- Клиент получил уведомление об успешной оплате и наличии чека в ЛК.
- Шлагбаум (СКУД) разрешает выезд для данного ТС.

## Основной поток

1. ****Система создает [платеж] со следующими атрибутами:
   - суммаКОплате = Тариф * Продолжительность парковки (в копейках)
   - датаСоздания = текущая дата и время
   - статус = Инициализирован
   - идентификаторСессии = [ID активной сессии]
   - идентификаторКлиента = [ID клиента]
2. Система обновляет [сессия] со следующими атрибутами:
   - статус = Ожидание оплаты
   - датаОбновления = текущая дата и время
3. Система передаёт в Платёжную систему запрос на регистрацию операции оплаты (по уже созданной в Системе записи [платеж]) с параметрами:
   - сумма = суммаКОплате
   - ID сессии = [ID активной сессии] (как invoice_id)
   - callbackURL = [URL для возврата]
   - данные клиента = [ФИО, email, если есть]
4. Система перенаправляет пользователя на страницу оплаты Платежной системы.
5. После оплаты клиентом на стороне Платёжной системы Система получает от Платёжной системы подтверждение успешного завершения операции.
6. Система извлекает из ответа Платёжной системы однозначный статус успешной оплаты и номер транзакции у шлюза для последующей записи в [платеж].
7. Система обновляет [платеж] со следующими атрибутами:
   - статус = Успешен
   - датаОплаты = текущая дата и время
   - идентификаторТранзакции = [номер транзакции от шлюза]
8. Система обновляет [сессия] со следующими атрибутами:
   - статус = Оплачено
   - датаОплаты = текущая дата и время
9. Система создает [чек] со следующими атрибутами:
   - суммаЧека = суммаКОплате
   - датаСоздания = текущая дата и время
   - статус = Ожидание
   - идентификаторПлатежа = [ID платежа]
10. Система передаёт в ОФД запрос на создание чека с параметрами:
   - сумма = суммаЧека
   - дата = текущая дата и время
   - тип операции = Приход
   - ИНН = [ИНН организации]
   - ККТ = [идентификатор ККТ]
   - ФИО покупателя = [ФИО клиента, если есть]
11. Система получает от ОФД ответ с номером чека.
12. Система получает от ОФД статус регистрации чека.
13. Система обновляет [чек] со следующими атрибутами:
   - номерЧека = [номер от ОФД]
   - статус = Зарегистрирован
   - датаРегистрации = текущая дата и время
14. Система отправляет клиенту уведомление по каналу СМС, e-mail или push с текстом:
    "Оплата парковки успешно завершена. Чек №[номер] доступен в ЛК."
15. Система завершает сценарий.

## Расширение

1a. Система не может создать [платеж] (ошибка расчёта суммы, нарушение целостности данных или иные блокирующие ошибки до шага 2).

1a 1. Система не обновляет [сессия] в статус «Ожидание оплаты» и не обращается к Платёжной системе.
1a 2. Сценарий прерывается.
1a 3. Клиент видит в ЛК сообщение о невозможности начать оплату; предлагается обновить экран или обратиться в поддержку.
1a 4. Система при повторяемости фиксирует событие для технической поддержки (в оповещении — без избыточных персональных данных).

2a. После создания [платеж] Система не может зафиксировать обновление [сессия] в статус «Ожидание оплаты» (ошибка записи, блокировка, таймаут внутренней транзакции).

2a 1. Система откатывает или аннулирует [платеж] этой попытки по политике и не обращается к Платёжной системе.
2a 2. Сценарий прерывается.
2a 3. Клиент видит сообщение о сбое при подготовке оплаты; предлагается повторить позже.
2a 4. Система направляет оповещение в техническую поддержку / эксплуатацию.

3a. Платёжная система не отвечает на запрос создания платежа (таймаут, недоступность контура) на шаге 3.

3a 1. Система не получает подтверждения создания операции; [платеж] остаётся в статусе «Инициализирован» или переводится в «Ошибка» по политике.
3a 2. Система не перенаправляет клиента на страницу оплаты.
3a 3. Сценарий прерывается.
3a 4. Клиент видит сообщение о временной недоступности онлайн-оплаты.
3a 5. Система направляет оповещение в техническую поддержку / эксплуатацию.

4a. Система не получила от Платёжной системы действующий адрес/средство перехода к оплате.

4a 1. Система не выполняет перенаправление клиента; при необходимости отменяет частично созданную операцию у Платёжной системы по регламенту.
4a 2. [сессия] остаётся в статусе «Ожидание оплаты» или возвращается к предыдущему допустимому статусу по политике.
4a 3. Клиент видит сообщение о невозможности открыть страницу оплаты.
4a 4. Система фиксирует инцидент интеграции.

5a. Платёжная система не подтвердила успешную оплату в согласованный срок (таймаут ожидания результата после шага 4).

5a 1. Система фиксирует таймаут в журнале.
5a 2. Система обновляет [платеж]: статус = «Ошибка» или «Отменён» по политике; датаОплаты и идентификаторТранзакции остаются незаполненными при отсутствии подтверждения.
5a 3. Система возвращает [сессия] к допустимому статусу (сессия не «Оплачено») по политике.
5a 4. Клиент получает уведомление о неподтверждённой оплате; при политике продукта предлагается повторить попытку позже.
5a 5. Система направляет оповещение в техническую поддержку.

6a. Ответ Платёжной системы неполон: нет однозначного признака успеха и/или идентификатора транзакции.

6a 1. Система не переводит [платеж] в «Успешен» автоматически; запрашивает статус у Платёжной системы по регламенту либо инициирует ручную проверку.
6a 2. Клиент видит сообщение об уточнении статуса оплаты; повторное списание не рекомендуется до обновления статуса.
6a 3. Система уведомляет эксплуатацию до финального резолва.

8a. Платёжная система сообщила об отказе в оплате или клиент отменил оплату на стороне Платёжной системы.

8a 1. Система обновляет [платеж]: статус = «Отклонён» / «Ошибка» по политике; идентификаторТранзакции заполняется только если передан провайдером.
8a 2. Система не переводит [сессия] в «Оплачено»; доступ к выезду / СКУД — по правилам продукта.
8a 3. Клиент видит, что оплата не завершена.
8a 4. Сценарий завершается неуспешно по оплате.

11a. ОФД не отвечает или возвращает отказ при запросе регистрации чека при уже успешном [платеж].

11a 1. Система не откатывает успех оплаты у Платёжной системы; [чек] остаётся в статусе «Ожидание» или «Ошибка» по политике.
11a 2. Система ставит повтор обращения к ОФД в очередь по регламенту.
11a 3. Клиент видит, что оплата принята, фискальный чек формируется.
11a 4. Система оповещает эксплуатацию при исчерпании повторов или при критичном отказе ОФД.

12a. Данные ответа ОФД не согласуются с [чек] и [платеж] (номер, сумма, статус неоднозначны).

12a 1. Система не переводит [чек] в «Зарегистрирован» автоматически; выполняет сверку с ОФД; при неудаче — ручная обработка по политике.
12a 2. Клиент получает безопасное сообщение об уточнении статуса чека.
12a 3. Система открывает приоритетный инцидент (риск расхождений с кассой).

13a. Ответ ОФД получен, но Система не может надёжно зафиксировать обновление [чек] на шаге 13 (сбой хранилища, конфликт версий, прерванная транзакция).

13a 1. Система не считает [чек] согласованно «Зарегистрирован» в данных ЛК до восстановления записи; ставит повтор записи или постановку сверки с ОФД по регламенту.
13a 2. Клиент видит, что оплата принята, реквизиты чека уточняются; номер из ОФД не показывается как окончательный до подтверждения Системой.
13a 3. Система оповещает эксплуатацию; выполняется сверка с ОФД до согласованного состояния.

14a. Канал СМС / Email / push недоступен или доставка уведомления не удалась.

14a 1. Для клиента подтверждением остаются ЛК и записи [платеж] и [чек] в Системе.
14a 2. Система повторяет отправку уведомления с тем же ключом идемпотентности по регламенту.
14a 3. При исчерпании повторов Система оповещает эксплуатацию; сценарий по деньгам и фискализации считается завершённым с точки зрения оплаты.
