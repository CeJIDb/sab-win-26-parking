## UC-12.2 «Создать бронирование автоматически на въезде»

### 1. Область действия / контекст

Автоматическое создание Бронирования при въезде ТС на парковку, если политика парковки допускает автозапись и у клиента есть право на въезд, но явного предварительного Бронирования нет.

### 2. Акторы

- **Система** – принимает решение о создании Бронирования и создает его.

### 3. Цель

Создать автоматическое Бронирование для ТС в момент въезда, чтобы:

- зафиксировать право клиента на использование парковки в определенном интервале;
- привязать последующую ПС к Бронированию для корректной тарификации и отчетности.

### 4. Предусловия

- сценарий запущен как продолжение UC-12.1 «Пройти автоматическую идентификацию на въезде» (расширение 3а);
- для ТС нет активного Бронирования;
- бизнес‑правила парковки разрешают автоматическое создание Бронирования при въезде.

### 5. Триггер

- Система инициирует создание автоматического Бронирования.

### 6. Постусловия / результат

- Создано автоматическое Бронирование, привязанное к ТС.

### 7. Основной поток

1. Система ищет краткосрочный Тариф, соответствующий типу ТС, определённому в UC‑12.1.
2. Система определяет допустимые Типы зон для найденного Тарифа и типа ТС.
3. Система ищет свободное ПМ в Секторах с допустимыми Типами зон в соответствии со стратегией выбора:
   - ближайший к текущему КПП;
   - при равном расстоянии — в наименее загруженном Секторе;
   - при наличии приоритетных зон для типа клиента (например, для клиентов с ОВЗ — «ОВЗ» зоны) — в приоритетном Секторе.
4. Система создает Бронирование с атрибутами:
   - идентификатор;
   - ТС;
   - Сектор;
   - ПМ;
   - дата‑время начала = дата и время события, полученные от СКУД в UC‑12.1;
   - тип = «автоматическое»;
   - статус = «активно»;
   - Тариф.
5. Система изменяет ПМ:
   - «Зарезервировано» = 1.
6. Система завершает выполнение сценария и возвращает выполнение в UC‑12.1 на шаг 4 Основного потока.

### 8. Расширения

**2а. Тариф не найден**

2а1. Система добавляет в «Последние события КПП» и «Журнал въезда‑выезда» на панели КПП запись:

- время события = дата и время события, полученные от СКУД в UC‑12.1;
- ГРЗ = госномер ТС;
- направление = «Въезд»;
- статус = «Въезд отклонен (нет подходящего тарифа)».
2а2. Система передает на Дисплей въезда сообщение:
- ГРЗ;
- «Доступ запрещен»;
- «Нет тарифа для этого типа ТС. Обратитесь к охране».
2а3. Система завершает выполнение сценария.

**3а. Тип зоны не найден**

3а1. Система добавляет в «Последние события КПП» и «Журнал въезда‑выезда» на панели КПП запись:

- время события = дата и время события, полученные от СКУД в UC‑12.1;
- ГРЗ = госномер ТС;
- направление = «Въезд»;
- статус = «Въезд отклонен (нет подходящей зоны)».
3а2. Система передает на Дисплей въезда сообщение:
- ГРЗ;
- «Доступ запрещен»;
- «Нет доступной зоны для этого типа ТС. Обратитесь к охране».
3а3. Система завершает выполнение сценария.

**4а. Нет свободного ПМ в Секторах, доступных по Тарифу**

4а1. Система добавляет в «Последние события КПП» и «Журнал въезда‑выезда» на панели КПП запись:

- время события = дата и время события, полученные от СКУД в UC‑12.1;
- ГРЗ = госномер ТС;
- направление = «Въезд»;
- статус = «Въезд отклонен (нет свободных мест)».
4а2. Система передает на Дисплей въезда сообщение:
- ГРЗ;
- «Доступ запрещен»;
- «Нет свободных мест для этого типа ТС. Обратитесь к охране».
4а3. Система завершает выполнение сценария.
