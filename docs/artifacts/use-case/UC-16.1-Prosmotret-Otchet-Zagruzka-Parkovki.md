## UC-16.1 «Просмотреть отчет по загрузке парковки»

### 1. Область действия / контекст

Аналитический сценарий для Управляющего/Владельца парковки: просмотр отчетов по загрузке парковки за выбранный период (по дням, часам, зонам, тарифам), для принятия управленческих решений по тарифам, квотам и операционной деятельности.

### 2. Акторы

- **Управляющий парковки** – анализирует загрузку для оперативного управления.
- **Владелец парковки** – анализирует долгосрочные тренды загрузки и доходности.
- **ИС парковки** – накапливает данные о ПС, бронированиях, оплатах и строит отчеты.

### 3. Цель

Управляющий/Владелец хочет получить наглядную и достоверную аналитику по загрузке парковки, чтобы:
- оценивать текущую и историческую загрузку;
- выявлять пиковые/провальные периоды;
- принимать решения о корректировке тарифов, квот и маркетинговых акций.

### 4. Предусловия

- В системе накапливаются данные о ПС, бронированиях и, при необходимости, отказах во въезде (для оценки неудовлетворенного спроса).
- Пользователь авторизован с ролью, имеющей доступ к аналитическим отчетам.

### 5. Триггер

- Управляющий/Владелец открывает раздел «Аналитика» и выбирает отчет «Загрузка парковки».

### 6. Постусловия / результат

- Пользователь получает отчет по загрузке парковки за выбранный период с возможностью детализации до нужного уровня (по зонам, часам, типам клиентов).
- На основе отчета могут быть инициированы другие сценарии (например, изменение тарифов UC-13.1–13.6).

### 7. Основной поток

1. Управляющий/Владелец открывает раздел «Аналитика» в административном интерфейсе.
2. Пользователь выбирает тип отчета «Загрузка парковки».
3. Система запрашивает параметры отчета:
   - период (дата начала/конца);
   - уровень детализации (по дням, по часам, по сменам);
   - разрезы (по зонам, по типу тарифов, по типам клиентов ФЛ/ЮЛ).
4. Пользователь задает параметры и подтверждает построение отчета.
5. Система извлекает необходимые данные:
   - данные о ПС (время начала/окончания, зона, тип клиента, договор/разовый заезд);
   - данные о бронированиях (в т.ч. незавершенных/неиспользованных, если требуется);
   - данные об отказах во въезде (если фиксируются).
6. Система рассчитывает показатели загрузки:
   - средний и максимальный процент занятых мест по периодам;
   - количество ПС и бронирований;
   - плотность потока въездов/выездов;
   - при необходимости – показатели неудовлетворенного спроса (отказы).
7. Система визуализирует отчет (таблицы, графики, диаграммы) в соответствии с выбранными параметрами.
8. Пользователь анализирует отчет, при необходимости меняет параметры фильтрации/агрегации (повтор шагов 3–7).

### 8. Альтернативные и исключительные потоки

**A1. Экспорт отчета**

1. После шага 7 пользователь выбирает опцию «Экспорт» (например, в CSV/XLSX/PDF).
2. Система формирует файл с данными отчета и предоставляет его для скачивания.

**E1. Недостаточно данных для периода**

1. На шаге 5 система обнаруживает, что для выбранного периода данных недостаточно (например, период выходит за пределы хранения истории).
2. Система отображает уведомление и предлагает скорректировать период.

**E2. Ошибка построения отчета**

1. При извлечении или агрегации данных (шаг 5–6) возникает ошибка.
2. Система:
   - отображает сообщение о временной недоступности отчета;
   - логирует ошибку для диагностики;
   - предлагает повторить попытку позже.

### 9. Бизнес‑правила и ограничения

- **BR-16.1-001**: Показатели загрузки должны рассчитываться по единому алгоритму, согласованному с бизнесом (например, как отношение количества занятых мест к общему количеству мест для каждого временного интервала).
- **BR-16.1-002**: Период доступной для анализа истории должен быть не меньше, чем минимум, установленный внутренними и регуляторными требованиями (конкретное значение подлежит уточнению).
- **BR-16.1-003**: Доступ к аналитическим отчетам должен быть ограничен ролями (уполномоченные лица), и отчеты не должны раскрывать персональные данные клиентов сверх необходимого (анонимизация/агрегация, если нужно).

### 10. Функциональные требования (проверяемые)

- **FR-16.1-01**: Система должна предоставлять Управляющему/Владельцу доступ к отчету «Загрузка парковки» через раздел «Аналитика».
- **FR-16.1-02**: Система должна позволять задавать параметры периода, уровня детализации и разрезов анализа и строить отчет на их основе.
- **FR-16.1-03**: Система должна рассчитывать и отображать показатели загрузки в виде таблиц и/или графиков в соответствии с BR-16.1-001.
- **FR-16.1-04**: Система должна поддерживать экспорт данных отчета в распространенные форматы (например, CSV/XLSX) с сохранением ключевых показателей и разрезов.

### 11. Открытые вопросы / допущения

- Требуется уточнить состав метрик, которые критичны для Управляющего/Владельца (например, загрузка по времени суток, по дням недели, по сезонам, по типам клиентов).
- Требуется определить глубину истории аналитики и требования к производительности построения отчетов на больших объемах данных.
- Допущение: отчеты по загрузке не отображают персональные данные клиентов, а работают с агрегированными показателями, что упрощает требования к защите данных.
