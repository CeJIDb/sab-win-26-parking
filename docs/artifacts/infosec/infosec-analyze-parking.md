## Источники угроз

| **Источник угрозы** | **Угроза** |
|-|-|
| Сотрудник (Охранник) | Утечка персональных данных клиентов. Несанкционированное открытие шлагбаума в обход проверки оплаты или игнорирования чёрного списка. Несанкционированное внесение / удаление ТС из чёрного списка. Злоупотребление доступом к данным платежей. |
| Сотрудник (Управляющий / Владелец) | Несанкционированная выгрузка клиентской базы. Манипуляция тарифами и финансовыми данными. Изменение ролевых политик доступа в личных интересах. |
| Хакер | Скачивание / утечка БД с персональными данными. Подмена платёжных данных (MITM, инъекции). DDoS-атака. Подмена данных ГРЗ, полученных с LPR-камеры (UDP). Несанкционированный доступ к управлению шлагбаумом / СКУД. Перебор учётных данных (брутфорс). Кража API-ключей интеграций. |
| Клиент (ФЛ / ЮЛ) | Мошенничество с оплатой (chargeback / отмена транзакции после выезда). Попытка въезда по чужому ГРЗ. Регистрация с поддельными ПДн или фиктивным ИНН. Загрузка вредоносных файлов в качестве образов документов. |
| Внешние системы / интеграции | Компрометация API платёжного провайдера. Отказ или подмена данных СКУД / шлагбаума. Перехват UDP-пакетов LPR-камеры и подмена ГРЗ. Фишинг / спам через почтовый или SMS-шлюз. |
| Технические сбои | Недоступность сервиса (сбой сервера, сети, зависимостей). Потеря данных (отказ диска, ошибка БД). Сбой интеграции шлагбаума: невозможность въезда / выезда клиента. |

## Чувствительные данные и методы защиты

| **Данные** | **Метод защиты** |
|-|-|
| Паспортные данные / данные льготного документа / данные организации (ИНН) | Шифрование при хранении. Доступ только у минимального круга лиц по ролям. Маскирование в интерфейсе (частичное отображение). Передача только по HTTPS/TLS. |
| Данные ТС (ГРЗ) | Шифрование при хранении. Доступ по ролям. Журналирование запросов к данным ГРЗ. |
| ФИО, телефон, e-mail клиента и сотрудника | Шифрование при хранении. Доступ по ролям. Маскирование в экспортных выгрузках и логах. |
| Пароли учётных записей клиентов и сотрудников | Хеширование с солью (bcrypt / Argon2). Не хранятся в открытом виде. Не логируются. |
| Образы документов (сканы льготных документов, файлы договоров) | Шифрование файлов в хранилище. Доступ только у уполномоченных ролей (Управляющий, Владелец). Проверка типа и содержимого загружаемых файлов. |
| Данные платежей (транзакции, суммы, статусы) | Передача только по HTTPS/TLS. Доступ по ролям (Охранник — только создание; Управляющий — создание и возвраты). Журналирование всех операций. |
| API-ключи интеграций (платёжный провайдер, СКУД, шлюзы) | Хранение в защищённом хранилище секретов (Vault / переменные окружения). Ротация ключей. Не логируются и не передаются клиенту. Разные ключи для prod / stage. |
| Учётные данные сотрудников (логин, хэш пароля, роль) | Хеширование с солью. Управление учётными записями только у Управляющего / Владельца. Журналирование изменений ролей. |
| Парковочные сессии, бронирования, чёрный список | Доступ по ролям. Журналирование изменений. Soft-delete с сохранением истории. |

## Уязвимости

| **Уязвимость** | **Причина** |
|-|-|
| Недостаточный контроль доступа к функциям управления шлагбаумом | Отсутствие проверки прав на уровне каждого API-запроса к СКУД; нет двойного подтверждения (four-eyes) для ручного открытия |
| SQL / NoSQL-инъекции | Конкатенация пользовательского ввода в запросы к БД; отсутствие параметризованных запросов |
| Небезопасная прямая ссылка на объект (IDOR) | Предсказуемые идентификаторы бронирований / ТС / клиентов; отсутствие проверки владельца ресурса на уровне API |
| Подмена данных ГРЗ через незащищённый UDP-канал LPR | Отсутствие аутентификации и шифрования UDP-потока от LPR-камеры; нет верификации источника пакетов |
| Перебор учётных данных (брутфорс) | Отсутствие rate limiting и блокировки по IP / аккаунту на форме входа |
| XSS (межсайтовый скриптинг) | Отсутствие экранирования и Content Security Policy; вывод пользовательских данных без санитизации |
| CSRF (подделка межсайтовых запросов) | Отсутствие CSRF-токенов в мутирующих запросах |
| Небезопасная загрузка файлов | Принятие файлов без проверки типа, размера и содержимого (образы льготных документов и договоров) |
| Утечка API-ключей и секретов | Хранение ключей в открытом виде в конфигурации, логах или ответах API; наличие секретов в репозитории |
| Недостаточное логирование критических операций | Отсутствие записи в журнал для операций открытия шлагбаума, изменения чёрного списка, возвратов платежей |
| Слабая парольная политика и отсутствие MFA для привилегированных ролей | Нет требований к сложности пароля; нет второго фактора для Охранника, Управляющего, Владельца |
| Уязвимые или устаревшие компоненты зависимостей | Отсутствие процесса мониторинга CVE и регулярного обновления библиотек |
| Ошибки в обработке платёжного callback | Отсутствие верификации HMAC-подписи webhook от платёжного провайдера; возможность replay-атаки |

## Аутентификация и авторизация

| **Для кого** | **Какая** | **Политика авторизации** |
|-|-|-|
| КлиентФЛ | Логин (email / телефон) + пароль. SMS-код при смене номера телефона или подозрительной активности. | Доступ только к своим данным, ТС, бронированиям и платежам. Нет доступа к данным других клиентов, чёрному списку, управлению тарифами. |
| КлиентЮЛ | Логин (email) + пароль. SMS / email-код при добавлении нового пользователя организации. | Доступ к данным своей организации, ТС и бронированиям. Управление пользователями организации — только уполномоченным представителем ЮЛ. Нет доступа к данным других ЮЛ или ФЛ. |
| Охранник | Логин + пароль + MFA (TOTP или SMS). Блокировка после 5 неудачных попыток за 10 минут. | Создание платежей, ручное открытие шлагбаума (с обязательной записью в журнал и подтверждением), работа с чёрным списком. Нет доступа к возвратам, управлению сотрудниками, тарифами, ПДн за пределами своей задачи. |
| Управляющий | Логин + пароль + MFA (TOTP). Блокировка после 5 неудачных попыток за 10 минут. Принудительная смена пароля каждые 90 дней. | Полный доступ к управлению парковкой: тарифы, сотрудники (кроме Владельца), возвраты, отчёты. Нет доступа к системной конфигурации и изменению прав Владельца. |
| Владелец | Логин + пароль + MFA (TOTP). Аппаратный токен или дополнительное подтверждение для операций системной конфигурации. Принудительная смена пароля каждые 90 дней. | Полный доступ, включая системную конфигурацию и управление всеми ролями. Все привилегированные действия журналируются. |
| API-интеграции (платёж, СКУД, шлюзы) | API-ключ + HMAC-подпись запроса или OAuth 2.0 client credentials. IP Whitelist для СКУД и платёжного провайдера. | Минимальные права для каждой интеграции: СКУД — только команды управления шлагбаумом; платёжный провайдер — только операции оплаты / возврата; шлюзы — только отправка уведомлений. |

## Галстук-бабочка

**Ключевой риск: несанкционированный доступ к управлению шлагбаумом**

---

### Источники угрозы

- Хакер, получивший доступ к учётным данным Охранника или API-ключу СКУД
- Недобросовестный сотрудник (Охранник / Управляющий), действующий намеренно
- Злоумышленник, подменивший данные ГРЗ через незащищённый UDP-канал LPR-камеры

### Уязвимости — пути до события

- Слабая аутентификация / отсутствие MFA у Охранника → возможность компрометации учётной записи
- Незащищённый UDP-канал LPR-камеры → подмена ГРЗ и автоматическое срабатывание шлагбаума
- Отсутствие проверки прав на уровне API-запроса к СКУД (IDOR) → обход ролевых ограничений
- Хранение API-ключа СКУД в открытом виде → кража и прямое управление шлагбаумом

### Превентивные барьеры (до события)

1. **MFA для всех привилегированных ролей** — компрометация пароля недостаточна для входа.
2. **IP Whitelist для API СКУД** — команды шлагбаума принимаются только с доверенных IP-адресов.
3. **Шифрование и аутентификация UDP-канала LPR** — подмена ГРЗ из внешней сети исключена.
4. **Проверка прав на уровне каждого API-запроса** — IDOR к endpoint управления шлагбаумом невозможен.
5. **Блокировка после 5 неудачных попыток входа за 10 минут** — брутфорс учётных данных исключён.

### Реактивные барьеры (после события)

1. **Audit log с алертом на аномалию** — каждое открытие шлагбаума атрибутировано; подозрительная активность (открытие без ГРЗ, в нерабочее время) немедленно фиксируется.
2. **Немедленная блокировка скомпрометированной учётной записи / API-ключа** — прекращение несанкционированного доступа.
3. **Экстренный отзыв API-ключа СКУД → ручное управление** — закрытие канала несанкционированного управления.
4. **Расследование по журналу событий** — установление масштаба инцидента и оценка ущерба.
5. **Уведомление уполномоченных лиц** — немедленная эскалация по регламенту реагирования на инциденты.
