## Назначение документа

Документ фиксирует результаты анализа угроз информационной безопасности системы управления парковкой. Использован риск-ориентированный подход с применением методики Bow-Tie и базовых списков OWASP Top 10 2021.  
Документ служит исходником для формулировки нефункциональных требований безопасности.

**Контур анализа:** документ разрабатывается для **полной поставки**, а не только для MVP.

**Границы анализа:** публичный сайт (Business Site), клиентское веб-приложение (PWA / ЛК), админ-контур и внутренний контур сотрудников, серверная инфраструктура, а также интеграции: SSO для клиентов ФЛ (Яндекс ID / VK ID), платёжная система, платёжный терминал, ОФД, СКУД / шлагбаум, LPR-камера, почтовый и SMS-шлюз, ЭДО, биллинг / 1С, внешняя аналитика, геосервисы. Физическая инфраструктура площадки рассматривается в части оборудования СКУД, LPR, терминала и серверного / сетевого оборудования.

## Глоссарий аббревиатур

|Аббревиатура|Расшифровка|
|---|---|
|ГРЗ|Государственный регистрационный знак (номерной знак транспортного средства)|
|ТС|Транспортное средство|
|ПДн|Персональные данные|
|СКУД|Система контроля и управления доступом|
|LPR|License Plate Recognition — автоматическое распознавание номерных знаков|
|PWA|Progressive Web Application — прогрессивное веб-приложение|
|MFA / 2FA|Multi-Factor Authentication — многофакторная аутентификация|
|TOTP|Time-based One-Time Password — одноразовый пароль на основе времени|
|OIDC|OpenID Connect — протокол аутентификации поверх OAuth 2.0|
|PKCE|Proof Key for Code Exchange — защита OAuth 2.0 от перехвата авторизационного кода|
|IdP|Identity Provider — провайдер идентичности|
|IDOR|Insecure Direct Object Reference — небезопасная прямая ссылка на объект|
|SSRF|Server-Side Request Forgery — подделка запросов на стороне сервера|
|MITM|Man-in-the-Middle — атака «человек посередине»|
|HMAC|Hash-based Message Authentication Code — код аутентификации на основе хэша|
|CVE|Common Vulnerabilities and Exposures — реестр известных уязвимостей|
|DMZ|Demilitarized Zone — демилитаризованная зона (изолированный сетевой сегмент)|
|XSS|Cross-Site Scripting — межсайтовый скриптинг|
|CSRF|Cross-Site Request Forgery — подделка межсайтовых запросов|
|TLS|Transport Layer Security — протокол защиты транспортного уровня|
|HSTS|HTTP Strict Transport Security — принудительное использование HTTPS|
|RFC-1918|Стандарт приватных IP-диапазонов: 10.x.x.x, 172.16–31.x.x, 192.168.x.x|
|DR|Disaster Recovery — план восстановления после катастрофы|
|SAST|Static Application Security Testing — статический анализ безопасности кода|
|WAF|Web Application Firewall — файервол веб-приложений|
|DevOps|Development & Operations — инженер разработки и эксплуатации|
|API|Application Programming Interface — программный интерфейс приложения|
|UDP|User Datagram Protocol — протокол передачи данных без установления соединения|
|KMS|Key Management Service — сервис управления ключами шифрования|
|ОФД|Оператор фискальных данных|

## Источники угроз

| **Источник угрозы** | **Угроза** |
|---|---|
|Сотрудник (Охранник)|• Утечка персональных данных клиентов<br>• Несанкционированное открытие шлагбаума в обход проверки оплаты или игнорирования чёрного списка<br>• Несанкционированное внесение / удаление ТС из чёрного списка<br>• Злоупотребление доступом к данным платежей|
|Сотрудник (Управляющий / Владелец)|• Несанкционированная выгрузка клиентской базы<br>• Манипуляция тарифами и финансовыми данными<br>• Изменение ролевых политик доступа в личных интересах|
|Хакер|• Скачивание / утечка БД с персональными данными<br>• Подмена платёжных данных (MITM, инъекции)<br>• DDoS-атака<br>• Подмена данных ГРЗ, полученных с LPR-камеры (UDP)<br>• Несанкционированный доступ к управлению шлагбаумом / СКУД<br>• Перебор учётных данных (брутфорс)<br>• Кража API-ключей интеграций|
|Клиент (ФЛ / ЮЛ)|• Мошенничество с оплатой (chargeback / отмена транзакции после выезда)<br>• Попытка въезда по чужому ГРЗ<br>• Регистрация с поддельными ПДн или фиктивным ИНН<br>• Загрузка вредоносных файлов в качестве образов документов|
|Внешние системы / интеграции|• Компрометация API платёжного провайдера<br>• Подмена статуса оплаты от платёжного терминала или статуса фискализации от ОФД<br>• Компрометация SSO / IdP или ошибки OIDC-конфигурации<br>• Отказ или подмена данных СКУД / шлагбаума<br>• Перехват UDP-пакетов LPR-камеры и подмена ГРЗ<br>• Утечка данных через ЭДО, биллинг / 1С или внешнюю аналитику<br>• Фишинг / спам через почтовый или SMS-шлюз|
|Технические сбои|• Недоступность сервиса (сбой сервера, сети, зависимостей)<br>• Потеря данных (отказ диска, ошибка БД)<br>• Сбой интеграции шлагбаума: невозможность въезда / выезда клиента|
|Физический нарушитель|• Несанкционированный физический доступ к серверному оборудованию (изъятие диска, подключение носителя, перезагрузка)<br>• Физическое подключение к сетевому порту LPR-камеры или контроллера СКУД<br>• Замена LPR-камеры на подконтрольное устройство<br>• Отключение питания оборудования на парковке|

## Чувствительные данные и методы защиты

| **Данные** | **Метод защиты** |
|---|---|
|Паспортные данные / данные льготного документа / данные организации (ИНН)|• Шифрование при хранении<br>• Доступ только у минимального круга лиц по ролям<br>• Маскирование в интерфейсе (частичное отображение)<br>• Передача только по HTTPS/TLS|
|Реквизиты организации и банковские реквизиты (ИНН, ОГРН/ОГРНИП, КПП, р/с, БИК, банк)|• Хранение в РФ<br>• Доступ по ролям, связанным с договорами и оплатой<br>• Маскирование при массовом просмотре и выгрузке<br>• Журналирование обращений и экспортов|
|Данные ТС (ГРЗ)|• Шифрование при хранении<br>• Доступ по ролям<br>• Журналирование запросов к данным ГРЗ|
|ФИО, телефон, e-mail клиента и сотрудника|• Шифрование при хранении<br>• Доступ по ролям<br>• Маскирование в экспортных выгрузках и логах|
|Пароли учётных записей клиентов и сотрудников|• Хеширование с солью (bcrypt / Argon2)<br>• Не хранятся в открытом виде<br>• Не логируются|
|Секрет TOTP сотрудника и резервные коды восстановления|• Хранение в зашифрованном виде<br>• Доступ только в контуре проверки аутентификации<br>• Запрет отображения и выгрузки<br>• Отдельный аудит операций перепривязки устройства|
|Данные SSO-провайдера для клиента ФЛ (`sub`, код провайдера, минимальные claims)|• Обработка по принципу минимальной достаточности<br>• Пароль провайдера не хранится<br>• Связь с учётной записью клиента — только для идентификации<br>• Ограниченный срок жизни токенов и защищённое хранение серверной сессии|
|Записи согласий и акцепта оферты / правил / обработки ПДн|• Неизменяемый журнал с фиксацией даты предоставления и отзыва<br>• Хранение в РФ<br>• Доступ только для ролей аудита и обслуживания прав субъектов ПДн|
|Образы документов (сканы льготных документов, файлы договоров)|• Шифрование файлов в хранилище<br>• Доступ только у уполномоченных ролей (Управляющий, Владелец)<br>• Проверка типа и содержимого загружаемых файлов|
|Данные платежей (транзакции, суммы, статусы)|• Передача только по HTTPS/TLS<br>• Доступ по ролям (Охранник — только создание; Управляющий — создание и возвраты)<br>• Журналирование всех операций|
|Чеки, квитанции терминала, фискальные статусы ОФД|• Доступ по ролям<br>• Идемпотентная обработка по `id платежа`. Проверка целостности / подписи входящих уведомлений<br>• Хранение в РФ и журналирование всех изменений статуса|
|Идентификатор плательщика во внешнем платёжном сервисе (`externalPayerId`)|• Хранение в РФ<br>• Не отображается в интерфейсах<br>• По возможности шифруется<br>• Доступ только у ограниченного технического контура|
|API-ключи интеграций (платёжный провайдер, СКУД, шлюзы)|• Хранение в защищённом хранилище секретов (Vault / переменные окружения)<br>• Ротация ключей<br>• Не логируются и не передаются клиенту<br>• Разные ключи для prod / stage|
|Учётные данные сотрудников (логин, хэш пароля, роль)|• Хеширование с солью<br>• Управление учётными записями только у Управляющего / Владельца<br>• Журналирование изменений ролей|
|Парковочные сессии, бронирования, чёрный список|• Доступ по ролям<br>• Журналирование изменений<br>• Soft-delete с сохранением истории|
|Журналы аудита, причины статусов, комментарии ручного допуска, пакеты выгрузок в ЭДО / 1С / аналитику|• Защита логов и экспортных файлов<br>• RBAC и отдельное журналирование выгрузок<br>• Маскирование чувствительных полей в отчётах<br>• Ограниченные сроки хранения и контроль получателей выгрузки|
|Все данные в транзите (все интеграции)|• HTTPS / TLS минимум версии 1.2 (рекомендуется 1.3)<br>• Cipher suites: только AEAD (AES-256-GCM, CHACHA20-POLY1305)<br>• Запрет TLS 1.0/1.1<br>• HSTS для PWA<br>• Мониторинг срока действия сертификатов с алертом за 30 дней|

## Уязвимости

| **Уязвимость** | **Причина** |
|---|---|
|Недостаточный контроль доступа к функциям управления шлагбаумом|Отсутствие проверки прав на уровне каждого API-запроса к СКУД; нет двойного подтверждения (four-eyes) для ручного открытия|
|SQL / NoSQL-инъекции|Конкатенация пользовательского ввода в запросы к БД; отсутствие параметризованных запросов|
|Небезопасная прямая ссылка на объект (IDOR)|Предсказуемые идентификаторы бронирований / ТС / клиентов; отсутствие проверки владельца ресурса на уровне API|
|Подмена данных ГРЗ через незащищённый UDP-канал LPR|Отсутствие аутентификации и шифрования UDP-потока от LPR-камеры; нет верификации источника пакетов|
|Перебор учётных данных (брутфорс)|Несоблюдение или обход мер из раздела «Аутентификация и авторизация»: лимиты попыток входа, временная блокировка / CAPTCHA, раздельные пороги для клиентов и сотрудников — при ошибках реализации, ослаблении настроек или обходе (например, распределённый перебор)|
|XSS (межсайтовый скриптинг)|Отсутствие экранирования и Content Security Policy; вывод пользовательских данных без санитизации|
|CSRF (подделка межсайтовых запросов)|Отсутствие CSRF-токенов в мутирующих запросах|
|Неправильная реализация OAuth 2.0 / OIDC для SSO клиента ФЛ|Отсутствие проверки `state` / `nonce`, `redirect_uri` allowlist и PKCE; избыточные scope; небезопасное хранение токенов или серверной сессии|
|Небезопасная загрузка файлов|Принятие файлов без проверки типа, размера и содержимого (образы льготных документов и договоров)|
|Утечка API-ключей и секретов|Хранение ключей в открытом виде в конфигурации, логах или ответах API; наличие секретов в репозитории|
|Недостаточное логирование критических операций|Отсутствие записи в журнал для операций открытия шлагбаума, изменения чёрного списка, возвратов платежей|
|Остаточные риски парольной политики и жизненного цикла сессий|• Для Клиента ФЛ / ЮЛ в разделе «Аутентификация и авторизация» заданы минимальная длина пароля, проверка на утечку (Have I Been Pwned для ФЛ), лимиты попыток входа / CAPTCHA и параметры сессии и cookie (idle / absolute timeout, Secure, HttpOnly, SameSite) — **уязвимость проявляется при несоответствии реализации** или ослаблении политики в эксплуатации<br>• Для сотрудников обязательна MFA (TOTP), но сохраняется остаточный риск: обход 2FA через процедуру сброса пароля, общие учётные записи, неинвалидированные сессии при увольнении, ошибки привязки SSO|
|Уязвимые или устаревшие компоненты зависимостей|Отсутствие процесса мониторинга CVE и регулярного обновления библиотек|
|Ошибки в обработке платёжного callback|Отсутствие верификации HMAC-подписи webhook от платёжного провайдера; возможность replay-атаки|
|Недостаточная защита интеграции с платёжным терминалом и ОФД|Доверие неподписанным статусам терминала / квитанциям / уведомлениям ОФД; отсутствие идемпотентности и сверки `id платежа`; ошибки фискализации не мониторятся и не эскалируются|
|Подделка запросов на стороне сервера (SSRF)|Формирование URL для исходящих интеграционных запросов (платёжный провайдер, СКУД, шлюзы) с использованием непроверенных входных данных; отсутствие allowlist разрешённых хостов и блокировки внутренних диапазонов IP (RFC-1918, link-local 169.254.x.x)|
|Небезопасная конфигурация сервера и смежных систем|Детальные сообщения об ошибках в production-ответах API (стек трейс, версия СУБД, SQL-фрагмент); дефолтные учётные данные LPR-камеры и СКУД-оборудования; отсутствие HTTP security headers (HSTS, X-Content-Type-Options, X-Frame-Options, Referrer-Policy)|
|Бизнес-логические уязвимости (race conditions)|Отсутствие идемпотентности при обработке платёжных webhook и запросов бронирования; конкурирующие запросы могут создать дублирующую сессию или дважды засчитать платёж|
|Несоблюдение требований TLS в транзите|В разделе «Чувствительные данные и методы защиты» заданы минимум TLS 1.2 (рекомендуется 1.3), только AEAD-шифронаборы, запрет TLS 1.0/1.1, HSTS для PWA и мониторинг сертификатов — **уязвимость проявляется при неверной конфигурации** TLS-терминации (прокси, CDN, балансировщик), слабом наборе шифров по умолчанию, отключённой проверке сертификатов или ошибках миграции; TLS 1.0/1.1 и downgrade остаются актуальны при срыве этих требований|
|Избыточные выгрузки и небезопасные отчёты во внешние системы|Отсутствие RBAC на экспорт в ЭДО / 1С / аналитику; нет маскирования и watermarking; файлы выгрузок доступны дольше необходимого|
|Нарушение жизненного цикла согласий и сроков хранения ПДн|Нет retention policy; отсутствует фиксация отзыва согласия и процедур блокирования / уничтожения; гостевой контур через терминал собирает данные без полной трассировки оснований обработки|
|Отсутствие контроля физического доступа к серверу и оборудованию|Нет требований к серверному помещению, политике доступа к стойке; контроллер СКУД и LPR-камера физически доступны на парковке без описания мер защиты|
|Отсутствие сетевой изоляции веб-слоя от внутренней сети и БД|Веб-сервер и сервер БД размещены в одном сегменте без DMZ; компрометация веб-приложения даёт прямой доступ к БД|
|Отсутствие мониторинга ресурсов сервера и внешних зависимостей|Нет инструмента сбора метрик CPU/RAM/диска; нет healthcheck-проверки доступности платёжного API, СКУД API, LPR-канала, шлюзов; сбой зависимости не обнаруживается до жалоб пользователей|

## Аутентификация и авторизация

| **Для кого** | **Какая** | **Политика авторизации** |
|---|---|---|

|Неавторизованный пользователь (Business Site / гость)|• Без входа в систему<br>• Для гостевого проезда через терминал: минимальная идентификация (имя, телефон) и явное согласие с тарифом, правилами и обработкой ПДн до создания оплаты / сессии|• Только просмотр публичной информации<br>• Нет доступа к ЛК, админке, ПДн, данным договоров и отчётности|

|Владелец|• Логин + пароль + обязательная MFA по TOTP<br>• Аппаратный токен или дополнительное подтверждение для операций системной конфигурации и критичных финансовых действий<br>• Принудительная смена пароля каждые 90 дней|• Полный доступ, включая системную конфигурацию, роли, критичные отчёты и ручное управление доступом<br>• Все привилегированные действия журналируются|
|SSO / IdP (только для Клиента ФЛ)|• OAuth 2.0 / OIDC<br>• Серверная проверка `state`, `nonce`, `redirect_uri`, ограничение scope, PKCE для публичных клиентов<br>• Логирование событий входа и ошибок федерации|• Только аутентификация клиента ФЛ<br>• После завершения входа права определяются локальной учётной записью платформы; провайдер SSO не получает права внутри системы парковки|

## Анализ Bow-Tie (галстук-бабочка)

**Ключевой риск: несанкционированный доступ к управлению шлагбаумом**

Метод Bow-Tie («галстук-бабочка») показывает пути реализации риска (слева) и его последствия (справа) с барьерами на обе стороны от центрального события.

---

### Источники угрозы

- Хакер, получивший доступ к учётным данным Охранника или API-ключу СКУД
- Недобросовестный сотрудник (Охранник / Управляющий), действующий намеренно
- Злоумышленник, подменивший данные ГРЗ через незащищённый UDP-канал LPR-камеры

### Уязвимости — пути до события

- Компрометация учётной записи Охранника (фишинг, обход MFA через сброс пароля, общая учётная запись) → вход от имени сотрудника
- Незащищённый UDP-канал LPR-камеры → подмена ГРЗ и автоматическое срабатывание шлагбаума
- Отсутствие проверки прав на уровне API-запроса к СКУД (IDOR) → обход ролевых ограничений
- Хранение API-ключа СКУД в открытом виде / утечка через репозиторий → кража и прямое управление шлагбаумом

### Превентивные барьеры (до события)

1. **MFA для всех привилегированных ролей** — компрометация пароля недостаточна для входа.
2. **IP Whitelist для API СКУД** — команды шлагбаума принимаются только с доверенных IP-адресов.
3. **Шифрование и аутентификация UDP-канала LPR** — подмена ГРЗ из внешней сети исключена.
4. **Проверка прав на уровне каждого API-запроса** — IDOR к endpoint управления шлагбаумом невозможен.
5. **Блокировка после 5 неудачных попыток входа за 10 минут** — брутфорс учётных данных исключён.

### Реактивные барьеры (после события)

1. **Audit log с алертом на аномалию** — каждое открытие шлагбаума атрибутировано; подозрительная активность (открытие без ГРЗ, в нерабочее время) немедленно фиксируется.
2. **Немедленная блокировка скомпрометированной учётной записи / API-ключа** — прекращение несанкционированного доступа.
3. **Экстренный отзыв API-ключа СКУД → ручное управление** — закрытие канала несанкционированного управления.
4. **Расследование по журналу событий** — установление масштаба инцидента и оценка ущерба.
5. **Уведомление уполномоченных лиц** — немедленная эскалация по регламенту реагирования на инциденты.

### Соответствие карточке проекта

Каноничные ограничения заказчика — в [`project-charter.md`](../project-charter.md) (раздел «Ограничения на решение»). Ниже — как они **пересекаются** с настоящим анализом ИБ; окончательная правовая квалификация и выбор средств — за оператором / юристом / УИБ.

| Тема | Суть для системы парковки | Статус в документе |
|---|---|---|
| **152-ФЗ (ПДн)** | ЛК, уведомления, идентификация на КПП, интеграции, согласия, локализация | Раскрыто в таблицах данных, рисках RISK-02 / RISK-16 / RISK-17 и в [открытых вопросах по 152-ФЗ](#соответствие-152-фз-и-пп--1119--открытые-вопросы-для-оператора) |
| **63-ФЗ (ЭП и электронные документы)** | Онлайн-договоры ФЛ; при развитии ЭДО — виды ЭП и юридическая значимость действий в ЛК | Требует явной проработки в ФТ/процессах: не дублируется полным перечнем мер в этом артефакте; **оператор** должен согласовать модель подписания и хранения с юристом |
| **54-ФЗ (ККТ)** | Онлайн-оплата, терминал на КПП, ОФД; минимизация лишних фискальных данных в проектной документации | Отражено в контуре платежей / ОФД, RISK-15, мерах подписи и идемпотентности; **бухгалтерский регламент** — вне объёма настоящего анализа (см. чартер) |
| **149-ФЗ** | Общие требования к защите информации в ИС; связь мер по ПДн и эксплуатации | Покрывается совокупностью организационных и технических контрмер в реестре рисков; детальный ИБ-регламент оператора — отдельный документ |
| **Реестр отечественного ПО** | Выбор и обоснование компонентов (в т.ч. СКЗИ при необходимости) | В анализе не фиксируется номенклатура продуктов; при проектировании стека **система/заказчик** должны проверять включение в реестр и допустимость импортозамещения |
| **Обязательная минимальная идентификация клиентов (имя + телефон)** | Поднимает вопросы ПДн, оснований обработки, согласий, хранения и передачи провайдерам связи / платежам | Отражено для гостевого терминала в разделе «Аутентификация и авторизация» (неавторизованный пользователь) и в RISK-16; **трассировка оснований** — в открытых вопросах по 152-ФЗ |

**Неясно:** при появлении в продукте полноценного ЭДО или иных сценариев с КЭП/НЭП — дополнить этот артефакт отдельным подразделом угроз и контрмер под согласованную с юристом модель.

### Соответствие 152-ФЗ и ПП № 1119 — открытые вопросы для оператора

|Вопрос|Норма|Статус|
|---|---|---|
|Уведомление Роскомнадзора о начале обработки ПДн|ст. 22 152-ФЗ|Требует решения оператора и юриста: проверить применимость исключений и при необходимости подать уведомление до начала обработки|
|Определение категории ПДн и уровня защищённости (УЗ)|ПП РФ № 1119 от 01.11.2012|• Предварительная оценка: иные ПДн → УЗ-4; если образы документов используются для идентификации (фото) — биометрические ПДн → УЗ-3<br>• Окончательно определяет оператор|
|Локализация ПДн на территории РФ|ст. 18 152-ФЗ|Нужно явно зафиксировать размещение primary БД на территории РФ и проверить трансграничные потоки через внешнюю аналитику, почтовый и SMS-шлюз|
|Уведомление об инцидентах и утечках ПДн|ст. 21 152-ФЗ|Требуется отдельный регламент: сроки и состав уведомлений Роскомнадзору сверять по действующей редакции закона и подзаконных актов|
|Сроки хранения и уничтожения ПДн|ст. 21 152-ФЗ|Не определены для паспортных данных и образов документов — требуется retention policy|
|Назначение ответственного за организацию обработки ПДн|ст. 18.1 152-ФЗ|Уполномоченный по ПДн упомянут как роль, но формальное назначение приказом не описано|
|Права субъектов ПДн|ст. 14 (право доступа), ст. 9 (отзыв согласия), ст. 21 (исправление / блокировка / уничтожение) 152-ФЗ|• Техническая реализация не описана — требуется: API доступа к данным, механизм удаления аккаунта, форма отзыва согласия<br>• Оформить отдельными ФТ|
|Санкции за нарушения|ст. 13.11 КоАП РФ, актуальная редакция|Размеры штрафов, включая повышенные и оборотные составы, нужно верифицировать у юриста по ПДн перед выпуском финальной версии документа|
|Специальные категории ПДн (льготы по состоянию здоровья)|ст. 10 152-ФЗ|Если данные льготных документов раскрывают состояние здоровья субъекта — это специальная категория ПДн, требующая отдельного явного согласия и усиленных мер защиты; уточнить у юриста на этапе категоризации|

### Границы обработки видеоданных

Настоящий анализ охватывает данные LPR (ГРЗ-событие + метаданные), но **не** архивные видеозаписи КПП. Если в системе предусмотрено хранение видеоархива:

- требуется отдельный анализ угроз для CCTV-подсистемы;
- применяется режим обработки биометрических ПДн (изображения лиц) по **ст. 11 152-ФЗ**;
- необходимо определить сроки хранения записей и права доступа.

**Если видеоархив не предусмотрен — зафиксировать явно в архитектурных требованиях.**
