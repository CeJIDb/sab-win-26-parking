## Введение

Данный документ описывает **концептуальную модель системы управления парковкой**.

Модель охватывает ключевые бизнес‑сущности, связанные с:

- организацией парковочного пространства: `Парковка`, `Сектор`, `ПМ`, `КПП`;
- оказанием услуг парковки: `Бронирование`, `Парковочная сессия`, `Платеж`, `Чек`, `Тариф`, `Договор`;
- взаимодействием с клиентами и их транспортными средствами: `Клиент`, `Профиль`, `ТС`, `Обращение`, `Уведомление`;
- поддержкой операционной деятельности и внешних взаимодействий: роли `Администратор`, `Оператор`.

По уровню детализации услуг выделяются три уровня: **Договор** задаёт долгосрочные рамки и квоты; **Бронирование** — план использования места; **Парковочная сессия** — факт нахождения ТС на месте и основание для начисления платы.

Основные участники предметной области:

- **Клиент** — физическое или юридическое лицо, пользующееся услугами парковки.
- **Администратор** (управляющий, владелец) — настраивает парковки, тарифы, справочники.
- **Оператор КПП** (охранник, старший охранник) — обрабатывает обращения и выполняет оперативные действия.

## Сущности

### `Парковка`

**Назначение.** Парковочный объект (площадка/комплекс), в рамках которого организованы парковочные места и действует определенная тарифная политика.

**Связи.**

- `Парковка` —(1:N)→ `Сектор`: одна парковка содержит несколько секторов.
- `Парковка` —(1:N)→ `КПП`: у парковки может быть несколько КПП.
- `Парковка` —(1:N)→ `Тариф` (опционально): на парковке действует набор тарифов.
- `Парковка` ←(N:1)— `Договор` (опционально): договор может относиться к одной или нескольким парковкам.

### `Сектор`

**Назначение.** Логически/физически выделенная часть парковки (зона, этаж и т.п.), в которой размещены парковочные места.

**Связи.**

- `Сектор` ←(N:1)— `Парковка`: каждый сектор принадлежит одной парковке.
- `Сектор` —(1:N)→ `ПМ`: сектор включает множество парковочных мест.
- `Сектор` —(1:N)→ `Тариф` (опционально): могут быть отдельные тарифы по зонам.

### `ПМ` (Парковочное место)

**Назначение.** Конкретное физическое место для размещения транспортного средства.

**Связи.**

- `ПМ` ←(N:1)— `Сектор`: место принадлежит сектору.
- `ПМ` —(1:N)→ `Бронирование`: на одно место может быть несколько бронирований в разные периоды.
- `ПМ` —(1:N)→ `Парковочная сессия`: на месте фиксируются различные сессии парковки.
- `ПМ` ←(N:1)— `Тариф` (опционально): место может наследовать/иметь свой тариф.

### `Бронирование`

**Назначение.** Запись о заранее зарезервированном использовании парковочного места на определенный период.

**Связи.**

- `Бронирование` ←(N:1)— `Профиль`: клиент может создавать множество бронирований.
- `Бронирование` ←(N:1)— `ПМ`: бронирование относится к ПМ.
- `Бронирование` —(1:0..N)→ `Парковочная сессия`: из бронирования может возникнуть одна или несколько фактических сессий либо ни одной.
- `Бронирование` ←(0..N:1)— `Тариф`: стоимость бронирования рассчитывается по тарифу.
- `Бронирование` -(1:N)→ `Платеж`: бронирование оплачивается одним или несколькими платежами (в случае договора). Для разовых бронирований оплата чаще привязана к парковочной сессии; для долгосрочных — к самому объекту бронирования (в т.ч. в рамках договора).
- `Бронирование` —(1:N)→ `Уведомление`: уведомления о создании, изменении, отмене.
- `Бронирование` —(1:N)→ `Обращение` (опционально): обращения по конкретному бронированию.

### `Парковочная сессия`

**Назначение.** Фактический период нахождения транспортного средства на парковочном месте (от въезда до выезда), по которому начисляется плата.

**Связи.**

- `Парковочная сессия` ←(0..N:1)— `ПМ`: каждая сессия привязана к одному месту.
- `Парковочная сессия` ←(0..N:1)— `ТС`: сессия относится к конкретному транспортному средству (по ГРЗ); через `ТС` и `Профиль` логически связана с клиентом и, при необходимости, с договором.
- `Парковочная сессия` ←(0..N:1)— `Бронирование` (по смыслу): сессия может быть привязана к бронированию.
- `Парковочная сессия` —(1:1)→ `Платеж`: каждая сессия связана с конкретным платежом.
- `Парковочная сессия` ←(0..N:1)— `Тариф`: применяется тариф для расчета стоимости.
- `Парковочная сессия` —(1:N)→ `Уведомление`: уведомления о начале, окончании, превышении времени и т.п.
- `Парковочная сессия` —(1:N)→ `Обращение` (опционально): споры по расчету, штрафам и др.
- `Парковочная сессия` ←(0..1:1)— `КПП` (КПП въезда): сессия хранит КПП, через который был выполнен въезд.
- `Парковочная сессия` ←(0..1:1)— `КПП` (КПП выезда): сессия хранит КПП, через который был выполнен выезд (может отличаться от КПП въезда).

### `Платеж`

**Назначение.** Факт оплаты услуг (парковки, бронирования и др.) с суммой, способом оплаты и статусом.

**Связи.**

- `Платеж` ←(0..N:1)— `Клиент`: клиент совершает платеж.
- `Платеж` ←(0..1:1)— `Парковочная сессия` (чаще всего): платеж относится к сессии.
- `Платеж` ←(N:1)— `Бронирование` (опционально): оплата бронирования.
- `Платеж` —(1:0..N)→ `Чек`: успешный платеж может порождать один или несколько фискальных чеков; при неуспешном платеже чек не формируется.
- `Платеж` —(1:N)→ `Уведомление`: сообщения о списании, возврате, неуспехе.
- `Платеж` —(1:N)→ `Обращение` (опционально): споры по платежу.

### `Чек`

**Назначение.** Фискальный/расчетный документ, подтверждающий платеж.

**Связи.**

- `Чек` ←(0..N:1)— `Платеж`: чек связан с платежом.
- `Чек` —(1:N)→ `Обращение` (опционально): вопросы по фискальным документам.
- `Чек` —(1:N)→ `Уведомление` (опционально): отправка копий чека.

### `Договор`

**Назначение.** Юридическое соглашение между клиентом и оператором парковки (условия, лимиты, период, перечень парковок).

**Связи.**

- `Договор` ←(0..N:1)— `Профиль`: договор принадлежит одному профилю клиента; у профиля может быть ноль или несколько договоров.
- `Договор` —(1:N)→ `Бронирование` (для долгосрочных бронирований): каждое долгосрочное бронирование (>24 часов) должно быть привязано к действующему договору.
- `Договор` —(1:N)→ `Уведомление` (опционально): отправка сообщения о статусе договора.
- `Договор` —(1:N)→ `Обращение` (опционально): вопросы по договору.

Связь договора с клиентом задаётся через профиль: один профиль может иметь 0..N договоров; долгосрочные бронирования заключаются из профиля и привязываются к одному из его договоров.

### `Профиль`

**Назначение.** Набор данных и настроек, по которым система взаимодействует с клиентом (каналы связи, настройки уведомлений, привязанные способы оплаты и т.п.).

**Связи.**

- `Профиль` ←(0..1:N)— `Клиент`: один клиент может иметь один или несколько профилей, или ни одного.
- `Профиль` —(1:0..N)→ `Договор`: у профиля может быть ноль или несколько договоров (долгосрочные бронирования оформляются в рамках договора профиля).
- `Профиль` —(1:N)→ `Бронирование`: бронирования создаются и управляются из профиля клиента.
- `Профиль` —(1:N)→ `ТС`: в профиле хранится список транспортных средств клиента.
- `Профиль` —(1:N)→ `Уведомление`: уведомления отправляются по данным профиля.
- `Профиль` —(1:N)→ `Обращение`: обращения создаются из конкретного профиля/канала.

### `Клиент`

**Назначение.** Субъект, получающий услуги парковки.

**Связи.**

- `Клиент` —(1:0..N)→ `Профиль`: через профили клиент выполняет операции в системе.

Бронирования, парковочные сессии, платежи, обращения и уведомления связаны с клиентом опосредованно — через его профили, транспортные средства и договоры.

### `ТС`

**Назначение.** Транспортное средство клиента (автомобиль, мотоцикл и т.п.), для которого оформляются доступ на парковку, договоры и сессии.

**Связи.**

- `ТС` ←(0..N:1)— `Профиль`: транспортные средства управляются из профиля клиента.
- `ТС` —(1:0..N)→ `Парковочная сессия`: сессии фиксируются по конкретному ТС (по ГРЗ).

### `КПП`

**Назначение.** Контрольно‑пропускной пункт (шлагбаум/ворота), через который выполняется въезд или выезд на парковку. У одной парковки может быть несколько КПП; клиент может въехать через один КПП и выехать через другой.

**Связи.**

- `КПП` ←(N:1)— `Парковка`: каждый КПП принадлежит одной парковке.
- `КПП` —(1:N)→ `Парковочная сессия`: один КПП выступает точкой въезда и/или выезда для множества сессий (у каждой сессии хранятся отдельно КПП въезда и КПП выезда).

### `Тариф`

**Назначение.** Правила расчета стоимости (ставки, интервалы, льготы).

**Связи.**

- `Тариф` —(1:0..N)→ `Парковочная сессия`.
- `Тариф` —(1:0..N)→ `Бронирование`.

### `Обращение`

**Назначение.** Запрос/жалоба/вопрос клиента к оператору.

**Связи.**

- `Обращение` ←(N:1)— `Клиент`.
- `Обращение` ←(N:1)— `Профиль` (опционально).
- `Обращение` ←(N:1)— `Оператор`.
- `Обращение` ←(N:1)— `Бронирование` / `Парковочная сессия` / `Платеж` / `Чек` (опционально).

### `Уведомление`

**Назначение.** Сообщение, отправляемое клиенту о событиях в системе.

**Связи.**

- `Уведомление` ←(N:1)— `Клиент`.
- `Уведомление` ←(N:1)— `Профиль`.
- `Уведомление` ←(N:1)— `Бронирование` / `Парковочная сессия` / `Платеж` / `Чек` / `Обращение` (опционально).

## Связи (обобщенный список)

- **`Парковка` —(1:N)→ `Сектор`** — структурирование территории на зоны/этажи.
- **`Сектор` —(1:N)→ `ПМ`** — размещение конкретных парковочных мест в секторе.
- **`Парковка` —(1:N)→ `КПП`** — у парковки несколько точек въезда/выезда.
- **`КПП` —(1:N)→ `Парковочная сессия`** — КПП выступает точкой въезда и/или выезда для множества сессий.
- **`ПМ` —(1:N)→ `Бронирование`** — одно место может многократно бронироваться в разные периоды.
- **`ПМ` —(1:N)→ `Парковочная сессия`** — через одно место проходит множество фактических сессий.
- **`Клиент` —(1:0..N)→ `Профиль`** — клиент может иметь ноль или несколько профилей (операции в системе — через профили; минимальная идентификация без ЛК возможна без профиля).
- **`Профиль` —(1:N)→ `Бронирование` / `ТС`** — из профиля создаются бронирования и управляются транспортные средства.
- **`Профиль` —(1:0..N)→ `Договор`** — у профиля может быть ноль или несколько договоров; долгосрочные бронирования оформляются в рамках договора профиля.
- **`Бронирование` —(1:0..N)→ `Парковочная сессия`** — из одного бронирования может не возникнуть ни одной или возникнуть несколько сессий.
- **`ТС` —(1:0..N)→ `Парковочная сессия`** — по одному ТС может быть множество сессий.
- **`Тариф` —(1:0..N)→ `Бронирование` / `Парковочная сессия`** — тариф применяется при расчёте стоимости бронирований и сессий.
- **`Договор` —(1:N)→ `Бронирование`** — долгосрочные бронирования оформляются в рамках договора.
- **`Бронирование` —(1:N)→ `Платеж`** — бронирование может оплачиваться одним или несколькими платежами (например, по договору).
- **`Парковочная сессия` —(1:1)→ `Платеж`** — каждая сессия имеет свой итоговый платёж (при пост‑оплате; предоплата может быть у бронирования).
- **`Платеж` —(0..N)→ `Чек`** — успешный платеж порождает один или несколько фискальных чеков; при неуспешном платеже чек не формируется.
- **`Обращение` ←(N:1)— `Клиент` / `Профиль` / `Оператор`** — обращение всегда принадлежит клиенту, инициируется из профиля и обрабатывается оператором.
- **`Обращение` ←(N:1)— `Бронирование` / `Парковочная сессия` / `Платеж` / `Чек` (опционально)** — обращение может быть привязано к конкретному объекту.
- **`Уведомление` ←(N:1)— `Клиент` / `Профиль`** — уведомления адресуются клиенту и отправляются по данным его профиля.
- **`Уведомление` ←(N:1)— `Бронирование` / `Парковочная сессия` / `Платеж` / `Чек` / `Обращение` (опционально)** — уведомление связано с конкретным событием или документом.

## Акторы и внешние системы

### `Администратор`

**Роль.** Настройка структуры парковок (`Парковка`, `Сектор`, `ПМ`, `КПП`), тарифов (`Тариф`), базовых правил и справочников, административные операции по договорам и сессиям.

### `Оператор`

**Роль.** Обработка `Обращение`, анализ и корректировка данных по `Бронирование`, `Парковочная сессия`, `Платеж`, `Чек`, инициирование `Уведомление` в ответ на обращения.

### `Клиент` (как актор)

**Роль.** Создает `Бронирование`, инициирует `Парковочная сессия` (въезд/выезд), совершает `Платеж`, получает `Уведомление`, создает `Обращение`.
