# Аутентификация и авторизация

Артефакт фиксирует, **для кого** выполняется вход в систему, **какой способ аутентификации** используется и **какая политика авторизации** (разграничение доступа) применяется. Согласовано с реестром Use Case, концептуальной моделью (Клиент, Сотрудник, роли) и мерами из «Данные и методы защиты».

| Для кого | Какая аутентификация | Политика авторизации |
|----------|----------------------|----------------------|
| **Клиент (ФЛ, ЮЛ)** | **ФЛ:** логин + пароль в ЛК **или вход через SSO** (Яндекс ID, VK ID — OAuth 2.0 / OpenID Connect). **ЮЛ:** только логин + пароль (SSO не применяется). Восстановление пароля по e-mail/телефону (UC-2.2, UC-2.3) — по ссылке или коду с ограниченным сроком действия. При пароле: только необратимый хэш (bcrypt/Argon2), защита от перебора; блокировка учётной записи после N неудачных попыток входа (порог и срок блокировки — по политике оператора). При SSO: связь с идентификатором провайдера (sub), минимальный объём данных от провайдера, пароль провайдера не хранится | Доступ только к собственным данным: профиль, ТС, бронирования, договоры, обращения, уведомления, оплаты. Запрет доступа к данным других клиентов и к админке. Политика единая для пароля и SSO. Логирование входов (успешных и неуспешных) для аудита и расследования инцидентов |
| **Сотрудник (Охранник, Управляющий, Владелец)** | Логин + пароль в админ-интерфейсе и на посту КПП (UC-15.5) **и обязательная 2FA по TOTP** (приложение-аутентификатор, например Google Authenticator или аналог). Хранение хэша пароля (bcrypt/Argon2), защита от перебора; секрет TOTP хранится в зашифрованном виде, привязка к учётной записи сотрудника; отдельная таблица учётных записей сотрудников; при первом входе с временным паролем — обязательная смена пароля перед доступом | RBAC по ролям. **Охранник:** КПП, ручной допуск (UC-12.13, UC-12.14), просмотр списков ТС/ПС, ограниченный просмотр профилей клиентов. **Управляющий:** + управление бронированиями, обращениями, уведомлениями, тарифами, секторами и ПМ, просмотр отчётов. **Владелец:** полный доступ, включая шаблоны договоров, финансовые отчёты, профили сотрудников (UC-15.x). Аудит ручных допусков, доступа к спискам и входов (успешных и неуспешных); срок хранения логов — по политике оператора |
| **Неавторизованный пользователь (гость)** | Без входа в систему | Только просмотр публичной информации: тарифы, контакты, доступность мест, правила и условия (UC-1.1–UC-1.4). Запрет доступа к ЛК, админке и персональным данным |
| **Система (сервисные/фоновые процессы)** | Техническая аутентификация: сервисный аккаунт, API-ключ или сертификат для автоматических сценариев (создание бронирований при въезде UC-12.2, отправка уведомлений UC-5.8–5.10, интеграция с платёжным провайдером). Ротация ключей и срок жизни — по политике оператора; разделение по окружениям (dev/stage/prod) | Минимальные права только на необходимые операции; без доступа к ПДн сверх необходимого для сценария; логирование вызовов; разделение секретов по окружениям; запрет выгрузки и отображения учётных данных в UI |

## SSO для клиентов — только ФЛ (Яндекс ID, VK ID)

Вход через SSO доступен **только клиентам — физическим лицам (ФЛ)**. Для клиентов — юридических лиц (ЮЛ) используется только вход по логину и паролю. С точки зрения ИБ: для ЮЛ один аккаунт под контролем организации; снижение риска привязки корпоративного доступа к личному аккаунту ФЛ; соответствие 152-ФЗ при получении ПДн от провайдера — минимальный объём и определённые цели.

**Согласовано (протокол №6):** раздельные входы для клиентов и для сотрудников; на текущем этапе в приоритете — **Яндекс ID** и **VK ID**. Госуслуги (ЕСИА) могут быть добавлены по отдельному решению.

- **Яндекс ID** — OAuth 2.0 / OpenID Connect; идентификация по аккаунту Яндекса.
- **VK ID** — аналогчно, вход через аккаунт ВКонтакте.

**Реализация:**

- Связь учётной записи клиента (ФЛ) с идентификатором субъекта у провайдера (`sub` в id_token).
- Запрос у провайдера только необходимых scope (идентификация; при необходимости e-mail/телефон). Обработка данных от провайдера — в соответствии с 152-ФЗ.
- Защита от CSRF: использование параметра `state` при редиректе на провайдера и проверка при возврате. Для клиентов без сохранения client_secret (SPA, мобильные приложения) — использование PKCE (code_verifier / code_challenge).
- Хранение токенов (access/refresh): минимально необходимое для сессии; ограниченный срок жизни; обновление по refresh с логированием; не хранить refresh дольше необходимого.
- Пароль провайдера не хранится; при первом входе через SSO — создание или привязка профиля клиента (ФИО/контакты при необходимости из ответа провайдера или вводом пользователя).
- Политика авторизации после входа через SSO та же, что и при входе по паролю.

## 2FA (TOTP) для сотрудников

Для всех сотрудников (Охранник, Управляющий, Владелец) при входе в админ-интерфейс и на посту КПП предусмотрена **обязательная двухфакторная аутентификация по TOTP** (RFC 6238):

- Второй фактор — одноразовый код из приложения-аутентификатора (Google Authenticator, Яндекс.Ключ, аналог).
- Секрет TOTP при привязке устройства хранится в системе в зашифрованном виде и привязан к учётной записи сотрудника; не передаётся третьим лицам.
- Защита от перебора кодов TOTP: лимит неудачных попыток ввода кода за сессию/период; при превышении — блокировка или задержка; логирование подозрительных попыток.
- Настройка (привязка нового устройства) — через защищённый канал, с подтверждением личности сотрудника (логин + пароль); при смене устройства — процедура перепривязки только после повторной аутентификации (логин + пароль) и при необходимости с санкции Управляющего/Владельца.
- Рекомендуется: одноразовые резервные коды при включении TOTP (хранение в виде хэша, одноразовое использование; ограниченное количество кодов) для восстановления доступа при потере устройства.
- Политика авторизации после прохождения 2FA не меняется — действует RBAC по ролям.

**Согласование:** обязательность 2FA по TOTP для всех сотрудников зафиксирована в настоящем артефакте; при расхождении с Контекстной диаграммой («при необходимости OTP») приоритет — у данного артефакта до обновления диаграммы по решению заказчика.
