## Как использовать глоссарий

- использовать как единый словарь терминов для артефактов системного анализа;
- опираться на него при написании требований, use case, ADR и UI-описаний;
- при расхождении с устаревшими формулировками в старых материалах ориентироваться на этот документ и связанные актуальные артефакты;
- если термин меняет смысл в ходе проектирования, обновлять его здесь и в ключевых артефактах проекта.

## 1. Основные термины предметной области

| Термин | Определение |
|--------|-------------|
| **Клиент** | Пользователь парковки, который получает услуги парковки и взаимодействует с системой как физическое или юридическое лицо. |
| **ФЛ** | Физическое лицо, использующее парковку для личных целей. Может пользоваться разовой парковкой, бронированием или договором. |
| **ЮЛ** | Юридическое лицо, использующее парковку через организацию и ее представителей. Как правило, управляет списком ТС, доступами и условиями использования парковки в рамках договора. |
| **Организация** | Юридическое лицо, от имени которого действует представитель клиента ЮЛ. Хранит реквизиты, данные для договоров и расчетов. |
| **Транспортное средство (ТС)** | Автомобиль, мотоцикл или другое средство передвижения, которое клиент использует для въезда, размещения и выезда с парковки. |
| **ГРЗ** | Государственный регистрационный знак транспортного средства, который используется для идентификации ТС при въезде и выезде. |
| **Парковка** | Объект бизнеса, для которого создается цифровая платформа: территория с парковочными местами, секторами, КПП, правилами доступа и тарифами. |
| **Сектор** | Логически или физически выделенная часть парковки, например зона, участок или этаж. Используется для навигации, распределения мест и применения правил. |
| **Парковочное место (ПМ)** | Конкретное место на парковке, которое может быть закреплено по договору, использовано в бронировании или занято в рамках парковочной сессии. |
| **Слот в секторе** | Единица краткосрочного планирования в рамках сектора: назначение доступного ресурса (и/или временного интервала) для допуска конкретного ТС, используемое при создании/проверке бронирования и связанной парковочной сессии. |
| **Тип зоны** | Категория парковочной зоны, определяющая ее правила использования. Например: краткосрочная, долгосрочная, зона для ОВЗ, зона для мотоциклов или резерв. |
| **Параметры ТС** | Характеристики транспортного средства, которые система учитывает при размещении на парковке и применении правил. В проекте прежде всего используются тип и категория ТС. |
| **Тип ТС** | Категория транспортного средства, определяющая типовые правила использования парковки и/или расчет тарифа. Например: легковой автомобиль, мотоцикл, муниципальная техника. |
| **Льготная категория** | Категория клиента, влияющая на доступ в определенные зоны и на применение льготных условий или скидок. Например: ОВЗ, пенсионер, ветеран. |
| **ОВЗ** | Ограниченные возможности здоровья. В проекте термин используется для обозначения клиентов, которым могут предоставляться специальные парковочные зоны и льготные условия. |

## 2. Термины, связанные с использованием парковки

| Термин | Определение |
|--------|-------------|
| **Тариф** | Набор правил расчета стоимости использования парковки в зависимости от времени, длительности, типа клиента, типа ТС, типа зоны и других условий. |
| **Бронирование** | Запись о предоставлении права пользоваться парковкой в определенный период. Для краткосрочного сценария бронирование включает бронь места или `Слот в секторе`; для долгосрочного сценария — бронирование, созданное на основании договора. Бронирование используется как основа для расчетов и проверки `Право доступа`. |
| **Автоматическое бронирование** | Бронирование, которое система создает автоматически при въезде конкретного ТС, если до события нет действующего бронирования, достаточного для допуска. Используется как контекст для парковочной сессии, расчетов и последующих проверок `Право доступа`. |
| **Договор** | Соглашение с клиентом о длительном пользовании парковкой, в рамках которого определяются сроки, тарифы, условия расчетов и закрепление мест. На основании договора в системе создаются связанные бронирования на период его действия. |
| **Парковочная сессия** | Фактическое использование парковки конкретным ТС: с момента въезда до момента выезда. Сессия фиксирует реальное время и КПП въезда/выезда и связана с бронированием, по которому автомобиль был допущен на парковку. |
| **Право доступа** | Результат вычисления бизнес-правил допуска для конкретной попытки въезда/выезда конкретного ТС в конкретный момент времени. Выражается как разрешение/запрет и может содержать основание ограничения. |
| **Задолженность** | Непогашенная стоимость использования парковки или неисполненное финансовое обязательство клиента перед парковкой. Учитывается при расчете `Право доступа`, в том числе при выезде. |
| **Черный список (ЧС)** | Перечень ограничений доступа по конкретным клиентам и/или ТС, сформированный на основании нарушений/конфликтных ситуаций/решений безопасности. Каждая запись фиксирует основание ограничения и применяется при вычислении `Право доступа`. |
| **Грейс‑период** | Максимальное время (в минутах) от момента успешной оплаты или окончания оплаченного периода до выезда, в течение которого повторная тарификация не производится и клиент может выехать без повторного начисления стоимости (если иное не задано правилами тарифа). |
| **Решение допуска (разрешить / запретить)** | Команда СКУД, которая на основании `Право доступа` открывает или не открывает `Шлагбаум` на КПП для соответствующего направления (въезд/выезд). |

## 3. Термины, связанные с оплатой и документами

| Термин | Определение |
|--------|-------------|
| **Платеж** | Операция оплаты услуг парковки, выполняемая в контексте конкретного бронирования. Может относиться к использованию парковки по договору или к фактической парковочной сессии через связанное бронирование. |
| **Чек** | Фискальный документ, подтверждающий успешную оплату, сформированный через ОФД. |
| **ОФД** | Оператор фискальных данных, через которого происходит фискализация чеков по оплатам парковки. |
| **ЭДО** | Электронный документооборот, используемый для обмена договорами, счетами, актами и другими документами, в первую очередь в сценариях с ЮЛ. |
| **Шаблон договора** | Преднастроенная форма договора, на основе которой может формироваться конкретный договор клиента. |

## 4. Термины, связанные с коммуникацией и обслуживанием

| Термин | Определение |
|--------|-------------|
| **Уведомление** | Сообщение, которое система отправляет клиенту или сотруднику по определенному событию. Например: оплата, задолженность, окончание договора, изменение доступа. |
| **Шаблон уведомления** | Преднастроенный текст и правила формирования уведомления для конкретного типа события. |
| **Согласие** | Зафиксированное волеизъявление клиента по юридически значимому вопросу, например согласие с офертой, обработкой ПДн или маркетинговыми рассылками. Используется для соблюдения правовых требований и настройки коммуникаций. |
| **Обращение** | Зафиксированное взаимодействие клиента с системой или сотрудниками по вопросу, жалобе, претензии или предложению. |
| **Личный кабинет (ЛК)** | Пользовательский интерфейс клиента, в котором доступны профиль, ТС, бронирования, договоры, уведомления, оплаты и история использования парковки. |
| **PWA** | Формат клиентского веб-приложения, приближенного по пользовательскому опыту к мобильному приложению и доступного с мобильных устройств. В проекте используется для клиентского контура. |

## 5. Термины, связанные с доступом и инфраструктурой

| Термин | Определение |
|--------|-------------|
| **Контрольно-пропускной пункт (КПП)** | Точка въезда и выезда на парковку, где осуществляется контроль доступа транспортных средств и взаимодействие клиента с персоналом и системой. |
| **СКУД** | Система контроля и управления доступом, которая получает события въезда и выезда и исполняет решение платформы о допуске или запрете проезда через КПП. |
| **LPR-камеры** | Камеры распознавания автомобильных номеров, которые фиксируют ГРЗ на въезде и выезде и передают результат распознавания в систему. |
| **Проксимити-карта** | Средство доступа, используемое в текущем AS-IS процессе для идентификации клиента в СКУД. Для целевой модели не является основным способом допуска, но важно как часть исходного контекста и переходного периода. |
| **Шлагбаум** | Исполнительный механизм на КПП, который СКУД может открывать или оставлять закрытым по результату проверки допуска (въезд/выезд). |
| **Режим деградации** | Сценарий работы системы при отказе части сервисов или интеграций, когда допуск на КПП должен обеспечиваться по заранее определенным правилам запасного поведения. |

## 6. Термины, связанные с ролями и контурами системы

| Термин | Определение |
|--------|-------------|
| **Охранник КПП** | Сотрудник, работающий на КПП и обрабатывающий исключения, спорные ситуации, ручные проверки и ручные операции допуска. |
| **Управляющий** | Сотрудник, отвечающий за операционное управление парковкой: тарифы, доступы, задолженности, обращения, настройки и контроль процессов. |
| **Владелец** | Пользователь с максимальным уровнем управленческого доступа, принимающий ключевые бизнес-решения и контролирующий эффективность работы парковки. |
| **Сотрудник** | Общий термин для внутренних пользователей системы: охранник, управляющий, владелец. |
| **Business Site** | Публичный информационный сайт парковки для незарегистрированных клиентов: тарифы, правила, контакты, карта парковки и переход в ЛК. |
| **Consumer App** | Клиентское веб-приложение с личным кабинетом, в котором клиент управляет своим профилем, ТС, бронированиями, оплатами и договорами. |
| **Admin Panel** | Административный контур системы для управляющего и владельца, где доступны функции настройки парковки, управления данными и контроля процессов. |
| **Custom Internal Tool** | Внутренний рабочий контур для сотрудников с CRM-функциями, работой с договорами, обращениями, аналитикой и вспомогательными операциями. |

## 7. Термины, связанные с аутентификацией и безопасностью

| Термин | Определение |
|--------|-------------|
| **SSO** | Вход через внешнего провайдера идентификации. В проекте используется только для клиентов ФЛ. |
| **TOTP** | Time-based One-Time Password: одноразовый код (пароль), генерируемый по времени в приложении-аутентификаторе; используется как обязательный второй фактор для сотрудников. |
| **RBAC** | Разграничение доступа на основе ролей. В проекте используется для сотрудников: охранник, управляющий, владелец. |
| **ПДн** | Персональные данные клиентов и сотрудников, которые система обрабатывает в рамках работы парковки и должна защищать в соответствии с требованиями законодательства. |
| **КИД** | Конфиденциальность, целостность, доступность — триада свойств, по которым оценивают влияние угроз и рисков на активы и сервис. |
| **LPR** | License Plate Recognition: распознавание ГРЗ по изображению/видеопотоку; в проекте используется на КПП и интегрируется с `СКУД` (оборудование) и платформой (бизнес-решение допуска). |
| **MFA** | Многофакторная аутентификация: требование использовать два и более независимых факторов для подтверждения личности. В проекте применяется для сотрудников (в сочетании с `TOTP`) при входе и выполнении привилегированных операций. |
| **2FA** | Двухфакторная аутентификация: MFA, при которой требуется ровно два фактора. В проекте второй фактор реализован через `TOTP`. |
| **OAuth 2.0** | Протокол авторизации для выдачи токенов доступа. В проекте используется как базовая основа для SSO/федеративного входа, поверх которой применяются `OIDC` и `PKCE`. |
| **OIDC** | OpenID Connect: слой аутентификации над OAuth 2.0, позволяющий получать сведения об идентичности пользователя для SSO-сценариев. |
| **PKCE** | Proof Key for Code Exchange: механизм защиты OAuth 2.0, снижающий риск перехвата авторизационного кода (актуален для публичных клиентов). |
| **IdP** | Identity Provider: провайдер идентичности, который аутентифицирует пользователя и выдает токены/утверждения для `OIDC`-входа. |
| **IDOR** | Insecure Direct Object Reference: уязвимость, при которой доступ к объекту определяется предсказуемым идентификатором объекта без корректной проверки прав/владельца. |
| **SSRF** | Server-Side Request Forgery: атака, при которой сервер выполняет запросы к произвольным URL/адресам, подставленным злоумышленником. |
| **MITM** | Man-in-the-Middle: атака перехвата/подмены трафика между участниками обмена с целью подмены данных или событий. |
| **HMAC** | Hash-based Message Authentication Code: код аутентификации на основе хэша, используемый для проверки целостности и подлинности сообщений (например, подписи webhook). |
| **CVE** | Common Vulnerabilities and Exposures: публичный реестр идентификаторов известных уязвимостей, применяемый для мониторинга рисков и обновлений компонентов. |
| **DMZ** | Demilitarized Zone: изолированный сетевой сегмент между публичной сетью и внутренними ресурсами для снижения последствий компрометации. |
| **XSS** | Cross-Site Scripting: уязвимость, позволяющая внедрить вредоносный скрипт в веб-контент/ответ и выполнить его в браузере пользователя. |
| **CSRF** | Cross-Site Request Forgery: уязвимость, при которой браузер пользователя выполняет запрос от имени пользователя без явного подтверждения действия. |
| **TLS** | Transport Layer Security: криптографический протокол защищенного транспорта, обеспечивающий шифрование и проверку соединения. |
| **HSTS** | HTTP Strict Transport Security: заголовок-политика, принуждающий браузер всегда использовать HTTPS для домена. |
| **RFC-1918** | Приватные диапазоны IP-адресов (10/8, 172.16–31/12, 192.168/16). В анализе используется как ориентир для фильтрации попыток обращения к внутренней сети. |
| **DR** | Disaster Recovery: план восстановления после крупной аварии/катастрофы, включая процедуры возврата системы в работоспособное состояние (например, по резервным копиям). |
| **SAST** | Static Application Security Testing: статическое тестирование/анализ кода на наличие уязвимостей до деплоя (например, в CI/CD). |
| **WAF** | Web Application Firewall: межсетевой фильтр HTTP-трафика для защиты веб-приложений от известных паттернов атак и некорректных запросов. |
| **DevOps** | Development & Operations: подход к организации разработки и эксплуатации для обеспечения повторяемых и контролируемых процессов релиза/обновлений (в т.ч. безопасность в pipeline). |
| **API** | Application Programming Interface: программный интерфейс системы (эндпойнты/контракты), через который выполняются запросы клиентов и интеграций; в ИБ фокус — на правах, валидации и защите от типовых атак. |
| **UDP** | User Datagram Protocol: протокол передачи данных без установления соединения; используется в контексте защищенности канала обмена (в т.ч. при передаче телеметрии/событий). |
| **KMS** | Key Management Service: сервис управления ключами шифрования (хранение, ротация, выдача и контроль доступа к ключам). |
| **DoS / DDoS** | Denial of Service / Distributed DoS: атака отказа в обслуживании (в DDoS — распределенная), целью которой является недоступность сервиса или деградация качества обслуживания. |

## 8. Методы и артефакты системного анализа

Эти термины не относятся напрямую к предметной области парковки, но активно используются в работе команды над проектом.

| Термин | Определение |
|--------|-------------|
| **Event Storming** | Метод совместного моделирования бизнес-процессов через цепочку событий, команд, правил и ролей. Используется для понимания логики предметной области и поиска границ системы. |
| **BPMN** | Нотация для формального описания бизнес-процессов, ролей и взаимодействий между участниками процесса. |
| **Opportunity Canvas** | Артефакт для фиксации проблем, ограничений, возможностей и ожидаемых эффектов бизнеса. |
| **Impact Map** | Артефакт, связывающий бизнес-цели проекта с изменениями в поведении пользователей и с возможностями системы. |
| **User Story Map** | Артефакт, который помогает структурировать пользовательские сценарии, ожидания пользователей и состав MVP. |
| **ADR** | Architectural Decision Record — документ, фиксирующий важное архитектурное решение, его контекст, альтернативы и последствия. |
| **Wireframe** | Упрощенный макет интерфейса, который используется для визуализации сценариев и проверки структуры экранов без реализации полноценного продукта. |
| **Bow-Tie (галстук-бабочка)** | Метод анализа риска, визуализирующий пути “слева” (источники угрозы и уязвимости), “в центре” (центральное событие) и “справа” (последствия), с фиксацией превентивных и реактивных барьеров. |
| **OWASP Top 10 (2021)** | Чек-лист наиболее критичных классов уязвимостей веб-приложений, используемый как опорный набор для построения/проверки перечня уязвимостей в анализе угроз. |

## 9. Замечания по использованию терминов

- термин `Проксимити-карта` относится прежде всего к текущему AS-IS состоянию и не должен подменять целевую модель допуска через `ГРЗ`, `СКУД`, `LPR` и решение `разрешить / запретить`;
- термин `СКУД` в проекте используется как часть интеграционного контура, а не как носитель основной бизнес-логики доступа;
- `Бронирование` и `Парковочная сессия` не являются взаимозаменяемыми терминами: бронирование описывает план и право использования парковки, а сессия — факт использования;
- `Задолженность` и `Черный список (ЧС)` в проекте рассматриваются как разные основания для ограничения допуска: задолженность связана с оплатой/расчетами, а черный список — с конфликтами/нарушениями/решениями безопасности;
- если в отдельных ранних протоколах встречаются более узкие или устаревшие трактовки терминов, приоритет следует отдавать актуальной карточке проекта, контекстной диаграмме, концептуальной модели и ADR.
