## Назначение

Этот документ помогает добавлять новые артефакты без потери структуры и навигации.

Он особенно полезен для материалов, которые изначально существуют в виде картинок, скриншотов, экспортов из draw.io, drawsql.app или других визуальных инструментов.

## Базовые принципы

- Каноничным артефактом в репозитории должен считаться Markdown-файл, а не изображение.
- Изображение хранится рядом как приложение или визуальное представление артефакта.
- Один смысловой артефакт лучше держать в одном каталоге вместе с его `md`-описанием и связанными файлами.
- Материалы демо не должны подменять собой каноничные артефакты.
- Если артефакт затрагивает требования, архитектуру или структуру репозитория, обновляйте `docs/process/traceability-matrix-log.md`.
- Текст Markdown-ссылок должен быть человекочитаемым: используйте название артефакта, а не путь или имя файла.

## Куда размещать артефакты по этапам

| Этап | Что относится | Куда класть |
|------|---------------|-------------|
| Этап 1. Моделирование бизнеса | AS-IS, BPMN, ES AS-IS, UML Class Diagram, UML StateChart Diagram | `docs/artifacts/as-is/` |
| Этап 2. Концептуальное проектирование | Opportunity Canvas, Impact Map, project charter, User Story Map, navigation map, контекст, use case, концептуальная модель, ES TO-BE | `docs/artifacts/`, `docs/artifacts/use-case/`, `docs/artifacts/es-to-be/`, `ui/` |
| Этап 3. Требования к ПО | FR, NFR, ограничения | `docs/specs/` |
| Этап 4. Архитектура и БД | ADR, DDD, C4, архитектурные решения, ES-артефакты, которые стали входом в архитектурную проработку | `docs/architecture/` |
| Этап 4. ERD и практика SQL | ERD, проработка физической модели, SQL-практика | `docs/architecture/database/erd/` и `sql/practice/` |
| Этап 4. Инфобез | анализ угроз, bow-tie, контекстные материалы | `docs/artifacts/infosec/` |
| Этап 4. Алгоритмизация | DRAKON и учебные алгоритмы | `docs/artifacts/algorithms/` |
| Этап 5. Интеграции | сценарии, sequence, DFD, API, контракты, ошибки, журналирование | `docs/architecture/integration/` |
| Demo Days | сценарий выступления, слайды, выжимка для показа | `docs/demo-days/demo-N/` |

## Как хранить артефакты, пришедшие как изображения

Рекомендуемый паттерн для каждого артефакта:

```text
docs/artifacts/as-is/
  bpmn-parking-as-is.md
  assets/
    bpmn-parking-as-is.png
```

Правила оформления:

- `*.md` должен объяснять смысл артефакта человеческим языком.
- В `*.md` должна быть вставка изображения через относительную ссылку.
- Если изображение обновляется, файл `md` обновляется в том же изменении.
- Если артефакт имеет несколько итераций, используйте версионирование в имени файла или отдельный блок истории внутри `md`.
- Если ES-артефакт перестал быть только аналитической схемой и используется как вход в DDD/C4, его можно перенести из `docs/artifacts/es-to-be/` в `docs/architecture/ddd/` с обновлением всех ссылок в том же изменении.
- В подписях ссылок используйте названия вроде `Event Storming AS-IS`, `Индекс архитектуры`, `Шаблон артефакта из изображения`, а не `event-storming-as-is.md`, `readme.md` или `../templates/...`.

Для стартового шаблона используйте [Шаблон артефакта из изображения](templates/artifact-from-image-template.md).

## Рекомендуемая структура каталогов

Ниже приведен рекомендуемый каркас для дальнейшего развития репозитория без массового переезда уже существующих файлов:

```text
docs/
  artifacts/
    as-is/
    es-to-be/
    use-case/
    infosec/
    algorithms/
  architecture/
    adr/
    ddd/
    c4/
    database/
      erd/
    integration/
  specs/
    functional-requirements/
    nonfunctional-requirements/
    constraints/
  process/
  interviews/
    plans/
    protocols/
  demo-days/
    demo-1/
    demo-2/
    demo-3/
    demo-4/
    demo-5/
```

Если артефакт готовится для демо, но уже является частью каноничного анализа, размещайте основной файл в профильном разделе, а в `demo-days/` кладите только презентационную адаптацию или ссылки.

## Именование файлов

- Для новых файлов используйте lowercase `kebab-case`.
- Для индексов разделов используйте `readme.md`.
- Для диаграмм из изображений удобно сохранять одинаковое имя у `md` и файла изображения.
- Для версий и дат используйте понятные суффиксы, например `-v01`, `-2026-04-02`.

## Чеклист перед добавлением нового артефакта

1. Определите, какой этап проекта и какой раздел репозитория являются для артефакта каноничными.
2. Создайте `md`-файл с текстовым описанием.
3. Положите изображение рядом в `assets/` или в тот же каталог, если вложенность не нужна.
4. Добавьте относительные ссылки на связанные артефакты с человекочитаемыми подписями, а не путями.
5. Обновите `readme.md` нужного раздела, если появился новый подпункт навигации.
6. Обновите `docs/process/traceability-matrix-log.md`, если изменение содержательное.
