# Контекст сессии: ФТ парковочной сессии и доработка шаблона FR

**Дата:** 2026-03-28
**Тема:** подготовка `docs/specs/functional-requirements/fr-parking-session.md`, согласование с реестром UC, концептуальной моделью и правилами; итеративное устранение «воды»; обновление шаблона и правила `create-fr`.

## Исходный запрос

Составить функциональные требования для объекта «парковочная сессия» по правилу [`.cursor/rules/create-fr.mdc`](../../.cursor/rules/create-fr.mdc) и шаблону [`_template-fr.md`](../specs/functional-requirements/_template-fr.md), с опорой на реестр UC, UC-артефакты и концептуальную модель.

## Ключевые решения по содержанию ФТ

- Объект в кандидатах: **`ПарковочнаяСессия`**; финансовый мастер-контекст остается у **`Бронирование`** / **`Платеж`**; сессия фиксирует время, КПП, статус, связь `бронированиеИд`.
- Идентификаторы: **`FR-PARKSESSION-001`** … **`FR-PARKSESSION-017`** (в т.ч. кросс-объектный шаг **`ПарковочнаяСессия.Завершить.СД.завершениеСвязанногоБронирования`** с отсылкой на [`fr-booking.md`](../specs/functional-requirements/fr-booking.md)).
- Трассировка **UC** — только в разделе **«Входные UC»** и в строках реестра **`FR-*`** (`— UC-x.y`), не в тексте формулировок под ролями и не в **`## Системы`**.
- Родительские **`[Объект].[Действие].СД`**: одно предложение (триггер + вход); условия и алгоритмы — в подпунктах **`.СД.[атрибут]`** или **`.СД.[имя]`** с формулировкой **«по следующему алгоритму»**.
- Списки у ролей: одна вводная фраза по шаблону; атрибуты перечислением; представления связей вида **`кппВъездаИд — `[КПП]:наименование`**; фильтры списков — отдельная строка **`Фильтры:`**; поля, задаваемые системой при сохранении (охранник) — в том же списке с пометкой **(система при сохранении)** вместо второго абзаца.
