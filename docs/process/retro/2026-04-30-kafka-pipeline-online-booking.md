# Ретро: Kafka-конвейер для процесса онлайн-бронирования (учебный TO-BE)

**Дата:** 2026-04-30
**Длительность:** одна сессия (~2 часа от Фазы 0 до Фазы 8)
**Связанный план:** [plans/2026-04-30-kafka-pipeline-online-booking.md](../../../plans/2026-04-30-kafka-pipeline-online-booking.md)
**Агент:** Claude Code (оркестратор + 3 параллельных general-purpose исполнителя)

## 1. Задача

Подготовить ДЗ курса по теме брокеров сообщений: ADR с обоснованием выбора Kafka, двухуровневую DFD конвейера потоков данных в Mermaid и две таблицы требований к Kafka. Все три артефакта — учебный TO-BE поверх ADR-003 для сквозной цепочки «бронь → оплата → подтверждение → уведомление» (склейка UC-12.2 + UC-10.2 + постусловное уведомление).

## 2. Как решал

Работа шла по плану из `plans/`, фазы 0–8. Использовал параллельную многоагентную схему:

- Фаза 0 (контекст и план) — план был согласован пользователем заранее, с зафиксированными решениями A–G по открытым вопросам (включая антипаттерн-решения по `Topic_InvoiceCreated` и `Pricing Service`, и пункт G про dual-write через outbox + CDC). Это сильно упростило брифы агентов — оркестратор передавал готовые решения, а не выводил их.
- Волна 1 — три general-purpose агента параллельно. Контракт строгий: один агент = один файл (ADR-007 / DFD / requirements), пересечений нет, индексные файлы не трогать. У каждого свой развернутый промт с источником истины (план), референсами (JPG из `plans/kafka/`, ADR-005/006 как шаблон ADR), форматом отчета (путь файла + список ключевых решений + что пришлось додумать).
- Сборка (Фазы 5–6) — выполнил сам через Edit:
  - индекс `docs/architecture/adr/readme.md` дополнен записью про ADR-007 (с упоминанием трех антипаттерн-решений как teaser);
  - индекс `docs/architecture/integration/readme.md` дополнен ссылками на DFD и requirements в секции «Текущие материалы»;
  - запись `CHG-20260430-006` добавлена в журнал трассировки.
- Фаза 7 (`npm run ci:check`) — `lint:md:legacy` упал на длине строки в журнале трассировки: первая версия записи была 788 символов, лимит 500. Сократил до 477 за две итерации (главное — упаковать список файлов через раскрываемый brace-pattern `docs/architecture/{adr/{...},integration/{...}}.md`).
- Фаза 8 — заполнение `## Итог` в плане, отметка фаз `[x]`, написание этого ретро.

## 3. Решил ли

Да, полностью.

- ADR-007 создан в стиле ADR-005/006 (Status / Context / Options / Decision / Rationale / Trade-offs / Consequences / Next steps). Status явно фиксирует учебный TO-BE и три расхождения с ADR-003. Decision явно отбрасывает три антипаттерна.
- DFD `message-flow-kafka-online-booking.md` содержит две Mermaid-диаграммы (K-L1 — обзор с одним блоком Kafka, K-L2 — три топика синими прямоугольниками), словарь потоков для каждой, текстовый сценарий в 10 шагов, таблицу балансировки с UC-12.2/UC-10.2 и примечание про схемы PostgreSQL. Префикс `K-` отделяет учебный артефакт от производственного [DFD L1](../../artifacts/dfd-l1.md).
- `kafka-requirements.md` содержит обе таблицы по формату телемед-референса (технические параметры + семантика доставки) для трех топиков с partition key `bookingId`, retention 14/90/14 дней, `enable.idempotence=true` + `acks=all` + дедупликация по `eventId` (UUID v7).
- Индексы и журнал трассировки обновлены. `npm run ci:check` зеленый (включая `lint:mermaid` — обе диаграммы валидны).
- Соблюдены правила: учебный TO-BE живет только в `docs/architecture/`, `docs/specs/` не тронут, ADR-003 не правлен, имена файлов латиница kebab-case, буква «е с двумя точками» нигде не используется.

## 4. Эффективность

Что сработало хорошо:

- **Параллельные агенты с непересекающимся scope.** Три агента писали три независимых файла по 2–4 минуты каждый — суммарное реальное время Волны 1 ≈ время самого медленного. Жесткое правило «не трогай чужие файлы и индексы» сработало: ни одного конфликта при сборке.
- **План с зафиксированными решениями A–G.** Пользователь заранее закрыл открытые вопросы (включая «двойной» антипаттерн `Pricing` + `Topic_InvoiceCreated` и dual-write через outbox + CDC). Это превратило промты из «обсуди и реши» в «сформулируй уже принятое». Агенты не уходили в сторону, не предлагали альтернатив.
- **Упоминание JPG-референсов в промтах.** Агенты по DFD и requirements сами открыли JPG из `plans/kafka/` через Read и держали палитру/колонки соответствующими образцу. Это сняло необходимость проверять формат вручную.
- **Прогон ADR-005/006 как шаблона стиля.** ADR-007 получился структурно идентичен ADR-005/006, без переизобретения секций. В Consequences даже добавлен Mitigation для парности с эталонами — это инициатива агента, но укладывающаяся в сложившийся стиль.

Что можно сделать лучше:

- **Лимит строки в `traceability-matrix-log.md` снова сюрпризом.** Тот же случай, что в ретро 2026-04-30 по ЮKassa — 500 символов. После прошлого случая стоило сразу писать запись короче. Теперь это паттерн: для журнала тратеабилити писать одной строкой меньше 480 символов с самого начала.
- **Учебная пометка про фискализацию ОФД.** Агент по ADR добавил отдельный абзац в Status про «фискализация out of scope» — это сверх плана, но полезно. План это не требовал явно — стоило прописать в брифе. На будущее: бриф ADR-агента должен содержать список явных out-of-scope позиций (помимо включаемых).
- **Дублирование расхождений с ADR-003 в трех файлах.** ADR-007, DFD и requirements каждый отдельно повторяют список расхождений (инв. 4, инв. 5, переопределение `Notification Worker`). Это намеренно — каждый артефакт читается самостоятельно, — но при будущем изменении формулировки надо будет менять в трех местах. Возможный путь — вынести в отдельный артефакт и ссылаться. В этой сессии не делал, потому что для трех артефактов оверхед инклюда дороже дубля.
- **Mermaid с двойным лейблом стрелки.** На K-L1 у Booking стрелки от Kafka объединены лейблом `BookingCreated, BookingConfirmed` — упрощение, чтобы не дублировать стрелки. На K-L2 события идут раздельно из разных топиков, что корректнее. Парность ОК, но если в будущем добавятся события (например, `BookingCancelled`), агрегированный лейбл начнет ломаться. На будущее — на K-L1 делать раздельные стрелки даже за счет объема.
