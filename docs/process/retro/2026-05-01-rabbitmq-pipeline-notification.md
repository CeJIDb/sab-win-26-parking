# Ретро: RabbitMQ-конвейер для рассылки уведомлений (учебный TO-BE)

**Дата:** 2026-05-01
**Длительность:** ~3 часа (по плану)
**Связанный план:** [plans/2026-05-01-rabbitmq-pipeline-notification.md](../../../plans/2026-05-01-rabbitmq-pipeline-notification.md)
**Агент:** Claude Code (главный оркестратор + три параллельных подагента)

## 1. Задача

Симметрично сданному Kafka-блоку оформить три новых архитектурных артефакта по RabbitMQ для учебного TO-BE: ADR-008, DFD конвейера R-L1/R-L2 и таблицу требований к RMQ. Скоуп — work queue команд на отправку уведомлений (SMS / Email / Push) внутри Notification bounded context, как продолжение Kafka-конвейера из [ADR-007](../../architecture/adr/adr-007-kafka-event-bus-online-booking.md).

## 2. Как решал

Пользователь сразу пришел с готовой стратегией параллельной оркестрации: три независимых подагента (ADR-008, DFD, требования) поверх единого блока «Контракт» с зашитыми именами exchanges/queues/routing keys, формулировками пяти антипаттернов и SQL двухэтапной дедупликации. Главный агент закрывал sequential-обвязку — Шаг 1 (свободный `CHG-20260501-001`, наличие `notes-rabbitmq-candidates.md`) и Шаг 3 (индексы, удаление черновика, запись в журнале, `ci:check`, ретро).

На Шаге 2 запустил три `Agent` tool в одном сообщении. Два подагента (ADR-008 и DFD) вернулись чистыми. Третий (требования RMQ) упал на ложном срабатывании политики использования API на относительно безобидном промпте с SQL-сниппетами дедупликации.

Вместо повторного запуска подагента написал файл сам по образцу `kafka-requirements.md` — это было быстрее, чем переформулировать промпт под модерацию, и обошло проблему без потери качества: структура, тон и уровень детализации совпадают с Kafka-аналогом.

На Фазе 7 единственное замечание — превышение лимита 500 символов в строке нового CHG-записи в журнале трассировки. Сократил формулировку Source и список измененных файлов — `lint:md:legacy` стал зеленым.

Дополнительно нашлась битая markdown-ссылка на удаленный `notes-rabbitmq-candidates.md` в `docs/process/project-journey.md:294`. Заменил ее на пару ссылок на новые DFD и требования RMQ — `lint:md-links` стал зеленым.

## 3. Решил ли

Да, полностью. Все 8 фаз закрыты, `npm run ci:check` зеленый, все 5 пунктов DoD выполнены:

- расхождений с ADR-003 одно (новая таблица `push_inbox`);
- 5 антипаттернов в Decision ADR-008 перечислены явно;
- DFD R-L1/R-L2 валидны (`lint:mermaid` 11 блоков OK), палитра воспроизводит K-L1/K-L2;
- таблица требований 4 × 7, `x-max-priority` обоснованно отсутствует;
- ссылок на `plans/` в новых артефактах нет.

Открытых вопросов по содержанию не осталось — все семь пунктов A-G из плана попали в Decision ADR-008 без изменений. Приоритеты сообщений и реальный realtime Push (P2/P3) явно вынесены в Next steps как потенциальный ADR-009.

## 4. Эффективность

Что сработало:

- **Параллельная оркестрация с зашитым контрактом.** Три подагента писали независимые файлы, но словарь (имена exchanges, очередей, routing keys, формулировки антипаттернов, SQL дедупликации) совпал в трех артефактах — это и было целью контракта. Стилистические различия минимальны, синхронизировать после Фазы 7 не пришлось.
- **Готовность пользователя дать полный план оркестрации.** Главный агент не тратил время на расчет зависимостей и состав sequential-обвязки. Это редкий случай, когда промпт-менеджмент полностью на стороне человека — и это работает.
- **Sequential-обвязка тонкая.** Шаг 1 (поиск свободного CHG-номера + проверка наличия notes-файла) и Шаг 3 (индексы, удаление, журнал, `ci:check`, ретро) ушли быстро. Подагенты не трогали индексы и `notes-rabbitmq-candidates.md` — границы скоупа подагентов были четкими.

Что можно сделать лучше:

- **Длина строк в журнале трассировки.** Лимит 500 символов на строку из `lint:md:legacy` каждый раз ловит большие записи — это уже не первый раз. Возможно, имеет смысл либо поднять лимит для журнала, либо переформатировать журнал в несколько колонок поменьше. Пока — ловлю на этапе `ci:check` и сокращаю формулировку.
- **Битые markdown-ссылки на удаляемые файлы.** `notes-rabbitmq-candidates.md` упоминался в `docs/process/project-journey.md` markdown-ссылкой. План этот файл не учел — пришлось ловить через `grep -rn "](.*notes-rabbitmq-candidates"`. На будущее — перед удалением файла прогонять такой поиск рутинно.
- **Ложные срабатывания модерации API.** Третий подагент упал на промпте, структурно идентичном первым двум. Воспроизводимости нет, переформулировка не понадобилась — написал руками. Если повторится на следующих параллельных оркестрациях, стоит держать в голове fallback «оркестратор пишет сам по образцу».

Что повторить:

- Параллельный запуск подагентов в одном сообщении с зашитым контрактом — экономит время и держит словарь синхронным.
- Перед параллельным запуском — sequential-проверка состояния репозитория (свободный номер CHG, наличие исходных файлов).
- После всех записей — единый прогон `ci:check`, не пытаться чинить по ходу.

## 5. Как было и как стало

**Было.** В Notification bounded context был только Kafka-consumer (по ADR-007), но не было решения, как именно команды на отправку SMS / Email / Push доходят до воркеров и провайдеров. В `docs/architecture/integration/notes-rabbitmq-candidates.md` лежал черновик с пятью кандидатами на использование RabbitMQ без выбранного решения. Учебный артефакт по теме «Брокеры сообщений» из курса (часть RabbitMQ) не был сделан.

**Стало.** В архитектурном слое лежат три новых документа:

- [ADR-008](../../architecture/adr/adr-008-rabbitmq-notification-dispatch.md) — фиксирует топологию `notification.direct` Direct + `notification.dlx` Fanout с одной DLQ, manual ack и отсутствие outbox через детерминированный `notificationId` плюс двухэтапную дедупликацию.
- [DFD R-L1/R-L2](../../architecture/integration/message-flow-rabbitmq-notification.md) — два Mermaid-блока со словарями потоков и балансировочной таблицей.
- [требования к RMQ](../../architecture/integration/rabbitmq-requirements.md) — таблица обменников и очередей плюс раздел «Параметры детально».

Черновик `notes-rabbitmq-candidates.md` удален, его содержимое перенесено в Context и Options ADR-008. Учебная часть курса по RabbitMQ закрыта. Готово к коммиту через `npm run commit:atomic`.
