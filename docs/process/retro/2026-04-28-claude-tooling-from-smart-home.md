# Ретро: перенос Claude-tooling из smart-home

**Дата:** 2026-04-28
**Длительность:** ~1.5 часа
**Связанный план:** [plans/2026-04-28-claude-tooling-from-smart-home.md](../../../plans/2026-04-28-claude-tooling-from-smart-home.md)
**Агент:** Claude Code (Opus 4.7, 1M context)

## 1. Задача

Перенести из `../smart-home` в parking автоматику Claude Code (хуки, allowlist, команды), которая уже отлажена и закрывает правила CLAUDE.md программно, а не текстом договоренности.

## 2. Как решал

Сначала прошел по `../smart-home` и составил карту полезного: 8 хуков в `scripts/claude-hooks/`, секции `permissions` и `hooks` в `.claude/settings.json`, команда `/plan`, правило `ast-index`, скрипт `check-file-names.mjs`. Отсек Node-RED-специфику (`flow-inspector`, `block-flow-edit`, `/handoff`) и тяжелый skill `bulletproof` — для docs-first проекта избыточны.

Дальше — стандартный рабочий цикл: ветка `chore/claude-tooling-from-smart-home`, план в `plans/`, фазы 1-5. Хуки переносил `cp`, потом адаптировал сообщения (убрал ссылки на правила smart-home, заменил на parking-аналоги). `block-secret-write` упростил — выкинул Home Assistant паттерны (`secrets.yaml`, `known_devices.yaml`). `format-on-write` расширил под parking-расширения (`.css`, `.scss`, `.html`, `.ts`).

Атомарные коммиты делал поштучно `git add <files>` без `commit:atomic` — хотел контроль над разбивкой по фазам, а не по бакетам скрипта. На первом же коммите наткнулся на свой же только что установленный хук: `block-unsafe-git-add` сматчил буквальную фразу `git add -A` в моем commit message (heredoc передается в bash как часть командной строки). Переписал сообщение без триггер-фразы — коммит прошел. Это не баг хука, это его нормальное поведение.

## 3. Решил ли

Да, целиком. Все 5 фаз закрыты, `npm run ci:check` зеленый, smoke-тест блокирующего хука сработал прямо в процессе работы. PR готов к открытию.

## 4. Эффективность

Что сработало:

- Дробление на 8 атомарных коммитов по фазам — каждый закрывает одну подзадачу, легко ревьюить, легко откатить. Стоит повторить.
- Переиспользование уже имеющегося `git-workflow-agent-reminder.mjs` — `remind-atomic-commit.sh` оказался тонкой оберткой, не пришлось дублировать логику.
- `validate-plan-on-write.mjs` поддержал `--staged` и одиночный путь — `validate-plans.mjs` parking-версии оказался совместим без правок.

Что можно было лучше:

- Не упоминать буквальные триггер-фразы хуков в commit messages — стоит запомнить как поведенческое правило для будущих сессий.
- В план изначально стоило добавить пункт про smoke-тест хука прямо в Фазе 4. Я добавил его постфактум.

## 5. Как было и как стало

**До:**

- `.claude/settings.json` содержал только `permissions.deny`. Правила CLAUDE.md (1 «не делай git add -A», Git-политика «не пушь в main», правило 5 «kebab-case латиница») держались только на текстовом договоре с агентом.
- Коммит с битым планом или массовый `git add` — на ответственности агента и code review.
- Auto-format и валидация плана — только через ручной `npm run ci:check`.

**После:**

- 5 блокирующих/валидирующих хуков активны на каждый Bash и Edit/Write вызов агента.
- `lint:file-names` включен в `ci:check` — кириллические имена ловятся автоматически.
- `permissions.allow` для 16 рутинных команд — меньше permission prompt-ов в работе.
- Команда `/plan <slug>` создает план по шаблону.
