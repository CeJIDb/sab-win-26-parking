# docs/process/retro/ — ретроспективы рабочих сессий

Эта папка — место для коротких ретроспектив после каждой нетривиальной сессии работы в репозитории. Одна сессия (чат с агентом, рабочий слот, спринт-день) = один файл. Цель — не отчётность, а осознанный взгляд назад: что сработало, что нет, как работать дальше.

Ретроспектива — **рекомендация, а не обязательство** в v0. Pre-commit хук её не требует. Но чем чаще формат устоится на практике, тем раньше он превратится в часть DoD.

## Когда писать

- После завершения плана из [plans/](../../../plans/) — всегда.
- После сессии, в которой было неожиданное решение, откат, тупик или важный инсайт.
- После сессии, результат которой хочется пересмотреть через неделю, не восстанавливая всё из памяти.

Не пишите ретроспективу ради того, чтобы она была. Если сессия — тривиальная правка одной опечатки, запись не нужна.

## Формат имени файла

`YYYY-MM-DD-краткое-название.md`, kebab-case на латинице, без буквы «ё». Примеры:

- `2026-04-24-gym-app-tooling-adoption.md`
- `2026-04-26-fr-booking-refinement.md`
- `2026-05-01-c4-level-2-review.md`

## Структура файла (пять пунктов)

Адаптирована из паттерна ретро, который хорошо работает в учебных чатах с LLM-агентами.

```markdown
# Ретро: [краткое название сессии]

**Дата:** YYYY-MM-DD
**Длительность:** [чистое время работы]
**Связанный план:** [ссылка на plans/YYYY-MM-DD-*.md, если есть]
**Агент:** [Claude Code / Cursor / человек / смешанная сессия]

## 1. Задача

Что было поставлено перед сессией. Одно-два предложения, без перечисления всего плана.

## 2. Как решал

Ключевые шаги и развилки. Не пересказ коммитов, а логика: «сначала хотел X, наткнулся на Y, переключился на Z».

## 3. Решил ли

Да / нет / частично. Если частично — что осталось и куда это вынесено (новый план, issue, TODO в коде).

## 4. Эффективность

Что можно было сделать лучше: технически, процессно, в планировании. Что стоит повторить в следующих сессиях.

## 5. Как было и как стало

Короткое «до/после»: состояние релевантной части репозитория или процесса до и после сессии. 2–4 строки.
```

## Связь с другими артефактами

- План в [plans/](../../../plans/) — **как** делается задача. Ретро — **что получилось и чему это научило**.
- Журнал трассировки [traceability-matrix-log.md](../traceability-matrix-log.md) — зачем/что изменено в артефактах. Ретро — про сам процесс работы, а не про содержание изменения.
- PR DoR/DoD [pr-dor-dod.md](../pr-dor-dod.md) — формальный чеклист. Ретро — субъективная рефлексия, параллельная ему.

## Что не попадает сюда

- Не дублируем сюда то, что уже в коммитах: описание изменений, затронутые файлы, diff.
- Не пишем здесь решений, которым место в ADR или в спецификации. Ретро — про процесс, а не про артефакт.
- Не превращаем в to-do: открытые задачи переводятся в новые планы или issues.

## Приватность

Ретро коммитим в репозиторий — они публичны для команды. Не пишите сюда чувствительные данные (приватные договорённости с ментором, оценки конкретных участников, персональные данные). Для такого есть локальные заметки и личная переписка.
