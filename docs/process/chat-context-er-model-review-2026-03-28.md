## Текущий набор замечаний (зафиксировано)

Ниже — **сводный перечень** с ранжированием по критичности (как согласовано в сессии).

### Blocker

| ID | Суть |
|----|------|
| B-целостность | Инварианты **XOR подтипов клиента** (`CLIENT_FL` / `CLIENT_UL`) и согласованность с `CLIENT.type` в диаграмме **не выражены**; одного `CHECK` на таблице `CLIENT` недостаточно — нужны триггеры, отложенная проверка или доменный слой. Без механизма на уровне БД/домена модель не гарантирует целостность подтипов. |
| B-APPEAL | Правило **«не более одного предметного FK»** у `APPEAL` в нотации ER не видно; без `CHECK` (например подсчет ненулевых FK) или эквивалента возможны недопустимые комбинации. |
| B-BOOKING-SECTOR | Согласованность **`BOOKING.sector_id`** с сектором выбранного **`PARKING_PLACE`** не обеспечивается одними FK; нужен триггер, строгая валидация приложения или пересмотр дублирования сектора. |

## Дополнительно от systems-analyst

- Внутренняя несогласованность в **концептуальной** модели по кардинальности **Бронирование → Платеж** (разные формулировки в разных местах) — не ошибка ER-файла, но источник путаницы при трассировке.
- Инварианты из текста ER **не закодированы** в Mermaid (ожидаемо): нужны явные ограничения в DDL/домене.
- Открытые вопросы (`Unclear:`), перенесенные из субагента:
  - нужны ли в БД явные связи уведомлений с бронированием/договором или достаточно внешнего сервиса/журнала;
  - куда переносится льготность тарифа без поля на `TARIFF`;
  - обязателен ли шаблон договора в целевой схеме (NULL vs обязательный FK);
  - нужна ли связь **обращение → уведомления** в целевой БД.

## Дополнительно от ревью БД (generalPurpose)

- Составные PK для `ZONE_TYPE_VEHICLE_TYPE` и `ZONE_TYPE_TARIFF` без лишнего суррогата — уместно, если нет ссылок на строку связи как на сущность.
- Для `APPEAL` в PostgreSQL: вариант `CHECK` с `num_nonnulls` по предметным столбцам (в поддерживаемых версиях) или явная сумма `(col IS NOT NULL)::int`.
- Для `BOOKING`/`PARKING_PLACE`: `CHECK` с подзапросом к другой таблице в PostgreSQL **недопустим** — триггер или приложение.
- Уточнение: при необходимости нескольких «предметов» в жизненном цикле обращения — возможна отдельная таблица предмета обращения.
