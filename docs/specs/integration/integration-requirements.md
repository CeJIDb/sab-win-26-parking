# Требования к интеграции

Документ фиксирует требования класса `INT-*` для обмена с внешними и смежными системами, а также сохраняет рабочий маппинг данных для онлайн-оплаты.
Он служит каноничной точкой входа для интеграционных требований в разделе `docs/specs/`.

## Пользовательские требования к интеграции

### `INT-001` `АдаптерСКУД.Событие.ПроверкаДоступа`

Модуль Проверка Доступа должен принимать от модуля Адаптер СКУД СобытиеСКУД со следующими атрибутами:

- типСобытия
- ГРЗ
- датаВремяСобытия
- кппИд
- полосаИд

### `INT-003` `АдаптерСКУД.СобытиеСКУД.ПроверкаДоступа`

Модуль ПроверкаДоступа должен получать от модуля АдаптерСКУД объект СобытиеСКУД (нормализованное представление события смежной СКУД) со следующими атрибутами:

- типСобытия
- ГРЗ
- датаВремяСобытия
- кппИд
- полосаИд

При временной недоступности API основного процесса АдаптерСКУД должен обеспечивать отложенную доставку тех же атрибутов после восстановления связи, сохраняя порядок событий, достаточный для корректной обработки въезда/выезда.

### `INT-002` `ПроверкаДоступа.ПравоДоступа.АдаптерСКУД`

Модуль ПроверкаДоступа должен передавать модулю АдаптерСКУД значение результата проверки права доступа (допуск на КПП) со следующими атрибутами:

- кппИд
- полосаИд
- решениеДопуска: разрешено или запрещено
- корреляцияЗапроса (идентификатор, связывающий ответ с принятым событием, если используется)

### `INT-002` `АдаптерСКУД.КомандаШлагбаума.СКУД`

Модуль АдаптерСКУД должен передавать в смежную систему СКУД команду управления шлагбаумом со следующими атрибутами:

- кппИд
- полосаИд
- команда = «открыть шлагбаум»

### `INT-004` `Платеж.ЗапросОплаты.ПлатежныйПровайдер`

Модуль Платеж должен передавать в ПлатежныйПровайдер запрос на проведение онлайн-оплаты (карты, СБП — по возможностям провайдера) со следующими атрибутами:

- сумма
- валюта
- способОплаты
- внутреннийИдПлатежа
- назначениеПлатежа
- срокОплаты (при необходимости)
- реквизиты, необходимые провайдеру для идемпотентности и сверки (в т.ч. значение, сопоставимое с идПровайдера после ответа)

### `INT-004` `ПлатежныйПровайдер.СтатусОплаты.Платеж`

Модуль Платеж должен получать от ПлатежныйПровайдер данные о результате и статусе операции оплаты со следующими атрибутами (минимум):

- идентификатор операции у провайдера (сопоставимый с идПровайдера)
- статусОперации (достаточный для перевода записи Платеж в одно из значений фиксированного набора статусов)
- времяЗавершения у провайдера (если применимо)

### `INT-004` `Платеж.ЗапросОтменыОплаты.ПлатежныйПровайдер`

Модуль Платеж должен передавать в ПлатежныйПровайдер запрос отмены незавершённой операции с атрибутами, достаточными для идентификации операции у провайдера (в т.ч. идПровайдера), когда Платеж находится в промежуточном статусе.

### `INT-004` `Платеж.ЗапросВозврата.ПлатежныйПровайдер`

Модуль Платеж должен передавать в ПлатежныйПровайдер запрос возврата по успешно проведённой операции с атрибутами: сумма возврата, валюта, ссылка на исходную операцию у провайдера, внутренний идентификатор платежа-возврата.

### `INT-004` `Платеж.ЗапросАктуальногоСтатуса.ПлатежныйПровайдер`

Модуль Платеж должен по расписанию или по событию запрашивать у ПлатежныйПровайдер актуальный статус операции по идПровайдера и получать ответ с атрибутами, достаточными для обновления статус и времяЗавершения записи Платеж.

### `INT-005` `Платеж.ЗапросОплаты.ПлатежныйТерминал`

Модуль Платеж должен передавать на ПлатежныйТерминал объекта (КПП, выезд) запрос на приём оплаты со следующими атрибутами:

- сумма
- валюта
- внутреннийИдПлатежа
- назначениеПлатежа
- идентификатор терминала или точки оплаты (если требуется провайдером терминалов)

### `INT-005` `ПлатежныйТерминал.РезультатОплаты.Платеж`

Модуль Платеж должен получать от ПлатежныйТерминал результат операции со следующими атрибутами (минимум):

- идентификатор операции у терминала/эквайера
- статусОперации
- времяЗавершения (если применимо)

### `INT-006` `Платеж.ЗапросФискализации.ОФД`

Модуль Платеж должен передавать в ОФД запрос на регистрацию фискального чека с реквизитами, достаточными по законодательству и политике выбранного ОФД/ФН, включая:

- идПлатежа и сумму, согласованную с успешным платежом
- состав позиций и налоговые признаки в формате провайдера
- идентификатор чека во внутренней системе для корреляции ответа

### `INT-006` `ОФД.РезультатФискализации.Платеж`

Модуль Платеж должен получать от ОФД результат фискализации со следующими атрибутами (минимум):

- номерФискальный (при успехе)
- статусФискализации (значение из фиксированного набора)
- датаВремяЧека (при необходимости уточнения)
- сведения об ошибке (при неуспехе), достаточные для повторной отправки и учёта

### `INT-007` `ОФД.ДанныеЧекаДляПредставления.Платеж`

Модуль Платеж (через связанную запись Чек) должен располагать после фискализации набором атрибутов, достаточным для отображения клиенту и выдачи по сценарию получения чека, на основании данных, полученных от ОФД, в том числе:

- номерФискальный
- датаВремяЧека
- статусФискализации
- сумма
- идПлатежа

### `INT-008` `Аутентификация.ЗапросАвторизации.IdP`

Модуль Аутентификация должен передавать во внешний IdP (Яндекс ID, VK ID) параметры потока OAuth2/OIDC для регистрации/входа клиента ФЛ, включая идентификатор клиента, перенаправления, запрашиваемые области (scopes), защиту от CSRF (state, nonce) - в соответствии с профилем провайдера.

### `INT-008` `IdP.КодАвторизации.Аутентификация`

Модуль Аутентификация должен получать от IdP ответ авторизации с кодом (или иным элементом потока), достаточным для обмена на токены.

### `INT-008` `Аутентификация.ЗапросТокенов.IdP`

Модуль Аутентификация должен передавать в IdP запрос на обмен кода авторизации на токены (grant_type, code, redirect_uri и иные обязательные поля провайдера).

### `INT-008` `IdP.НаборТокенов.Аутентификация`

Модуль Аутентификация должен получать от IdP набор токенов (в т.ч. access_token, id_token по профилю OIDC), достаточный для установления сеанса пользователя в платформе.

### `INT-009` `IdP.АтрибутыПрофиля.Аутентификация`

Модуль Аутентификация должен получать от IdP минимально необходимый набор атрибутов профиля (утверждения claims или UserInfo), достаточный для идентификации пользователя и ведения личного кабинета в соответствии с политикой согласий и ПДн (например: стабильный идентификатор субъекта, контакт для уведомлений - по согласованному с заказчиком минимуму).

### `INT-010` `АгентУведомлений.ЗапросДоставки.СервисУведомлений`

Модуль АгентУведомлений должен передавать во внешний СервисУведомлений запрос на доставку сообщения со следующими атрибутами:

- канал (SMS или e-mail)
- адресПолучателя (номер или e-mail)
- тема (для e-mail, при необходимости)
- текст или идентификаторШаблона
- типСобытияИсточник (для журналирования и поддержки)
- ключИдемпотентности

### `INT-011` `АгентУведомлений.ПолитикаПовтора.СервисУведомлений`

Модуль АгентУведомлений при сбоях доставки должен обеспечивать повторные попытки отправки с тем же ключИдемпотентности, чтобы внешний СервисУведомлений или логика платформы предотвращали дублирование критичных сообщений клиенту при семантике at-least-once.

### `INT-012` `Договор.ИсходящийДокументЭДО.ЭДО`

Модуль Договор должен передавать в ЭДО пакет по договорной документации юридического лица со следующими атрибутами (минимум):

- типДокумента / вид обмена
- содержимое или безопасная ссылка на документ
- внутреннийИдДоговора
- реквизиты контрагента, необходимые ЭДО

### `INT-012` `ЭДО.СтатусДокумента.Договор`

Модуль Договор должен получать от ЭДО сведения о статусе документооборота (подписание, отклонение, архивное хранение) с атрибутами, достаточными для обновления состояния договора в платформе и отображения в ЛК клиента ЮЛ.

### `INT-013` `Площадка.ДанныеИндикации.ИнформационноеТабло`

Модуль Площадка должен передавать на ИнформационноеТабло (дисплей въезда, выезда, навигации) объект ДанныеИндикации со следующими атрибутами (набор уточняется конфигурацией площадки):

- идУстройстваТабло
- сектор / зона (при необходимости)
- показательЗанятости или направление
- оперативныйТекст или кодСообщения для отображения
