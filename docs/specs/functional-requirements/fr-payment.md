## КлиентФЛ

- Платеж.Создать

  Система должна позволять пользователю с ролью КлиентФЛ инициировать создание объекта Платеж для выбранного Бронирования или ПарковочнойСессии, задав следующие атрибуты:

  - `бронированиеИд` или `идСессии`
  - валюта
  - способОплаты
  - `идПользователя`
  - `идСотрудника` (если платеж создаёт сотрудник)
  - назначениеПлатежа

  Автоматическое заполнение служебных атрибутов определяется отдельными требованиями `Платеж.Создать.СД`.

- Платеж.ПросмотретьСписок

  Система должна предоставлять пользователю с ролью КлиентФЛ возможность просматривать список своих платежей с атрибутами:

  - идентификатор
  - `бронированиеИд`
  - сумма
  - валюта
  - способОплаты
  - статус
  - времяИнициализации
  - времяЗавершения
  - назначениеПлатежа

  и фильтрацией по периоду и статусу.

- Платеж.Просмотреть

  Система должна предоставлять пользователю с ролью КлиентФЛ возможность просматривать детальную карточку Платеж, содержащую:

  - идентификатор
  - `бронированиеИд`
  - `идСессии`
  - сумма
  - валюта
  - способОплаты
  - статус
  - времяИнициализации
  - времяЗавершения
  - `идПровайдера`
  - имяПользователя
  - имяСотрудника
  - срокОплаты
  - назначениеПлатежа
  - список связанных чеков по `идПлатежа` с их `статусФискализации`

- Платеж.ЧастичнаяОплата

  Система должна предоставлять пользователю с ролью КлиентФЛ возможность осуществлять частичную оплату, при этом:

  - для одного `бронированиеИд` или `идСессии` может существовать несколько записей Платеж
  - значение `сумма` каждого платежа > 0 и не превышает остаток к оплате
  - суммарная сумма всех успешных платежей по объекту не превышает исходную сумму

- Платеж.Отменить

  Система должна позволять пользователю с ролью КлиентФЛ отменить Платеж, если он находится в промежуточном статусе:

  - ИНИЦИАЛИЗИРОВАН
  - ОЖИДАНИЕПРОВАЙДЕРА

  при этом:

  - атрибут `статус` выбранного платежа изменяется на значение, соответствующее отмене
  - при наличии незавершенного взаимодействия с платежной системой формируется запрос на отмену операции
  - дальнейшее изменение статуса этого Платежа и создание по нему новых чеков становится недоступными

## Системы

- Платеж.Создать.СД

  Система должна создавать записи Платеж с заполнением атрибутов:

  - `бронированиеИд`
  - `идСессии`
  - сумма
  - валюта
  - способОплаты
  - начальный статус
  - `идПользователя`
  - `идСотрудника`
  - срокОплаты
  - назначениеПлатежа

- Платеж.Создать.СД.сумма

  Система должна при создании Платеж для оплаты бронирования или парковочной сессии автоматически:

  - определять применимый Тариф по атрибутам:
    - типТарифа
    - льготнаяКатегория
    - действуетС
    - действуетПо
  - рассчитывать значение атрибута `сумма` с использованием:
    - ставка
    - шагТарификации
    - максимальнаяСумма
    - грейсПериод
  - обеспечивать, чтобы `сумма` >= 0 и имела 2 знака после запятой
  - ограничивать `сумма` значением `максимальнаяСумма`

- Платеж.Изменить.СД.статус

  Система должна автоматически:

  - принимать ответы платёжного провайдера по реквизитам, включая `идПровайдера`
  - изменять атрибут `статус` записи Платеж на одно из значений фиксированного набора
  - при изменении статуса заполнять `времяЗавершения`

- Платеж.СверитьСтатус.СД

  Система должна по заданному расписанию:

  - выбирать записи Платеж с промежуточными значениями `статус`
  - по атрибуту `идПровайдера` запрашивать у платёжного провайдера актуальный статус операции
  - обновлять `статус` и `времяЗавершения` в соответствии с ответом
  - помечать платёж как требующий ручной обработки при невозможности однозначно определить статус

### Проверка полноты набора ФТ

- Основные потоки UC-10.1-10.9 покрыты:
  - создание платежа
  - просмотр списка платежей
  - просмотр платежа
  - частичная оплата
  - возврат
  - отмена платежа
  - системный расчёт суммы и обновление статуса
- Потенциально отсутствующие ФТ (рекомендации к расширению):
  - ФТ по работе с задолженностью как отдельным представлением
  - ФТ по отображению пользовательских представлений `бронирование` и `сессия` вместо технических идентификаторов
  - ФТ по интеграции с внешними способами оплаты и повторным попыткам списания
