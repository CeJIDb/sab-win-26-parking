## КлиентФЛ

    ### Автор: Команда

        #### Бронирование.Создать

            Система должна позволять пользователю с ролью `Клиент`,`Охранник`, Управляющий, Владелец создать объект Бронирование со следующими атрибутами:

            - ТС

            - Тариф

            - ПМ

            - Сектор

            - Дата время начала

            - Длительность

        #### Бронирование.Редактировать

            Система должна позволять пользователю редактировать Бронирование со следующими атрибутами:

            - ТС

            - Тариф

            - ПМ

            - Сектор

            - Дата время начала

            - Длительность

            

            ---

            #### Бронирование. Редактировать СД

            Система должна позволять редактировать Бронирование cо следующими атрибутами:

            - Идентификатор

            - Договор

            - Тип

            - Статус

            - Сумма к оплате

            

            ### Бронирование. Отменить.СД

            Система должна редактировать [Бронирование]:Статус по следующему алгоритму:

            - **Статус = "Отменено"**

            

            ### Бронирование. Отменить

            Система должна позволять пользователю с ролью `Клиент` или `Охранник` отменить Бронирование.

            

            #### Бронирование. Создать.СД. Идентификатор бронирования

            Система должна присваивать Бронированию уникальный идентификатор по следующему алгоритму:

            - Значение должно быть целочисленным.

            - Значение должно автоматически инкрементироваться для каждого нового Бронирования.

            

        #### Бронирование.Просмотреть

            Система должна позволять пользователю с ролью КлиентФЛ просматривать принадлежащий ему объект Бронирование со следующими атрибутами:

            - ~~идентификатор~~ номерБронирования

            - ~~тсИд~~ [ТС]:госНомер

            - [Сектор]:наименование

            - [ПМ]:номерМеста

            - [Договор]:номерДоговора

            - датаНачала

            - датаОкончания

            - длительность

            - тип

            - статус

            - [Тариф]:наименование

            - суммаКОплате

    ### Автор: Артем

        #### Бронирование.ПросмотретьСписок

            Система должна позволять пользователю с ролью КлиентФЛ просматривать список принадлежащих ему объектов Бронирования с отображением для каждого бронирования следующих атрибутов:

            - идентификатор

            - [ТС]:госНомер

            - [Сектор]:наименование

            - [ПМ]:номерМеста

            - датаНачала

            - датаОкончания

            - длительность

            - тип

            - статус

            - суммаКОплате

        #### Бронирование.Создать.СД

            Система должна создать объект Бронирование при автоматическом въезде ТС на основании принятого СобытиеСКУД со следующими атрибутами:

            - идентификатор

            - тсИд = [ТС]:идентификатор, найденный по [СобытиеСКУД]:ГРЗ

            - секторИд

            - парковочноеМестоИд

            - датаНачала =  [СобытиеСКУД]:датаВремяСобытия

            - тип = «автоматическое»

            - статус = «активно»

            - тарифИд

        #### Бронирование.Создать.СД.идентификатор

            Система должна изменять [Бронирование]:идентификатор по следующему алгоритму:

            - [Бронирование]:идентификатор = max([Бронирование]:идентификатор) + 1.

        #### Бронирование.Создать.СД.секторИд

            Система должна изменять [Бронирование]:секторИд по следующему алгоритму:

            - система определяет по автоматически выбранному Тарифу и типу ТС множество допустимых Типов зон;

            - система формирует множество Секторов, у которых Тип зоны входит в множество допустимых Типов зон и в которых есть хотя бы одно свободное ПМ;

            - если для типа клиента или ТС определены приоритетные зоны, система ограничивает множество Секторов только приоритетными Секторами (если такие есть);

            - из оставшихся Секторов система выбирает Сектор, ближайший к КПП, по которому получено СобытиеСКУД (по [СобытиеСКУД]:кппИд); при равном расстоянии выбирается Сектор с наименьшей загрузкой (минимальной долей занятых ПМ);

            - [Бронирование]:секторИд = идентификатор выбранного Сектора.

        #### Бронирование.Создать.СД.парковочноеМестоИд

            Система должна изменять [Бронирование]:парковочноеМестоИд по следующему алгоритму:

            - система формирует множество ПМ в Секторе с идентификатором [Бронирование]:секторИд, удовлетворяющих условиям:

                - [ПМ].зарезервировано = 0;

            - из множества подходящих ПМ система выбирает одно ПМ по следующей стратегии:

                - выбирается ПМ с минимальным идентификатором внутри Сектора;

            - [Бронирование]:парковочноеМестоИд = идентификатор выбранного ПМ.

        #### Бронирование.Создать.СД.тарифИд

            Система должна изменять [Бронирование]:тарифИд по следующему алгоритму:

            - система определяет тип ТС по объекту ТС, связанному с [Бронирование]:тсИд;

            - система формирует множество краткосрочных Тарифов, доступных для данного типа ТС;

            - для каждого Тарифа из множества система определяет, какие Типы зон ему соответствуют, и какие Секторы парковки относятся к этим Типам зон;

            - из множества краткосрочных Тарифов система исключает Тарифы, для которых на текущей парковке нет ни одного Сектора с подходящим Типом зоны и хотя бы одним свободным ПМ;

            - если после исключения осталось несколько Тарифов, система выбирает среди них Тариф с минимальной базовой стоимостью (по настройкам тарифа); 

            - при равной базовой стоимости выбирается Тариф с наименьшей продолжительностью минимального расчетного интервала;;

            - [Бронирование]:тарифИд = идентификатор выбранного краткосрочного Тарифа.

        #### Бронирование.Изменить.СД

            Система должна изменять Бронирование со следующими атрибутами:

            - длительность

            - суммаКОплате

        #### Бронирование.Изменить.СД.длительность

            Система должна изменять [Бронирование]:длительность по следующему алгоритму:

            - [Бронирование]:длительность = округленная до минут разница между текущим системным временем и [Бронирование]:датаНачала.

        #### Бронирование.Изменить.СД.суммаКОплате

            Система должна изменять [Бронирование]:суммаКОплате по следующему алгоритму:

            - сумма рассчитывается по краткосрочному Тарифу с идентификатором [Бронирование]:тарифИд и текущему значению [Бронирование]:длительность (пересчитанному в минимальные тарифные единицы в соответствии с настройками тарифа);

            - результат округляется до минимальной денежной единицы (до копеек) по правилам тарификации и сохраняется в [Бронирование]:суммаКОплате.

        #### Бронирование.Завершить.СД

            Система должна изменить Бронирование при автоматическом завершении на выезде со следующими атрибутами:

            - датаОкончания = [СобытиеСКУД]:датаВремяСобытия

            - длительность

            - статус

        #### Бронирование.Завершить.СД.длительность

            Система должна изменять [Бронирование]:длительность по следующему алгоритму:

            - [Бронирование]:длительность = округлённая до минут разница между [Бронирование]:датаОкончания и [Бронирование]:датаНачала.

        #### Бронирование.Завершить.СД.статус

            - Бронирование.ЗавершитьАвтоВыезд.СД. статус

                Система должна изменять [Бронирование]:статус по следующему алгоритму:

                - если тип Бронирования ∈ {«краткосрочное», «автоматическое»} и существует ПарковочнаяСессия, связанная с [Бронирование], такая, что [ПарковочнаяСессия]:статус = «завершена», то [Бронирование]:статус = «завершено».

            Система должна предоставлять пользователям с ролями Клиент ФЛ возможность осуществлять частичную оплату, при этом: 

            - для одного бронирование или идСессии может существовать несколько записей Платеж,

            - значение сумма каждого платежа > 0 и не превышает остаток к оплате,

            - суммарная сумма всех успешных платежей по объекту не превышает исходную сумму.

        #### Платеж.Отменить

            Система должна позволять пользователю с ролью Клиент ФЛ отменить Платеж, если он находится в промежуточном статусе:

            - ИНИЦИАЛИЗИРОВАН,

            - ОЖИДАНИЕПРОВАЙДЕРА.

            при этом:

            - атрибут статус выбранного платежа изменяется на значение соответствующее отмене (ОТМЕНЕН),

            - при наличии незавершенного взаимодействия с платежной системой формируется запрос на отмену операции,

            - дальнейшее изменение статуса этого Платежа и создание по нему новых чеков становится недоступными.

## Охранник

    ### Автор: Артем

        #### Бронирование.Создать

            Система должна позволять пользователю Охранник создавать объект Бронирование от имени клиента со следующими атрибутами:

            - тсИд

            - секторИд

            - парковочноеМестоИд

            - договорИд

            - датаНачала

            - датаОкончания

            - тарифИд

        #### Бронирование.Создать.СД

            Система должна изменять Бронирование со следующими атрибутами:

            - идентификатор

            - длительность

            - тип

            - статус = «ожидает оплаты»

            - суммаКОплате

        #### Бронирование.Создать.СД.идентификатор

            Система должна изменять [Бронирование]:идентификатор по следующему алгоритму:

            - [Бронирование]:идентификатор = max([Бронирование]:идентификатор) + 1.

        #### Бронирование.Создать.СД.длительность

            Система должна изменять [Бронирование]:длительность по следующему алгоритму:

            - [Бронирование]:длительность = [Бронирование]:датаОкончания - [Бронирование]:датаНачала;

            - пересчитать в минуты, часы, дни, месяцы в зависимости от получившейся длительности.

        #### Бронирование.Создать.СД.тип

            Система должна изменять [Бронирование]:тип по следующему алгоритму:

            - если [Бронирование]:длительность < 24 ч, то [Бронирование]:тип = «краткосрочное»;

            - если [Бронирование]:длительность ≥ 24 ч и существует Договор, связанный с [Бронирование], то [Бронирование]:тип = «долгосрочное».

        #### Бронирование.Создать.СД.суммаКОплате

            Система должна изменять [Бронирование]:суммаКОплате по следующему алгоритму:

            - [Бронирование]:суммаКОплате рассчитывается по [Бронирование]:тарифИд и [Бронирование]:длительность;

            - выбирать записи Платеж с промежуточными значениями статус (например, ИНИЦИАЛИЗИРОВАН, ОЖИДАНИЕПРОВАЙДЕРА),

            - по атрибуту идПровайдера запрашивать у платёжного провайдера актуальный статус операции,

            - обновлять статус и времяЗавершения в соответствии с ответом,

            - помечать платёж как требующий ручной обработки при невозможности однозначно определить статус.

        #### Чек.Сформировать

            Система должна при успешном завершении записи Платеж со статус = "УСПЕШЕН" автоматически:

            - создавать связанную запись Чек,

            - заполнять идПлатежа,

            - устанавливать сумма, равную сумма платежа,

            - устанавливать датаВремяЧека текущим системным значением,

            - задавать статусФискализации = "ОЖИДАНИЕ".

        #### Чек.Фискализировать.СД

            Система должна для записей Чек со статусФискализации = "ОЖИДАНИЕ":

            - формировать запрос к фискальному регистратору/ОФД с необходимыми реквизитами чека,

            - по результатам присваивать атрибуту номерФискальный значение, полученное от внешней системы,

            - при необходимости обновлять датаВремяЧека,

            - изменять статусФискализации на значения из фиксированного набора (ФИСКАЛИЗАЦИЯ, ОШИБКА).

            

        #### Чек.ПовторноОтправить.СД

            Система должна для записей Чек со статусФискализации = "ОШИБКА":

            - обеспечивать возможность раз в сутки повторного формирования и отправки данных чека во внешнюю фискальную систему,

            - по результатам повторной попытки обновлять

                - номерФискальный, 

                - датаВремяЧека, 

                - статусФискализации.
