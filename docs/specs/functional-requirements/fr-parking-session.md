## Функциональные требования: объект Парковочная сессия

Кратко: парковочная сессия — период нахождения ТС на парковке от въезда до выезда, связан с одним `Бронирование`; ТС, тариф, договор и место задаются через бронирование ([концептуальная модель](../../artifacts/conceptual-model-with-attributes.md)). Оплата и чеки — в контексте `Бронирование`; у сессии — время, КПП, статус.

## Системы

- ПарковочнаяСессия.Создать.СД

  Система должна создать объект Парковочная сессия при разрешенном въезде ТС с активным бронированием, по сообщению СКУД (тип события, ГРЗ, дата-время, КПП, полоса).

- ПарковочнаяСессия.Создать.СД.идентификатор

  Система должна изменять `[ПарковочнаяСессия]:идентификатор` по следующему алгоритму:

  - сгенерировать уникальный идентификатор записи в формате, согласованном с [концептуальной моделью](../../artifacts/conceptual-model-with-attributes.md) (например UUID).

- ПарковочнаяСессия.Создать.СД.бронированиеИд

  Система должна изменять `[ПарковочнаяСессия]:бронированиеИд` по следующему алгоритму:

  - установить ссылку на активное бронирование ТС по допуску на въезд (в том числе созданное автоматически при въезде);
  - если бронирование не найдено — не создавать запись ПС, зафиксировать ошибку или событие для расследования.

- ПарковочнаяСессия.Создать.СД.времяВъезда

  Система должна изменять `[ПарковочнаяСессия]:времяВъезда` по следующему алгоритму:

  - установить из даты-времени события въезда в сообщении СКУД; часовой пояс — по политике хранения.

- ПарковочнаяСессия.Создать.СД.кппВъездаИд

  Система должна изменять `[ПарковочнаяСессия]:кппВъездаИд` по следующему алгоритму:

  - установить идентификатор КПП из сообщения СКУД, соответствующий точке въезда.

- ПарковочнаяСессия.Создать.СД.способДопуска

  Система должна изменять `[ПарковочнаяСессия]:способДопуска` по следующему алгоритму:

  - установить «автоматически».

- ПарковочнаяСессия.Создать.СД.статус

  Система должна изменять `[ПарковочнаяСессия]:статус` по следующему алгоритму:

  - установить «активна».

- ПарковочнаяСессия.Создать.СД.инвариантАктивнаяПС

  Система должна обеспечивать инвариант по следующему алгоритму:

  - для одной пары «ТС — парковка» (через бронирование и принадлежность ТС парковке) одновременно не более одной парковочной сессии в статусе «активна»;
  - при нарушении — не создавать новую активную запись, зафиксировать событие для расследования.

- ПарковочнаяСессия.Завершить.СД

  Система должна завершить активную парковочную сессию ТС при выезде, по сообщению СКУД, достаточному для идентификации выезда.

- ПарковочнаяСессия.Завершить.СД.завершениеСвязанногоБронирования

  Система должна инициировать завершение связанного бронирования по следующему алгоритму (после фиксации завершения парковочной сессии):

  - если `[Бронирование]:тип` не равен «долгосрочное» — по правилам [ФТ по бронированию](fr-booking.md);
  - иначе — не инициировать.

- ПарковочнаяСессия.Завершить.СД.времяВыезда

  Система должна изменять `[ПарковочнаяСессия]:времяВыезда` по следующему алгоритму:

  - установить из даты-времени события выезда в сообщении СКУД для активной ПС данного ТС.

- ПарковочнаяСессия.Завершить.СД.кппВыездаИд

  Система должна изменять `[ПарковочнаяСессия]:кппВыездаИд` по следующему алгоритму:

  - установить идентификатор КПП из сообщения СКУД, соответствующий точке выезда.

- ПарковочнаяСессия.Завершить.СД.статус

  Система должна изменять `[ПарковочнаяСессия]:статус` по следующему алгоритму:

  - установить «завершена» после установки `времяВыезда`.

- ПарковочнаяСессия.Завершить.СД.длительность

  Система должна изменять `[ПарковочнаяСессия]:длительность` по следующему алгоритму:

  - вычислить разность между `времяВыезда` и `времяВъезда`, округлить до целых минут по правилам тарификации;
  - если `времяВыезда` < `времяВъезда` — не изменять `длительность`, зафиксировать ошибку в журнале аудита.

### Проверка полноты набора ФТ

- Покрыты сценарии из **«Входные UC»**: создание и завершение сессии (в т.ч. автоматически), просмотр активной и списков, ручное создание охранником.
- Для последующего уточнения:
  - экран «журнал въезда-выезда» и граница с ФТ по событиям СКУД;
  - уведомления при смене статуса сессии;
  - обновление `фактическиЗанято` у ПМ и счетчиков мест, если не входит в интеграционные ФТ.

## Чеклист качества перед публикацией

- **Тестируемость:** для каждого FR ясны вход (роль при UI, либо событие/сообщение при `.СД`) и ожидаемое состояние `[ПарковочнаяСессия]:*` и при необходимости связанного `[Бронирование]:*`.
- **Модель данных:** атрибуты не противоречат [концептуальной модели](../../artifacts/conceptual-model-with-attributes.md); расширение списка полей на сессии без обновления модели — только через изменение модели или явный технический долг.
- **Трассируемость:** при изменении набора FR обновить `docs/process/traceability-matrix-log.md` и прогнать проверки репозитория.
- **Оглавление и якоря:** после правок заголовков обновить оглавление; проверить ссылки: `npm run lint:md-links` или `node ./scripts/check-markdown-links.mjs`.
