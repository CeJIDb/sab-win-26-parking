## Ход встречи

### Валидация BPMN

- Диаграмма описывает процесс «Предоставление парковочного места» с точки зрения КПП; три роли: Система шлагбаума, Клиент, Охранник.
- Внесённые изменения: при выдаче проксимити-карты — инициализация записи в базу договоров; уточнено, что в журналы въезда-выезда вносят охранники вручную; добавлена фиксация факта въезда ТС в журнал.
- Заказчик подтвердил: схема соответствует действительности.

**Заключение договора с ЮЛ:** передача подписанного экземпляра — по договорённости (водитель, представитель, почта, курьер). Доступ предоставляется после подтверждения договорённости, как правило после подписания договора клиентом; оплата может быть раньше или позже. Внесение данных — как только управляющий получает возможность; при делегациях номера ТС могут появиться в последний момент. Пожелание: гибкий инструмент для самостоятельного управления списком ТС корпоративными клиентами.

**Идентификация и доступ:** при неработающей карте — ручная проверка по Excel; при выезде краткосрочных — проверка факта оплаты, при отсутствии задолженности — шлагбаум вручную.

### Opportunity Canvas

**Пользователи:** Клиенты — ФЛ (краткосрочная), ФЛ (с абонементами), ЮЛ (в т.ч. спецтехника, делегации/временные списки ТС). Пользователи системы — охранники, старшие охранники, бухгалтер, владелец, управляющий. Учтены различия по возрасту и технической грамотности.

**Проблемы клиентов (ФЛ):** очереди; отсутствие единого онлайн-источника (тарифы, условия, доступность); долгое оформление договоров; отсутствие возможности заранее забронировать место; отсутствие уведомлений (окончание договора, оплата, ремонты).

**Проблемы клиентов (ЮЛ):** очереди; долгое заключение договоров; при отсутствии номера ТС в базе — согласование только через управляющего; отсутствие возможности самостоятельно управлять списком ТС.

**Общие:** отсутствие удалённого выбора, бронирования и оплаты; отсутствие информации о парковке на геосервисах с переходом к бронированию.

**Проблемы охранников:** высокая доля ручных операций; отвлечение от безопасности; отсутствие актуального учёта занятости мест; ручная проверка/замена карт; ручное внесение въезда/выезда; консультирование по тарифам; стресс из-за очередей и конфликтов; переработки, нерегулярный график. Старшие охранники: оплата и оформление договоров; при краткосрочной оплату может принять обычный охранник; активация/блокировка карт — любым охранником. Черный список — вручную, в базе отметка и комментарии.

**Проблемы управляющего:** отсутствие статистики и аналитики; высокая нагрузка из-за ручного учёта; приём наличных от старшего охранника и сдача бухгалтеру; внесение данных по договорам ЮЛ и спискам ТС; большая завязка по корпоративным клиентам; необходимость быть на связи 24/7; управляющий — узкое горло процесса.

**Проблемы владельца:** отсутствие отчётности и аналитики; непонятно по сегментам; ручная обработка в Excel трудозатратна; отсутствие гибкой настройки тарифов; плохое информирование клиентов; простой ~30% мест; невозможность масштабировать без изменений.

**Топ-3 проблемы бизнеса:** 1) Очереди на въезде; 2) Ручная рутинная работа охранников (~80% времени); 3) Отсутствие аналитики. Дополнительно: простой мест (~30%, цель после внедрения — не более 10%); отсутствие аналитики для стратегии и оперативного управления; операционные риски; недостаточная формализация правил.

**Метрики:** 80% операций — онлайн (предварительная целевая); простой — не больше 10% после внедрения; методику расчёта простоя команда предложит.

### Варианты решений (идеи)

- **Автоматизация въезда-выезда:** распознавание номеров, QR-коды, метки, талончики, мобильные приложения. Пожелание: отсутствие очередей, быстрый проезд без охранника; запасной вариант при отказе автоматики; дублирование и проработка отказов.
- **Учёт и оплата:** паркоматы. Пожелание: уход в онлайн; учесть пенсионеров (терминал на КПП); паркоматы могут быть препятствием для пенсионеров.
- **Управление:** CRM; система управления парковкой (визуализация в реальном времени, аналитика, схема занятости). CRM — уточнить место в концепции; отчётность выходит из системы управления парковкой.
- **Взаимодействие с клиентом:** онлайн-бронирование, онлайн-оплата, онлайн-просмотр абонементов.
- **Отчётность:** аналитика по загрузке, движению клиентов, визуализация неудобных зон.

### Технические и процедурные уточнения

- **Учётные таблицы:** файл в Google Таблицах, синхронизация через них; графы «кто открыл шлагбаум» в базе нет; при поломке ПК/нет доступа — возможна временная остановка (только выпуск), бэкапы в конце месяца на флешку, при офлайн — запись на бумаге/локально в Excel, затем перенос в файл.
- **КПП:** два отдельных места (въезд и выезд раздельно), оба КПП работают и на въезд, и на выезд.
- **Проксимити-карты:** у каждой карты свой номер; перепрограммирование до 10 минут; автоматической блокировки по дате нет (только активна/заблокирована).

**Работа с клиентами:** с ушедшими клиентами системно не работают; хотелось бы SMS/email рассылки, акции; гибкое управление тарифами по группам, скидки, промо-коды.

### Замечания по Event Storming

- Оплата ЮЛ не выделена как отдельное событие (происходит в разный момент).
- Логическая ошибка: ветка «если номера нет в базе» должна выходить из процесса охранника, а не из «номер найден».
- Упрощение схемы; формулировка «действительность договора» → «статус/срок договора проверен»; детали выезда — зафиксировать, что проверка при выезде идентична въезду.
