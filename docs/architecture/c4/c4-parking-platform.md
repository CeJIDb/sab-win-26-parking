## Level 1 — System Context

Показывает платформу как «чёрный ящик», акторов и внешние системы.

### Текстовое описание (Level 1)

**Система:** Цифровая платформа парковки (SAB) — веб-приложение для управления парковкой на 600 мест: бронирование, контроль доступа на КПП, тарификация, оплата, договоры, уведомления, аналитика.

**Акторы (люди):**

| Актор | Взаимодействие с платформой |
| --- | --- |
| Клиент ФЛ | PWA/ЛК: бронирование, оплата, история сессий, профиль, уведомления |
| Клиент ЮЛ (представитель) | ЛК: договоры, квоты, бронирование, счета, профиль организации |
| Охранник КПП | Служебный веб-интерфейс: ручной допуск, журнал, спорные ситуации |
| Управляющий | Служебный веб-интерфейс: тарифы, секторы, задолженности, обращения, настройки |
| Владелец | Служебный веб-интерфейс: аналитика, отчёты, бизнес-решения |

**Внешние системы:**

| Система | Протокол | Назначение |
| --- | --- | --- |
| СКУД + LPR-видеосистема | UDP / REST | Распознавание ГРЗ, события въезда/выезда; исполнение allow/deny (шлагбаум) |
| Платёжный провайдер (онлайн) | REST API | Приём онлайн-оплаты (карты, СБП) |
| Платёжные терминалы объекта | REST API | Терминалы оплаты на КПП/выезде; на L1 — один внешний узел |
| ОФД | REST API | Фискализация чеков |
| SSO/IdP (Яндекс ID, VK ID) | OAuth2/OIDC | Аутентификация клиентов ФЛ |
| Сервис уведомлений | REST API | Доставка SMS и e-mail |
| ЭДО | REST API | Электронный документооборот (договоры ЮЛ) |
| Информационные табло и дисплеи | REST API | Несколько устройств (въезд/выезд, навигация и т.д.); на L1 — один внешний узел, см. [C4 Level 1: System Context](c4-l1-system-context.md) |

## Level 2 — Container

Показывает развёртываемые единицы внутри платформы: основной процесс, изолированные адаптеры, БД и фронтенды.

> Отдельные процессы выделяются только при несовместимости транспорта, UDP у СКУД/LPR, или при необходимости изолировать блокирующую асинхронную операцию, пакетную рассылку уведомлений. `Адаптер дисплеев и табло` уже выделен как supporting-контекст, но пока исполняется внутри основного процесса; самостоятельным развёртываемым артефактом он станет только при появлении несовместимых протоколов или отдельных требований к изоляции.

### Текстовое описание (Level 2)

| Контейнер | Технология | Назначение |
| --- | --- | --- |
| **Web-приложение (PWA/ЛК)** | SPA / PWA | Клиентский интерфейс: бронирование, оплата, история, профиль |
| **Служебный веб-интерфейс** | SPA | Единый внутренний интерфейс для охранника, управляющего и владельца; доступ к разделам определяется ролями |
| **Основной процесс (API)** | Модульный монолит | Единый развёртываемый артефакт: 14 внутренних контекстов + Сервис приложения + инфраструктура аутентификации. Stateless. REST API. |
| **Адаптер СКУД** | Отдельный процесс | Принимает UDP от LPR-видеосистемы и СКУД, нормализует события и переводит их в REST-вызовы к основному процессу. Буферизация при недоступности. Anti-Corruption Layer. |
| **Агент доставки уведомлений** | Отдельный процесс | Потребляет очередь уведомлений (Outbox) → отправка через внешние шлюзы. At-least-once, idempotency key. |
| **БД PostgreSQL** | PostgreSQL | Единая БД с изоляцией по схемам (schema-per-module). Outbox-таблица для Агента доставки уведомлений. |

## Level 3 — Component (основной процесс)

Показывает доменные модули (bounded contexts) внутри основного процесса и их ключевые зависимости.

### Текстовое описание (Level 3)

Основной процесс содержит 14 внутренних контекстов, координируемых через Сервис приложения. Модули взаимодействуют только через публичные интерфейсы, прямой доступ к таблицам другого модуля запрещен. `Доступ` остается одним модулем, но внутри разделяется на `ПолитикаДопускаНаВъезд` и `ПолитикаДопускаНаВыезд`. Supporting-контекст `Адаптер СКУД` развернут отдельно из-за особенностей транспорта и оборудования.

**Core-домены** (уникальная бизнес-логика):

| Модуль | Ответственность |
| --- | --- |
| **Доступ** | Решение разрешить или запретить на КПП; черный список; аудит решений. Для въезда проверяет бронь и при необходимости запрашивает авто-бронирование через интерфейс `Бронирование`; для выезда проверяет активную сессию и статус оплаты. |
| **Бронирование** | Планирование и резервирование ресурса; автоматическое бронирование по запросу `Доступ`. |
| **Сессия** | Фактическое использование парковки; журнал въезда/выезда; корректировки охранника. Всегда привязана к Бронирование (ADR-002). |
| **Тариф** | Тарифные планы, версии тарифов и правила применимости. Не считает сумму сам, а поставляет правила в `Расчет`. |
| **Расчет** | Preview суммы, расчет стоимости и расчетный снимок для передачи в `Платеж`. Использует правила из `Тариф`, данные `Сессии`, `Бронирования`, `Договора` и `Инфраструктуры парковки`. |

**Supporting-домены:**

| Модуль | Ответственность |
| --- | --- |
| **Платеж** | Платежи, возвраты, задолженности, чеки, счета ЮЛ; фиксация зафиксированной суммы в момент оплаты. Владеет ACL-адаптерами для трех внешних систем: **платежный провайдер** (онлайн-эквайринг), **платежные терминалы объекта** (КПП/выезд) и **ОФД** (фискализация чеков). |
| **Договор** | Договоры, шаблоны, квоты, абонементные правила. Владеет ACL-адаптером для **ЭДО** (подписание и хранение договорных документов с ЮЛ). |
| **Клиент** | Мастер-данные: клиент, организация, ТС (ГРЗ), ПДн, согласия. |
| **Инфраструктура парковки** | Парковка, секторы, ПМ, КПП, конфигурация и статусы. Собирает операционную модель доступности мест и направлений. |
| **Обращение** | Обращения, тикеты, история переписки. |
| **Сотрудник** | Сотрудники, роли (RBAC), служебные профили. |
| **Адаптер дисплеев и табло** | Транслирует подготовленную операционную модель из `Инфраструктуры парковки` во внешние устройства отображения, изолируя vendor-specific протоколы. |

**Generic-домены:**

| Модуль | Ответственность |
| --- | --- |
| **Уведомление** | Уведомления, шаблоны, очередь на доставку. |
| **Отчет** | Read-модели (CQRS), аналитические агрегаты, отчеты. |

**Supporting-контекст вне основного процесса:**

| Модуль | Ответственность |
| --- | --- |
| **Адаптер СКУД** | Нормализует события СКУД/LPR, принимает vendor-specific транспорт, переводит `allow/deny` и изолирует основной процесс от деталей оборудования. |

**Сквозные компоненты:**

| Компонент | Ответственность |
| --- | --- |
| **Сервис приложения** | Оркестрация сценариев, пересекающих несколько модулей (въезд, выезд, оплата). |
| **Инфраструктура аутентификации** | SSO, пароли, TOTP 2FA, JWT — инфраструктурный слой, не доменный модуль. |

## Ключевые сценарии

### Въезд без предварительной брони

```
Камера LPR ──UDP──▶ Адаптер СКУД ──REST──▶ Сервис приложения
                                                       │
                                           (чтение вне транзакции):
                                           Бронирование.получитьСтатус()
                                           Договор.получитьОснование()
                                           Платеж.получитьСтатусДолга()
                                                       │
                                                       ▼
                                        ┌──────── ACID-транзакция ────────┐
                                        │  Доступ.проверитьВъезд()        │
                                        │    └─ Бронирование.создатьАвтоБронь│
                                        │       (если брони нет)          │
                                        │  Сессия.открыть(бронированиеИд) │
                                        └─────────────────────────────────┘
                                                       │
                                                       ▼
                              Сервис приложения ──REST──▶ СКУД → шлагбаум ▲
```

### Расчет суммы, оплата и выезд

```
PWA/ЛК клиента или выездной КПП ──REST──▶ Сервис приложения
                                                       │
                                                       ▼
                                              Расчет.рассчитатьТекущуюСумму()
                                   [использует Тариф, Договор, Сессию, Инфраструктуру]
                                                       │
                    ┌──────────────────────────────────┴──────────────────────────────────┐
                    ▼                                                                     ▼
      Клиент нажал «Оплатить» в ЛК                                          Автомобиль подъехал к выезду
                    │                                                                     │
                    └──────────────────────────────────┬──────────────────────────────────┘
                                                       ▼
                           Платеж.создать(фиксированнаяСумма, источник=ЛК_КЛИЕНТА | ВЫЕЗДНЫЕ_ВОРОТА)
                                                       │
                                              ┌────────┴────────┐
                                              ▼                 ▼
                                    Платежный провайдер   Платежный терминал объекта
                                              │                 │
                                              └────────┬────────┘
                                                       ▼
                                        ┌──────── ACID-транзакция ────────┐
                                        │  Доступ.проверитьВыезд()        │
                                        │  Бронирование.завершить()       │
                                        │  Сессия.завершить()             │
                                        │  Инфраструктура парковки        │
                                        │    .обновитьОперационную        │
                                        │    Проекцию()                   │
                                        │  ИсходящиеСообщения.сохранить() │
                                        └─────────────────────────────────┘
                                                       │
                       ┌────────────┼──────────────────────┬────────────────────┐
                       ▼            ▼                      ▼                    ▼
              Уведомление      Отчет.обновить         Адаптер дисплеев    Агент доставки
              .добавитьВОчередь(чек)  ReadМодели()     и табло.опубликоватьСтатус()  уведомлений → SMS/email/push
```
