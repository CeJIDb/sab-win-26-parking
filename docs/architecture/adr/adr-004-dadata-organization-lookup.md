## Context

При регистрации клиента-ЮЛ (юридического лица) оператор или представитель компании вводит реквизиты организации: наименование, ОПФ, юридический адрес, ИНН, КПП, ОГРН, контактные данные. Ручной ввод создаёт риски:

- опечатки и ошибки в реквизитах (критично для договоров и фискальных документов);
- дублирование организаций с разными написаниями одного и того же ИНН;
- устаревшие данные (изменение юридического адреса, смена ОПФ, ликвидация).

Схема БД требует `ORGANIZATION.inn NOT NULL UNIQUE` (ERD-059). ИНН является ключом поиска для получения актуальных реквизитов из внешнего реестра.

Ключевые параметры контекста:

- **Объём**: регистрация ЮЛ — нечастая операция (десятки в год для одной парковки).
- **Критичность**: реквизиты попадают в договоры и счета; ошибки дорого исправлять.
- **Запрет**: платные зарубежные компоненты запрещены (CONSTR-LIC-001); российские SaaS-сервисы допустимы.

## Options

### Вариант А — DADATA (dadata.ru)

Российский SaaS для стандартизации и обогащения данных; агрегирует ЕГРЮЛ/ЕГРИП, ФНС, Банк России.

- API: `POST https://suggestions.dadata.ru/suggestions/api/4_1/rs/findById/party` по ИНН → полные реквизиты в JSON.
- Покрытие: 99%+ юрлиц и ИП в РФ.
- Лицензия: SaaS; бесплатный тариф до 500 запросов/день (достаточно для одной парковки).
- Latency: < 200 ms (CDN в РФ).
- Sandbox: доступен для разработки и тестов.
- Данные ЕГРЮЛ обновляются ежедневно.

### Вариант Б — Прямой ЕГРЮЛ (ФНС Open API / egrul.nalog.ru)

Бесплатный, но нестабильный, без SLA, без документированного REST API для поиска по ИНН. Требует разбора XML-выгрузок или скрейпинга — неприемлемо для продакшена.

### Вариант В — Только ручной ввод

Нет зависимости от внешних сервисов; риски ошибок остаются. Достаточен для MVP/учебного проекта, но не решает проблему качества данных.

## Decision

**Вариант А — DADATA.** Для полной поставки реализовать `OrganizationLookupService`, который:

1. По введённому ИНН запрашивает DADATA API.
2. Возвращает предзаполненную форму реквизитов организации на фронтенд для подтверждения.
3. Сохраняет реквизиты в `ORGANIZATION` после подтверждения пользователем.
4. При недоступности DADATA переключается на **fallback: ручной ввод**.

## Rationale

- Точность реквизитов в договорах и счетах критична; DADATA устраняет ручные ошибки.
- Объём запросов мал: бесплатный тариф покрывает реальный трафик одной парковки.
- Российский сервис, данные хранятся в РФ — соответствует 152-ФЗ.
- Fallback на ручной ввод устраняет единую точку отказа.
- Не требует изменений схемы БД: `ORGANIZATION.inn` уже `NOT NULL UNIQUE` (ERD-059); поля `name`, `legal_form`, `legal_address`, `kpp`, `ogrn` уже присутствуют в таблице.

## Trade-offs

| Аспект | Плюс | Минус |
|--------|------|-------|
| Качество данных | Актуальные реквизиты из ЕГРЮЛ | Данные обновляются раз в сутки, не realtime |
| Зависимость | Малая; fallback есть | Дополнительный SaaS-контракт |
| Стоимость | Бесплатный тариф достаточен | При масштабировании (>500 запросов/день) нужен платный план |
| Сложность | 1–2 дня реализации | Требует API-ключа и HTTPS-клиента |
| Конфиденциальность | ИНН — открытые данные из ЕГРЮЛ | Логи запросов у DADATA (не PII) |

## Consequences

### Положительные

- Снижение ошибок в реквизитах и дублей организаций.
- Ускорение регистрации ЮЛ: пользователь вводит только ИНН.
- `ORGANIZATION.legal_form` заполняется в формате DADATA — нет необходимости в CHECK-constraint (см. ERD открытый вопрос №8).

### Отрицательные

- Зависимость от внешнего SaaS при регистрации ЮЛ.
- Необходимо хранить API-ключ DADATA в секретах (Vault / env).

### Fallback-поведение

При недоступности DADATA (HTTP 5xx, timeout > 3 s) система отображает форму ручного ввода реквизитов — регистрация ЮЛ не блокируется.

### Revisit criteria

- Тариф >500 запросов/день → переход на платный план или переоценка провайдера.
- Появление стабильного государственного бесплатного API (ЕГРЮЛ Open API) → переоценка Варианта Б.
