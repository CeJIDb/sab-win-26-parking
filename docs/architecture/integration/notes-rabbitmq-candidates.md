# Заметки: кандидаты для RabbitMQ (черновик к ADR-008)

> Временный документ. Создан по итогам проработки Kafka-плана (2026-04-30).
> При оформлении ADR-008 — перенести содержимое туда, этот файл удалить.

## Архитектурный принцип

**Kafka** — события между ограниченными контекстами (bounded contexts).
**RabbitMQ** — рабочая очередь команд внутри одного контекста.

Два уровня не противоречат друг другу: Kafka-план и RabbitMQ-слой живут на разных абстракциях.
Notification-сервис может быть потребителем Kafka и одновременно публикатором команд в RabbitMQ.

```text
Kafka topic (Topic_BookingConfirmed)
  → Notification Consumer (слушает Kafka)
      → RabbitMQ exchange
          → SMS Worker (retry, DLX)
          → Email Worker
          → Push Worker
```

## Кандидаты на RabbitMQ в парковочной системе

### 1. Notification dispatch worker (приоритет: высокий)

- **Что**: очередь команд на отправку уведомлений (SMS, email, push) внутри Notification-контекста.
- **Зачем RabbitMQ**: retry с экспоненциальной задержкой, Dead Letter Exchange (DLX) для неотправленных, конкурентные воркеры без дублирования.
- **Триггер**: событие из Kafka (BookingConfirmed, PaymentCompleted и т.д.) → Notification Consumer → команда в очередь.
- **Связь с текущим планом**: не противоречит, RabbitMQ скрыт внутри Notification bounded context.

### 2. ЭДО / 1С dispatch (приоритет: средний)

- **Что**: буфер повторных попыток при отправке документов во внешние системы (1С, электронный документооборот).
- **Зачем RabbitMQ**: внешние системы нестабильны, нужен retry-буфер с DLX; Kafka здесь избыточен (нет других потребителей).

### 3. Admin command queue (приоритет: низкий)

- **Что**: очередь административных команд (принудительное завершение сессии, блокировка карты).
- **Зачем RabbitMQ**: команды идемпотентны, нужна гарантия exactly-once обработки одним воркером.

### 4. СКУД / барьер CMD-канал (приоритет: средний)

- **Что**: команды управления барьером (открыть/закрыть) через очередь с приоритетами.
- **Зачем RabbitMQ**: приоритетные очереди, low-latency delivery, retry при недоступности контроллера.

### 5. Background job queue (приоритет: низкий)

- **Что**: фоновые задачи (генерация PDF-чека, ночная сверка тарифов).
- **Зачем RabbitMQ**: простая point-to-point очередь без необходимости в Kafka-партиционировании.

## Решения, принятые в рамках Kafka-плана

- Kafka-план (ADR-007) не включает RabbitMQ — тема вынесена в отдельный ADR-008.
- Topic_NotificationStatus не создается (Variant A): Notification пишет результат напрямую в свою БД.
- Pricing остается синхронным RPC — Topic_TariffCalculated не вводится (антипаттерн).
