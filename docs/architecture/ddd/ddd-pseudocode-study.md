## Зачем нужен этот файл

В [DDD Bounded Contexts: учебная версия](ddd-bounded-contexts-study.md) объясняется, **что** такое bounded contexts.

Здесь показано, **как** это выглядит на практике:

- где находится доменная логика;
- где работает `Сервис приложения`;
- где уместны события;
- где лучше оставить синхронный вызов.

Псевдокод специально упрощён. Он показывает структуру, а не конкретный язык программирования.

Имена контекстов (`Бронирование`, `Сессия`, `Доступ`, `Тариф`, `Платёж`) соответствуют таблице из [DDD Bounded Contexts: учебная версия](ddd-bounded-contexts-study.md) и являются частью ubiquitous language проекта на русском языке.

## Доменная логика

### `Тариф.рассчитать()`

`Тариф` получает готовые входные данные и возвращает сумму.

Он не ищет клиента в базе и не вызывает `Платёж`.

```text
Тариф.рассчитать(входные):
    основа = входные.ставка * входные.единицы

    если входные.скидкаПроцент существует:
        итого = основа * (1 - входные.скидкаПроцент / 100)
    иначе:
        итого = основа

    если входные.максимум существует и итого > входные.максимум:
        итого = входные.максимум

    вернуть итого
```

### `Доступ.проверитьВъезд()` и `Доступ.проверитьВыезд()`

`Доступ` остаётся одним модулем, но внутри может содержать две разные проверки: на въезд и на выезд.

```text
Доступ.проверитьВъезд(входные, шлюзБронирования):
    если входные.тсНайдено = ложь:
        вернуть запрет(НЕТ_ТРАНСПОРТНОГО_СРЕДСТВА)

    если входные.вЧёрномСписке = истина:
        вернуть запрет(В_ЧЁРНОМ_СПИСКЕ)

    если входные.естьДолг = истина:
        вернуть запрет(ЗАДОЛЖЕННОСТЬ)

    если входные.естьАктивнаяБронь = истина:
        вернуть разрешение(входные.бронированиеИд, БРОНЬ_НАЙДЕНА)

    если входные.автоБронированиеРазрешено = истина:
        бронированиеИд = шлюзБронирования.создатьАвтоБронь(входные.тсИд, входные.воротаИд)
        если бронированиеИд = пусто:
            вернуть запрет(НЕТ_ДОСТУПНЫХ_МЕСТ)
        вернуть разрешение(бронированиеИд, АВТО_БРОНЬ_СОЗДАНА)

    вернуть запрет(НЕТ_ПРАВИЛА_ДОСТУПА)

Доступ.проверитьВыезд(входные):
    если входные.сессияНайдена = ложь:
        вернуть запрет(НЕТ_АКТИВНОЙ_СЕССИИ)

    если входные.требуетсяОплата = истина и входные.оплатаПроведена = ложь:
        вернуть запрет(ТРЕБУЕТСЯ_ОПЛАТА)

    вернуть разрешение(ВЫЕЗД_РАЗРЕШЁН)
```

Важно: `шлюзБронирования` здесь означает публичный интерфейс `Бронирование`, а не прямой доступ `Доступ` к базе другого контекста.

### `Сессия.завершить()`

`Сессия` меняет только своё состояние.

```text
Сессия.завершить(сессия, времяВыезда):
    сессия.времяВыезда = времяВыезда
    сессия.длительность = времяВыезда - сессия.времяВъезда
    сессия.статус = ЗАВЕРШЕНА

    вернуть СессияЗавершена(сессия.ид, сессия.длительность)
```

Важно: `Сессия` не вызывает `Платёж` и не отправляет уведомления сама.

## Инфраструктурный адаптер

Адаптер только переводит внешний сигнал в вызов системы.

```text
АдаптерЛПРСКУД.слушать():
    пакет = удп.получить()
    номер = пакет.номер
    воротаИд = пакет.воротаИд

    // Упрощение: случай номер = пусто (нераспознанный номер) здесь не показан.
    // В реальной системе адаптер должен зафиксировать инцидент и уведомить охранника,
    // не передавая пустой номер в СервисПриложения.

    СервисПриложения.обработатьВъезд(номер, воротаИд)
```

Адаптер не решает, можно ли открыть шлагбаум. Это делает доменная логика.

## Оркестрация: въезд

Вот где работает `Сервис приложения`.

Он:

1. собирает данные из нескольких контекстов;
2. вызывает доменную логику;
3. координирует транзакцию.

```text
СервисПриложения.обработатьВъезд(номер, воротаИд):
    тс = Клиент.найтиТСПоНомеру(номер)
    статусБрони = Бронирование.получитьСтатус(тс, воротаИд)
    статусДолга = Платёж.получитьСтатусДолга(тс.клиентИд)
    правилаВорот = Площадка.получитьПравилаВорот(воротаИд)

    транзакция:
        решение = Доступ.проверитьВъезд({
            тсНайдено: тс существует,
            вЧёрномСписке: тс.вЧёрномСписке,
            естьДолг: статусДолга.естьДолг,
            естьАктивнаяБронь: статусБрони.активна,
            бронированиеИд: статусБрони.бронированиеИд,
            тсИд: тс.ид,
            автоБронированиеРазрешено: правилаВорот.автоБронированиеРазрешено
        }, Бронирование)

        если решение это запрет:
            откат транзакции
            АдаптерСКУД.удержатьЗакрытым(воротаИд)
            // Упрощение: в реальном сценарии здесь также вызываются
            // АдаптерТабло.показатьПричинуЗапрета(воротаИд, решение.причина) и ЖурналДоступа.записать(...)
            вернуть

        сессия = Сессия.открыть(решение.бронированиеИд)

    АдаптерСКУД.открытьВорота(воротаИд)
```

Главная мысль:

- `Доступ` может запросить авто-бронь только через публичный интерфейс `Бронирование`;
- `Бронирование` не открывает шлагбаум;
- всё склеивает `Сервис приложения`.

## Оркестрация: сумма и оплата

Пока платёж не начат, сумма может меняться. Поэтому `Тариф` считает её в реальном времени.

```text
СервисПриложения.получитьТекущуюСумму(сессияИд):
    сессия = Сессия.получить(сессияИд)
    бронирование = Бронирование.получить(сессия.бронированиеИд)
    клиент = Клиент.получитьПоТС(бронирование.тсИд)

    вернуть Тариф.рассчитать({
        ставка: бронирование.ставка,
        единицы: сессия.текущиеЕдиницы(),
        скидкаПроцент: клиент.скидкаПроцент,
        максимум: бронирование.максимум
    })

СервисПриложения.начатьОплату(сессияИд, источник):
    сумма = СервисПриложения.получитьТекущуюСумму(сессияИд)

    платёж = Платёж.создать({
        сессияИд: сессияИд,
        фиксированнаяСумма: сумма,
        источник: источник   // ЛК_КЛИЕНТА или ВЫЕЗДНЫЕ_ВОРОТА
    })

    вернуть Платёж.провести(платёж.ид)
```

Главная мысль:

- `Тариф` отвечает за текущую сумму;
- `Платёж` отвечает за зафиксированную сумму конкретного платежа.

## Оркестрация: выезд

Для выезда важен быстрый и предсказуемый результат, поэтому основной путь остаётся синхронным.

```text
СервисПриложения.обработатьВыезд(номер, воротаИд):
    сессия = Сессия.найтиАктивнуюПоНомеру(номер)
    бронирование = Бронирование.получить(сессия.бронированиеИд)
    статусОплаты = Платёж.получитьСтатусВыезда(сессия.ид)
    оплатаПроведена = статусОплаты.проведена

    если оплатаПроведена = ложь и статусОплаты.требуется = истина:
        результатОплаты = СервисПриложения.начатьОплату(сессия.ид, ВЫЕЗДНЫЕ_ВОРОТА)

        если результатОплаты.успех = ложь:
            АдаптерСКУД.удержатьЗакрытым(воротаИд)
            вернуть

        оплатаПроведена = истина

    транзакция:
        решение = Доступ.проверитьВыезд({
            сессияНайдена: сессия существует,
            требуетсяОплата: статусОплаты.требуется,
            оплатаПроведена: оплатаПроведена
        })

        если решение это запрет:
            откат транзакции
            АдаптерСКУД.удержатьЗакрытым(воротаИд)
            вернуть

        Бронирование.завершить(бронирование.ид)
        событие = Сессия.завершить(сессия, сейчас())
        ИсходящиеСообщения.сохранить(событие)

    АдаптерСКУД.открытьВорота(воротаИд)
```

Главная мысль:

- оплата на выезде — синхронная часть сценария;
- если клиент оплатил заранее в ЛК, на выезде используется уже зафиксированный платёж;
- события появляются **после** успешной фиксации результата;
- события не заменяют критический путь КПП.

## Где здесь EDA

EDA удобно использовать после завершения основной операции.

Например:

```text
при СессияЗавершена:
    Отчёт.обновитьСтатистикуПарковки()

при ОплатаПроведена:
    Уведомление.отправитьЧек()
    Отчёт.обновитьВыручку()
```

Это хороший пример событийной архитектуры:

- `Сессия` не знает про `Отчёт`;
- `Платёж` не знает про `Уведомление`;
- подписчики сами реагируют на событие.

## Где лучше не использовать EDA

На критическом пути лучше не делать так:

```text
ОбнаруженВъезд -> ШинаСобытий -> Доступ -> ШинаСобытий -> Бронирование -> ШинаСобытий -> Сессия -> ШинаСобытий -> ВоротаОткрыты
```

Для шлагбаума это слишком сложно и рискованно.

Поэтому для въезда и выезда лучше:

- синхронная оркестрация через `Сервис приложения`;
- события только для фоновых и вторичных действий.

## Что важно запомнить

- Доменная логика принимает решения внутри своего контекста.
- `Сервис приложения` связывает контексты между собой.
- Адаптеры общаются с внешним миром.
- EDA хорошо подходит для уведомлений и отчётов.
- EDA не должна ломать критический путь КПП.

Если коротко:

- **DDD** разделяет систему по смыслу;
- **EDA** помогает этим частям общаться менее жёстко;
- **Сервис приложения** удерживает основной сценарий под контролем.
