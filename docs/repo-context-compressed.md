- Репозиторий **документации системного анализа/проектирования** для темы **«Цифровая платформа парковки»**.
- Основная ценность — **набор артефактов**: контекстная диаграмма (DFD0), доменная/концептуальная модель с атрибутами, use case, функциональные требования (FR/UC), ADR, материалы по аутентификации/авторизации и защите данных (152‑ФЗ).
- `ui/` — вспомогательный **wireframe‑макет** (статические HTML, генерируемые из Nunjucks) для визуализации сценариев/навигации; это не реализация продукта.
- Бизнес‑цель (из «Карточка проекта»): довести долю клиентов, использующих онлайн‑сервисы, до **80% за 6 месяцев** после запуска.
- Основные сценарии: просмотр тарифов/правил/доступности, регистрация/вход, управление ТС, бронирования (в т.ч. авто‑при въезде), парковочные сессии (факт въезд/выезд), оплата, договоры, уведомления, обращения, администрирование (тарифы/сектора/места/долги/аналитика), КПП (решение allow/deny).
- Ключевые интеграции (целевая поставка): **СКУД+LPR**, онлайн‑платежи/терминал, **ОФД**, сервис уведомлений, SSO IdP (только ФЛ), ЭДО/1С/аналитика (в полной поставке).
- Репозиторий **не содержит реализации backend/API**; “код” здесь — в основном сборка макета и скрипты подготовки docs.
- Назначение для LLM: быстрый вход в предметную область + **дальнейшая разработка/уточнение документации** (FR/NFR/UC, контракты, трассируемость, риски).

## 2. Tech Stack
- **Основной формат артефактов**: Markdown в `docs/` (спеки/артефакты/ADR/протоколы).
- **Вспомогательный стек для wireframe‑макета** (не продукт):
  - JS (Node.js ESM), Nunjucks, SCSS/Sass → статические HTML/CSS в `ui/`.
  - Python‑утилиты для извлечения/подготовки текстов в `docs/`.

## 3. Repository Structure
- `/readme.md`: минимальный заголовок проекта.
- `/package.json`, `/package-lock.json`: dev‑зависимости для сборки wireframe.
- `/scripts/`
  - `build-templates.mjs`: рендер Nunjucks шаблонов в HTML по списку страниц.
- `/ui/` (wireframe)
  - `index.html`: точка входа (выбор контура).
  - `wireframe.css`: собранные стили.
  - `scss/`: исходники стилей (tokens/layout/forms/modals/lk).
  - `templates/`: Nunjucks (layouts/pages/partials).
  - `site/`, `client/`, `admin/`, `guard/`: сгенерированные/готовые HTML‑страницы контуров.
- `/docs/` (основной источник контекста)
  - `artifacts/`: артефакты анализа (контекст, доменная модель, ИБ, навигация, ES TO‑BE, UCM/impact/opportunity и др.).
  - `specs/`: формальные требования (FR по сущностям/UC, NFR по UI качеству).
  - `architecture/`: ADR по ключевым решениям.
  - `protocols/`: протоколы встреч и зафиксированные решения (источник решений/терминологии).
  - `demo-days/`: материалы презентаций/демо (в т.ч. pdf/pptx).
- Шум/нецелевое для контекста: `node_modules/`, `.venv*` (исключены `.gitignore`), бинарные презентации/пдф — использовать только как справку.

## 4. Runtime / Entry Points
- В репозитории **нет runtime приложения** (нет backend/деплоя/окружений).
- Единственная “исполняемая” часть — **локальная сборка wireframe‑макета**:
  - точка входа: `ui/index.html` (как статический файл),
  - сборка (опционально, чтобы пересобрать макет): `npm run build` (`build:css`, `build:templates`),
  - entrypoint сборки: `scripts/build-templates.mjs`.
- **Env/config**: отсутствуют; любые env‑зависимости будущей реализации не задокументированы в репо.

## 5. Core Architecture
- **Архитектура в этом репо = архитектурные артефакты**, а не модули кода.
- **Граница системы (контекст DFD0)**: “Цифровая платформа парковки” включает:
  - Business Site (публичный сайт),
  - Consumer App (PWA/ЛК клиента),
  - Admin Panel + внутренний инструмент (CRM/конструктор договоров/аналитика),
  - интерфейс охранника КПП.
- **Ключевые архитектурные решения (ADR)**:
  - ADR‑001: решение `allow/deny` для КПП считается **онлайн в платформе**, СКУД — исполнитель (нужны SLA/деградация/кэш).
  - ADR‑002: **Бронирование — мастер**, Парковочная сессия “тонкая” и всегда ссылается на бронирование (включая авто‑бронь “на лету”).
- **Основные потоки данных (high level)**:
  - СКУД/LPR → платформа: событие въезд/выезд + ГРЗ + метаданные КПП/полосы; платформа → СКУД: `allow/deny`.
  - Платежи/терминал ↔ платформа: инициация суммы/id ↔ статус/квитанция.
  - Платформа → ОФД: чек к фискализации; ОФД → платформа: фискальный чек/статус.
  - Платформа ↔ сервис уведомлений: уведомление/канал/адресат ↔ статус доставки.
  - SSO/IdP (только ФЛ): редиректы/токены, связывание `sub` с профилем клиента.
- **Слои (имплицитно по артефактам)**:
  - Доменная модель (сущности/инварианты) → FR/UC → UI‑экраны wireframe → внешние интеграции.
- **Контуры доступа (authn/authz)**:
  - Клиенты: ФЛ пароль или SSO; ЮЛ только пароль; доступ только к своим данным.
  - Сотрудники: пароль + обязательный TOTP 2FA; RBAC (охранник/управляющий/владелец).

## 6. Domain Knowledge
- **Термины (ядро)**: Тариф, Бронирование, Договор, Парковочная сессия, ПМ (парковочное место), Сектор, КПП, СКУД, LPR, ГРЗ, ТС, льготные категории (ОВЗ и др.).
- **Сущности (ключевые)** (подробно — в концептуальной модели):
  - `Клиент` (ФЛ/ЮЛ), `Организация` (для ЮЛ), `ТС`, `Тариф`, `Сектор`, `ПМ`, `КПП`,
  - `Бронирование` (план/право, мастер для финансов/доступа),
  - `Парковочная сессия` (факт: времена, КПП, способ допуска; ссылается на бронирование),
  - `Расчет стоимости` (preview суммы, расчетный снимок, применение тарифных правил к конкретной ситуации),
  - `Платеж`, `Чек`, `Уведомление`/`Шаблон уведомления`, `Обращение`, `Согласие`,
  - `Сотрудник` (роль + 2FA секрет), настройки уведомлений/оплаты.
- **Бизнес‑правила (выжимка)**:
  - Доступ на КПП определяется платформой по актуальным данным: договоры/брони/долги/ЧС/грейс‑периоды.
  - Авто‑сценарий: при въезде может создаваться **автоматическое бронирование** (выбор сектора/ПМ по типу ТС/тарифу/близости к КПП/загрузке).
  - Инвариант ADR‑002: нет ПС без брони; “въезд без предварительной брони” → создаётся бронь “на лету”.
  - Ограничения MVP (из карточки): договора ЮЛ заключаются **вне MVP** (ручной режим), но доступ/самообслуживание по спискам ТС — в MVP.
  - Минимальная идентификация всех клиентов: телефон + имя (в т.ч. краткосрочные/гостевые).
- **ИБ/ПДн (152‑ФЗ)**:
  - Категории чувствительных данных: паспортные данные, льготные документы, контакты, ГРЗ/ТС, финансы, логи/аудит.
  - Меры: хранение в РФ, RBAC, шифрование чувствительных полей/файлов, маскирование, аудит, rate‑limit, bcrypt/Argon2, TOTP секрет — только в шифрованном виде.

## 7. APIs / Integrations
- **СКУД+LPR**: события въезда/выезда (ГРЗ, время, КПП/полоса, качество распознавания); ответ платформы — `allow/deny`; режим деградации/кэш требуется (ADR‑001).
- **Платёжный провайдер (онлайн)**: суммы/id ↔ статус/квитанции/возвраты.
- **Платёжный терминал (офлайн)**: аналогично статус/id/квитанция.
- **ОФД**: фискализация чеков.
- **Уведомления**: SMS/e-mail/личный кабинет (push упоминается в контексте; в enum каналов встречаются ЛК/звонок/e‑mail/SMS).
- **SSO/IdP (только ФЛ)**: OAuth2/OIDC (Яндекс ID, VK ID; PKCE, state/CSRF).
- **ЭДО, 1С/биллинг, внешняя аналитика**: в “полной поставке” (интеграции и выгрузки).
- **Auth/permissions (резюме)**:
  - Клиенты: self‑scope данные.
  - Сотрудники: RBAC; 2FA обязателен; аудит ручных допусков/просмотров.

## 9. Developer Workflows
- **Основной workflow = развитие документации**:
  - начинать с `docs/artifacts/project-charter.md` → `docs/artifacts/context-diagram.md` → `docs/artifacts/conceptual-model-with-attributes.md` → `docs/specs/**` → `docs/architecture/*.md`.
  - при уточнениях ссылаться на интервью и решения встреч из `docs/interviews/*`.
  - фиксировать ключевые развилки как ADR в `docs/architecture/`.
  - держать согласованность: термины/сущности/инварианты из концептуальной модели ↔ FR ↔ UC ↔ wireframe.
- **Wireframe (опционально, только для визуализации)**:
  - пересобрать: `npm ci` → `npm run build`;
  - открыть `ui/index.html` в браузере.

## 10. Open Questions / Risks
- **Неясно:** целевой формат “исполняемых” спецификаций интеграций (OpenAPI/AsyncAPI, схемы сообщений, idempotency, retry) — в docs есть потоки данных, но нет контрактов.
- **Неясно:** желаемая степень формализации требований (EARS/IEEE, нумерация FR/NFR, матрица трассировки) — частично есть FR‑файлы, но нет единого правила “Definition of Done” для требований.
- **Неясно:** единый глоссарий как отдельный артефакт (сейчас размазан по “Карточке проекта”, протоколам и модели).
- **Риск:** ADR‑001 (онлайн `allow/deny`) требует строгих требований к доступности/латентности + продуманной деградации СКУД; это пока на уровне текста.
- **Риск:** “авто‑бронь на лету” (ADR‑002) → много “технических” бронирований; нужно заранее определить, как их показывать/фильтровать в UI и отчётах.
- **Риск/пробел:** политика хранения и сроки логов/аудита/ПДн (152‑ФЗ) обозначены частично; нет NFR/инфраструктурной спецификации.
- **Риск/пробел:** UX в wireframe есть, но отсутствуют критерии приемки по производительности/безопасности/наблюдаемости.

## Compression Notes
- Отброшено: `node_modules/`, виртуальные окружения `.venv*`, большинство повторяющихся артефактов демо‑дней (`.pdf/.pptx`) и подробные протоколы (оставлены только индексы и ключевые выводы).
- Сильно сжато: контекстная диаграмма (DFD0) и концептуальная модель (десятки страниц атрибутов/enum) → оставлены только центральные сущности/инварианты/интеграции.
- Почти не влияет на понимание проекта: вспомогательные python‑скрипты подготовки docs (не участвуют в целевой системе), детализация модалок и верстки отдельных страниц (важны как UX‑пример, но не как архитектурное ядро).
